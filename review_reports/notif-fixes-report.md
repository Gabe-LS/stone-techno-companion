# Implementation report

## Changes
- `server/chat/chat.html`, `loadRooms()` (~line 1547-1566): stopped resetting `roomTypeLookup = {}`; now merges the `/rooms` response into it. Rebuilt the prune loop to only delete `roomTypeLookup`/`unreadByRoom` entries whose `rid` is missing from the new response *and* whose type is `'stage'`/`'general'` — DM/meetup/unknown entries survive. Added `dbg('[BADGE] loadRooms pruned stale stage/general entries: ...')` logging the pruned ids.
- `server/chat/chat.html`, `refreshAllBadges()` (~line 1757): selector extended to `'.room-item[data-room-id], .member-item[data-room-id]'` so DM rows (rendered by `loadDMs()` as `.member-item`) get repaired on a bulk `badge_counts` sync. Confirmed meetup rows already use `.room-item[data-room-id]` (no extra selector needed) and `updateBadge()` already queries generically by `[data-room-id="..."]`, so it needed no change.
- `server/chat_ws.py`, `_do_send_push` payload (~line 436-452): added `"push_id": secrets.token_hex(8)` (a fresh random value per push) alongside the existing `push_index`. `secrets` was already imported.
- `server/static/sw.js`: bumped `SW_VERSION` to `'v10'` (line 4). Tag construction (line ~35) now prefers `data.push_id`, falling back to `data.push_index`, then a random string — so a server restart resetting `push_index` to 0 can never collide with a still-visible notification's tag. Close-previous-same-room logic untouched.
- `server/chat/chat.html`: extracted the subscribe+POST flow from `_enableAllNotifications()` into a new `_subscribePush()` helper (~line 3829) that returns `true` only when both the browser subscription and the server POST succeed (non-ok POST now counts as failure, previously ignored). `_enableAllNotifications()` (~line 3874) now branches on that result: shows `"Notifications enabled"` only on real success, otherwise shows the Brave-specific fallback toast and sets `_pushSubscribed = false`. Added `_repairPushSubscription()` (~line 3862) guarded by `_pushRepairAttempted` (runs at most once per page load) that silently re-subscribes when `Notification.permission === 'granted'` but no push subscription exists, using the same `_subscribePush()` helper, dbg-logging only. Wired into `route()` right after the existing `await _checkPushStatus();` call (~line 4390).

## Deviations from spec
- None. All four fixes implemented as specified.

## Concerns
- Fix 4, as specified, makes `_subscribePush()` return `false` immediately when `!('serviceWorker' in navigator && 'PushManager' in window)` — i.e., on any browser without push support at all, not just Brave-with-services-disabled. That means `_enableAllNotifications()` will show the Brave-specific remediation toast (`"...if you use Brave, enable..."`) even to non-Brave users on browsers that structurally can't do push. The spec's wording ("Only show `Notifications enabled` when the subscription was actually registered with the server... Treat a non-ok POST response as failure too") is explicit enough that I implemented it literally rather than adding an unspecified third branch, but the toast copy may read oddly for that edge case.
- One judgment call not explicitly pinned down by the spec: I `await`ed `_repairPushSubscription()` in `route()` (sequential with `_checkPushStatus()`) rather than firing it in the background. In the common case where `_pushSubscribed` is already `true`, the function returns immediately after the guard check with no network call, so this adds no meaningful latency to routing. Only the "permission granted but subscription missing" repair path performs network I/O, and blocking on it seemed more consistent with the surrounding `route()` code (which already awaits `_checkPushStatus()`, `loadBlockedUsers()`, `loadRooms()` sequentially) than deviating to fire-and-forget.
