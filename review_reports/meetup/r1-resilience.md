- ID: resilience-1
- Severity: critical
- Confidence: certain
- Location: server/chat/chat.html:3276-3296 (`submitMeetup`), :1086-1088 (`wsSend`)
- Finding: `submitMeetup` calls `wsSend('create_meetup', payload)` â€” a fire-and-forget call that is a pure no-op if the socket isn't open (`if (ws && ws.readyState === 1) ws.send(...)`, else nothing happens, no exception, no return value) â€” then unconditionally calls `closeMeetupModal()` and `showToast('Meetup created!')` regardless of whether anything was actually sent, ever reached the server, or was accepted. There is no ack/confirmation event handled for meetup creation (see resilience-5). On a flaky connection, a background tab reconnect window, or the ~30s exponential-backoff reconnect gap, the user gets a definitive "Meetup created!" success message and the modal closes, while the meetup silently never exists. There is no way for the user to discover this short of manually checking the meetups tab.
- Recommendation: Make creation request/response: either use the existing `POST /chat/api/meetups` REST endpoint (which already validates and returns errors, see resilience-3) with `await`, showing the toast only on a resolved success and an error toast on failure/timeout; or, if keeping the WS path, have the server emit a `meetup_create_ack`/`meetup_create_error` event keyed to a client-generated request id, and gate the toast/modal-close on receiving it (with a client-side timeout that shows a "couldn't create meetup, check your connection" error if no ack arrives in e.g. 5s).
- Effort: M
- Risk of change: low

- ID: resilience-2
- Severity: high
- Confidence: certain
- Location: server/chat_ws.py:1748-1832 (`create_meetup` WS handler)
- Finding: Every failure path in the `create_meetup` WS handler is silent from the client's perspective: rate-limit exceeded (`if not manager.check_rate_limit(user_id): continue`), missing title/meetup_time (`continue`), and invalid ISO timestamp (`continue`) all just drop the message with zero response sent back to the sender. Combined with resilience-1's optimistic UI, there is no code path by which a rejected meetup creation ever surfaces to the user â€” the toast always says success even when the server discarded the request outright.
- Recommendation: Send an explicit error event back to the requesting connection (e.g. `{"event": "create_meetup_error", "reason": "..."}`) on every `continue` branch, and have the client listen for it to replace the optimistic toast with a failure message.
- Effort: S
- Risk of change: low

- ID: resilience-3
- Severity: medium
- Confidence: certain
- Location: server/chat_api.py:1232-1264 (`POST /meetups`) vs server/chat/chat.html:3276-3296 (`submitMeetup`)
- Finding: A proper REST creation endpoint already exists with correct error semantics (`HTTPException(429/400)` on rate limit/validation failure, normal JSON response on success), but the frontend's only creation path (`submitMeetup`) never calls it â€” it exclusively uses the fire-and-forget WS event. All the resilience benefits of that endpoint (a rejectable promise, HTTP status codes, `api()`'s existing error-to-toast plumbing used elsewhere like `toggleMeetupJoin`) go unused for the one action (create) that most needs them.
- Recommendation: Switch `submitMeetup` to `await api('/meetups', { method: 'POST', body: JSON.stringify(payload), headers: {...} })` inside a try/catch, showing the toast/closing the modal only after a successful response, mirroring the pattern already used for join/leave.
- Effort: S
- Risk of change: low

- ID: resilience-4
- Severity: medium
- Confidence: likely
- Location: server/chat/chat.html:3276-3296 (`submitMeetup`), :3250-3252 (modal footer button)
- Finding: The "Create Meetup" button has no disabled/loading state and `submitMeetup` is fully synchronous (no `await`, no in-flight guard flag). A double-tap, a slow/uncertain UI causing a retry, or a user re-opening the modal after believing the first attempt failed (there is no failure signal per resilience-1/2 anyway, but a user might still retry after a perceived hang) can fire `create_meetup` twice with the same content. Nothing server-side deduplicates meetup creation (unlike regular messages, which have `check_duplicate` in `chat_ws.py`), so two nearly-identical meetup rooms/invite cards would be created.
- Recommendation: Disable the submit button synchronously on click and re-enable only on error/timeout; consider a short-window server-side dedup keyed on `(user_id, title, meetup_time)` similar to the existing message dedup.
- Effort: S
- Risk of change: low

- ID: resilience-5
- Severity: medium
- Confidence: certain
- Location: server/chat_ws.py:1825-1832 (broadcasts `meetup_created`) vs server/chat/chat.html (`handleWSEvent` switch, no `case 'meetup_created'`)
- Finding: The server broadcasts a `meetup_created` event to the stage/main room whenever a meetup is made, but the client's WS event switch has no handler for it at all (only `meetup_updated` and `meetup_expired` are handled, chat.html:1494-1509). The meetups list is only ever refreshed by `loadMeetups()` on explicit tab-open (chat.html:1672, 1963). So even with a perfectly healthy, connected WebSocket, other users never see a new meetup appear live â€” and this is strictly worse across any disconnect/reconnect gap, since there is also no reconciliation fetch on reconnect (see resilience-6).
- Recommendation: Add a `case 'meetup_created'` handler that appends/upserts the meetup into the in-memory list and re-renders the meetups tab if open (and optionally shows a toast/badge if closed).
- Effort: S
- Risk of change: low

- ID: resilience-6
- Severity: medium
- Confidence: certain
- Location: server/chat/chat.html:1059-1063 (`sock.onopen`), :1494-1509 (`meetup_updated`/`meetup_expired` handlers)
- Finding: `meetup_updated` (join/leave attendee changes) and `meetup_expired` are pure live broadcast-only updates with no backing state resync. `sock.onopen` only re-joins `currentRoom` â€” it never re-fetches meetups. Given the exponential-backoff reconnect (up to 30s per attempt, chat.html:1071-1080), any attendee-count change, join-button state flip, or expiry broadcast that occurs while a client is disconnected/reconnecting is permanently lost: the join button, "N going" count, and expired-dimming will show stale data indefinitely until the user happens to leave and re-enter the meetups tab (which calls `loadMeetups()` again).
- Recommendation: On `sock.onopen` (or specifically when the previously-open tab is "meetups"), re-run `loadMeetups()` to reconcile state after any reconnect, not just re-join the active room.
- Effort: S
- Risk of change: low

- ID: resilience-7
- Severity: high
- Confidence: likely
- Location: server/chat_ws.py:1333 (outer `while True` loop) and :2123-2126 (single outer `except Exception` wrapping the whole loop)
- Finding: `create_meetup`/`join_meetup`/`leave_meetup` handling has no per-event try/except â€” the only exception boundary is the one wrapping the entire `handle_chat_ws` message loop (`except Exception: logger.exception(...)`), which falls through to `finally: manager.disconnect(conn_id)`. Any unexpected exception raised while processing a single `create_meetup`/`join_meetup`/`leave_meetup` event (e.g. a DB error, a `None` field slipping past the minimal validation into `create_meetup`) tears down the user's entire WebSocket connection â€” every room, not just the meetup action â€” with no error surfaced to the client beyond a bare socket close, which the client then silently treats as a reconnect-worthy disconnect. This is a much larger blast radius than the triggering action warrants, and combines with resilience-1's optimistic toast to produce a confusing UX: "Meetup created!" immediately followed by the whole chat silently reconnecting.
- Recommendation: Wrap individual event branches (at minimum `create_meetup`/`join_meetup`/`leave_meetup`, which touch multiple tables) in their own try/except that logs and sends a scoped error event back to the sender instead of letting the exception propagate to the connection-level handler.
- Effort: M
- Risk of change: medium (touches the main WS dispatch loop structure)
