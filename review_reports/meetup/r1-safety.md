- ID: safety-1
- Severity: critical
- Confidence: certain
- Location: server/chat_ws.py:1748-1832 (`create_meetup` WS handler), server/chat_api.py:1232-1264 (`POST /meetups`), server/chat_db.py:1205-1264 (`create_meetup`)
- Finding: Meetup `title`, `note`, and `location_label` are free text (up to 60/200/100 chars) written straight to the DB and broadcast, but neither creation path calls `moderate_message()` or the word filter/OpenAI/GPT-nano pipeline that every normal chat message goes through. A user can put slurs, drug offers, sexual solicitation, or a harassment/lure message directly in a meetup's title/note and it is never scanned. Worse, when `stage_id` is set, the raw `title` is embedded verbatim in a `meetup_invite` message that's persisted via `create_message` and broadcast into that room (chat_ws.py:1773-1824) — also without going through `_moderate_and_broadcast`/`moderate_message`. This is a full moderation bypass: wrapping abusive/luring text as a "meetup" gets it broadcast into public rooms with zero automated review, unlike an identical string sent as a normal message.
- Recommendation: Run `title`, `note`, and `location_label` through the same word-filter + AI moderation pipeline used for messages before `create_meetup` commits (reject/hold on fail, mirroring the pending-message flow), and route the `meetup_invite` card through `_moderate_and_broadcast` instead of a raw `create_message`+broadcast.
- Effort: M
- Risk of change: medium

- ID: safety-2
- Severity: critical
- Confidence: certain
- Location: server/chat_ws.py:1748-1771; server/chat_api.py:1232-1264
- Finding: `check_ban_mute()` (used everywhere else, even for unmoderated DMs) is never called before `create_meetup`. `mute_user()` (chat_db.py:561-564) only sets `muted_until` — it does not revoke the session or close the socket — so a muted user retains a fully live WS connection and a valid session cookie for the mute duration. That user can still create meetups (with GPS location, note, and a public invite broadcast) via both the WS event and the REST endpoint, completely evading the mute the admin/AI just imposed. Combined with safety-1, an actively-muted abuser can keep luring/harassing via meetups with content that is never scanned and never enforced.
- Recommendation: Call `check_ban_mute(db, user_id)` (or full `moderate_message` per safety-1) at the top of both the WS `create_meetup` handler and `POST /meetups`/`/meetups/{id}/join` before doing anything, and reject with the same `message_removed`/error pattern used elsewhere on mute/ban.
- Effort: S
- Risk of change: low

- ID: safety-3
- Severity: critical
- Confidence: certain
- Location: server/chat_api.py:1159-1230 (`GET /meetups`, `GET /meetups/{meetup_id}`); server/chat/chat.html:3266-3273, 3286-3288 (`useMeetupGPS`/`submitMeetup`)
- Finding: `location_lat`/`location_lng` (full, un-rounded GPS precision — the client only truncates the *displayed* text field to 4 decimals, the payload sends raw `pos.coords.latitude/longitude`) and the full attendee roster (id + display_name) are returned by `GET /meetups` and `GET /meetups/{id}` to **any authenticated chat user**, with no check that the requester has joined ("going") or is even in the same room as the meetup. Anyone can silently poll the meetup list and see exactly where a specific person (creator or any listed attendee, identifiable by display name) is or will be, at what time, without ever RSVPing — so the target has no way to know they're being watched. This is a direct stalking/harassment enabler in an app explicitly tied to a physical festival.
- Recommendation: Gate `location_lat`/`location_lng` (and ideally the precise coordinates in general — consider only ever exposing `location_label`) behind meetup membership (`is_going`), or at minimum log/rate-limit repeated detail lookups; do not return attendee-level GPS to non-attendees.
- Effort: M
- Risk of change: medium

- ID: safety-4
- Severity: high
- Confidence: certain
- Location: server/chat_db.py (block logic absent from meetups.py functions); server/chat_ws.py:1834-1866 (`join_meetup`/`leave_meetup`); server/chat_api.py:1159-1287
- Finding: The block feature (`is_blocked`, used to filter messages/DM badges/unread counts elsewhere, e.g. chat_ws.py:391-407,1155,1872) is never checked anywhere in the meetup code path. Blocking an abusive user does not hide their created meetups from your list, does not prevent you from being shown in their attendee roster (or vice versa), and does not stop them from joining a meetup you created (with your GPS exposed, see safety-3) or you from joining theirs. A victim who has blocked their harasser gets no protection at all on the meetup surface — the one surface that discloses real-world physical location.
- Recommendation: Filter blocked/blocking users' meetups out of `GET /meetups` for each other, and reject `join_meetup` when either party has blocked the other.
- Effort: M
- Risk of change: low

- ID: safety-5
- Severity: high
- Confidence: certain
- Location: server/chat_ws.py:1772-1824 (invite card only created `if stage_id`); server/chat_ws.py:2045-2118 (`report_message`, message-scoped only); server/chat/chat.html (no meetup-level "Report" action in `loadMeetups`/meetup room UI)
- Finding: There is no dedicated "report this meetup" action anywhere (REST, WS, or UI) — the only reporting mechanism is `report_message`, tied to a specific message id/room. If a meetup is created **without** a `stage_id` (fully valid — `stage_id` is optional), no `meetup_invite` message is ever created, so there is literally no message a victim can right-click "Report" on. A meetup with a luring title/fake location and no stage attachment is completely unreportable through the app; a victim's only recourse is asking an admin to notice it manually in the admin Rooms tab.
- Recommendation: Add a `report_meetup` event/endpoint (mirroring `report_message`, snapshotting title/note/location/creator) reachable from the meetup list/detail UI regardless of `stage_id`.
- Effort: M
- Risk of change: low

- ID: safety-6
- Severity: medium
- Confidence: certain
- Location: server/chat_ws.py:1748-1771 (`stage_id` accepted from client payload, not validated); server/chat/chat.html:3287 (`stage_id: currentRoom`)
- Finding: `stage_id` in `create_meetup` is taken verbatim from the WS payload with no check that the sender is a member of that room, that it's actually a stage/general room (vs. some other room id), or that the room isn't `is_read_only`. Any regular attendee can therefore post an official-looking `meetup_invite` card into a high-traffic official stage/announcement room (impersonating a host/organizer, e.g. "DJ meet & greet backstage — official"), and this bypasses `is_read_only` enforcement that would normally block that user's ordinary messages in that room.
- Recommendation: Validate the sender is a current member of `stage_id`'s room and that the room is not `is_read_only` before posting the invite card; consider restricting stage-attached meetups to rooms the requester can otherwise post in.
- Effort: S
- Risk of change: low

- ID: safety-7
- Severity: medium
- Confidence: certain
- Location: server/chat_ws.py:1749 (`manager.check_rate_limit(user_id)`, shared 10-msgs/10s bucket); combined with safety-1/safety-2
- Finding: Meetup creation shares the generic message rate limit (10 per 10s) with normal chat sends, rather than having its own (tighter) limit, so a bad actor can fire up to 10 unmoderated meetup broadcasts (each with GPS/title/note, safety-1) into a public room in 10 seconds. Because meetup content never passes through `moderate_message`, none of this ever generates a strike — the standard 4-step strike/mute/ban escalation ladder never engages for meetup-based abuse, so a repeat offender using meetups to harass/spam is invisible to the automated enforcement system that would have caught the same text sent as a normal message.
- Recommendation: Apply a dedicated, stricter rate limit to `create_meetup` (e.g. 1-2 per minute), and once safety-1 routes meetup content through moderation, ensure a moderation failure there issues a strike exactly as a rejected message would.
- Effort: S
- Risk of change: low

- ID: safety-8
- Severity: low
- Confidence: certain
- Location: server/chat_db.py:1303-1312 (`get_all_meetups`); server/chat/admin.html:419 ("Delete meetup" only)
- Finding: The admin view of meetups (`GET /admin/meetups` → `get_all_meetups`) selects `location_label` but never `location_lat`/`location_lng`, and the admin UI only exposes a "Delete meetup" action with no way to view title/note/location detail at all. If a report ever does come in (once safety-5 is fixed) about a meetup using a fake/misleading location label to lure someone, an admin investigating has no way to see the actual submitted GPS coordinates to compare against the claimed label — they can only delete the room blind.
- Recommendation: Surface `location_lat`/`location_lng`/`note` in `get_all_meetups` and add a read-only detail view in the admin Rooms tab (or a dedicated Meetups view) so moderators can actually inspect a reported meetup before deciding to delete/ban.
- Effort: S
- Risk of change: low
