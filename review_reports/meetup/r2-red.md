- ID: RED-1
- Severity: critical
- Confidence: certain
- Location: server/chat_api.py:1159-1230 (`list_meetups`, `get_meetup`); server/chat_db.py:1283-1301 (`get_meetup_attendees`, `get_active_meetups`)
- Finding: `GET /chat/api/meetups` and `GET /chat/api/meetups/{id}` return `location_lat`/`location_lng` (the creator's real GPS coordinates per the manifest) plus the full attendee `id`+`display_name` list for **every active meetup**, to **any authenticated user** — there is no check that the caller is an attendee, invited, or even a member of the originating room (compare to `/rooms/{id}/online`, which does gate on `meetup_attendees` membership at chat_api.py:1143-1148). Any registered chat account (a stalker needs only a valid, unbanned session) can page through every meetup at the festival, read exact coordinates of where a named individual (by display name in the attendee list) will physically be and when, and build a full attendee/social graph without ever joining or being noticed.
- Recommendation: Require the caller to be an attendee (or the room to be public-by-design) before returning `location_lat`/`location_lng`/`location_label`/attendee identities; for the list endpoint, strip GPS fields and attendee identities unless `user_id in att_ids`, returning only aggregate `attendee_count`.
- Effort: S
- Risk of change: low

- ID: RED-2
- Severity: high
- Confidence: certain
- Location: server/chat_ws.py:1748-1832 (`create_meetup` WS handler); server/chat_api.py:1232-1264 (`create_meetup_endpoint`); server/chat_db.py:561-564 (`mute_user` — does not revoke sessions/WS)
- Finding: Neither the WS nor REST meetup-creation path calls `check_ban_mute`/`is_banned` — the only gate is `manager.check_rate_limit`. Because `mute_user` only sets `muted_until` and does not close the WS connection or delete the session (unlike `ban_user`, chat_db.py:609-611), a muted user (e.g. after 3 strikes, mid-30-minute mute) can keep creating meetups and broadcasting invite cards indefinitely — the one enforcement mechanism (mute) meant to stop them from posting content is silently bypassed by using `create_meetup` instead of `send_message`.
- Recommendation: Call `check_ban_mute(db, user_id)` at the top of both the WS `create_meetup` handler and `create_meetup_endpoint`, rejecting/early-returning exactly as `_moderate_and_broadcast` does for normal messages.
- Effort: S
- Risk of change: low

- ID: RED-3
- Severity: critical
- Confidence: certain
- Location: server/chat_ws.py:1760-1832 (`create_meetup` builds `stage_id`/invite unconditionally); compare gating in server/chat_ws.py:1549-1603 (`send_message`'s `is_read_only`/`allows_media`/room-type checks)
- Finding: When `stage_id` is supplied, the handler calls `get_room(db, stage_id)` only to read `ttl_minutes`, then unconditionally calls `create_message(db, stage_id, user_id, "meetup_invite", ...)` (default `moderation_status="approved"`, chat_db.py:965) and broadcasts it — with **no** `is_read_only` check, **no** `allows_media`/room-type check, **no** membership check, and **no** validation that `stage_id` is even a group room (it could be another user's DM room id or a meetup room id the caller never joined). This lets any authenticated, non-banned user: (a) post an "official-looking" meetup card into a read-only/announcement room they could never post a normal message into, and (b) inject a message into an arbitrary room by ID (private meetup room, or a DM room id if obtained/guessed) without being a participant. The card's `title` (up to 60 chars, attacker-controlled) is never passed through the word filter or AI moderation, so it can carry harassment/slurs/luring text and still broadcast instantly.
- Recommendation: Route the invite creation through the same checks `send_message` uses (`is_read_only`, room-type/participant checks, `check_ban_mute`) before calling `create_message`, or reuse `_moderate_and_broadcast` for the invite content instead of a bespoke path.
- Effort: M
- Risk of change: low

- ID: RED-4
- Severity: critical
- Confidence: certain
- Location: server/chat_api.py:1267-1286 (`join_meetup_endpoint`/`leave_meetup_endpoint`); server/chat_ws.py:1834-1849 (`join_meetup` WS handler); server/chat_db.py:1266-1273 (`join_meetup` — `INSERT OR IGNORE`, no checks); server/chat_ws.py:1598-1603 (meetup-room posting only checks self-granted `meetup_attendees` row)
- Finding: Joining a meetup has zero authorization: no check that the meetup exists, hasn't expired, or — critically — that the requester isn't blocked by the creator (contrast with the DM path, which calls `db_is_blocked`/`is_blocked` at chat_api.py:1344 and chat_ws.py:1584-1596). A user the victim has already blocked or reported can, once they locate the meetup via RED-1, self-join instantly, gain full read/post access to that meetup's dedicated chat room (which carries the real-world location), appear in real time as an attendee, and continue contacting the victim entirely outside the DM channel they were blocked from — a direct block/report-bypass communication and physical-proximity channel.
- Recommendation: Check `is_blocked(creator_id, joiner_id)` (either direction) before allowing `join_meetup`; consider requiring creator approval or an invite token for meetups that include a real GPS location, and reject joins to expired/nonexistent meetup ids.
- Effort: M
- Risk of change: medium

- ID: RED-5
- Severity: medium
- Confidence: certain
- Location: server/chat_db.py:1205-1264 (`create_meetup`)
- Finding: `title`, `note`, and `location_label` are stored and later served (RED-1) without ever passing through `moderate_message`/the word filter/OpenAI/GPT pipeline that gates every normal chat message. Combined with RED-1's unauthenticated-to-membership global visibility, this is effectively a broadcast channel to every logged-in user that completely skips the app's only content-safety layer — an attacker can put a targeted-harassment or luring message in a meetup title/note and it is served to the whole event's user base with no filtering and no resulting strike.
- Recommendation: Run `title`/`note`/`location_label` through the same word-filter (and ideally the AI layers) used for messages before insert; reject or flag on match, same as `_moderate_and_broadcast` does.
- Effort: S
- Risk of change: low

- ID: RED-6
- Severity: low
- Confidence: likely
- Location: server/chat_db.py:1266-1273 (`join_meetup`); server/chat_ws.py:1834-1849
- Finding: `join_meetup` never verifies the `meetup_id` refers to an existing meetup before inserting into `meetup_attendees`, and the WS handler broadcasts `meetup_updated` to that room id regardless. This allows silent creation of orphan attendee rows for arbitrary/expired/made-up ids and gives no signal distinguishing a real join from a no-op, which could be used to probe or pollute meetup state as part of a larger enumeration/harassment chain (e.g., pre-registering attendance on ids scraped before RED-1 is fixed).
- Recommendation: Look up the meetup first and 404/no-op cleanly if it doesn't exist or has already expired.
- Effort: S
- Risk of change: low

- ID: RED-7
- Severity: medium
- Confidence: likely
- Location: server/chat_ws.py:827-836 (`check_rate_limit`, shared 10-per-10s bucket) used identically by both `send_message` and `create_meetup`
- Finding: `create_meetup` uses the same generic 10-events/10s bucket as ordinary messages, with no dedicated per-meetup-creation limit and (per RED-2) no mute gating. An attacker can burst up to the rate-limit ceiling creating meetups with harassing/luring titles (RED-5) and, when `stage_id` is set, have each one auto-broadcast an unmoderated invite card into an arbitrary room (RED-3) — turning meetup creation into a repeatable harassment-blast primitive rather than a rare, deliberate action.
- Recommendation: Add a dedicated, stricter rate limit for `create_meetup` (e.g. 1 per minute) independent of the messaging bucket, and gate on ban/mute (RED-2) so the amplification vector closes with the moderation bypass fixes.
- Effort: S
- Risk of change: low
