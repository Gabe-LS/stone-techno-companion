- ID: notifications-1
- Severity: critical
- Confidence: certain
- Location: server/chat_db.py:881-913 (`get_unread_counts`), server/chat_db.py:1266-1272 (`join_meetup`), server/chat_db.py:1245 / 1226-1263 (`create_meetup`), server/chat_api.py:1267-1286 (join/leave endpoints), server/chat_ws.py:474-532 (`_do_send_push`), tests/test_notifications.py:106-113 (`setup_meetup` fixture)
- Finding: Meetup attendees are recorded only in `meetup_attendees` — nothing ever inserts a matching row into `room_memberships` for a meetup room (neither `create_meetup`'s auto-add-creator-as-attendee, nor `join_meetup`, nor the REST join/leave endpoints). `get_unread_counts`, which `_do_send_push` uses as its "is there anything unread to push about" gate (`if count == 0: return False`), is built from a UNION of `room_memberships` and `dm_participants` only — it has no meetup branch. This means `count` is always 0 for a meetup room, so `_do_send_push` always bails out before it ever reaches the meetup-specific title/body formatting (chat_ws.py:537-539, and the equivalent client dead-code at chat.html:3879/3885). Net effect: no chat message sent inside a meetup's own room can ever generate a push notification, for anyone, in production. This is masked by the test suite: `tests/test_notifications.py`'s `setup_meetup` fixture manually calls `join_room_membership(...)` for both users to make the test pass — a setup step that never happens in the real join-meetup flow — so the green test gives false confidence that meetup push works.
- Recommendation: Make attendee bookkeeping consistent with the notification pipeline — either (a) call `join_room_membership`/remove it symmetrically with `leave_room_membership` whenever `join_meetup`/`leave_meetup` run (and on meetup deletion/expiry), or (b) make `get_unread_counts` (and the on-connect `badge_counts` blast) meetup-aware by unioning `meetup_attendees` the same way `_get_room_notification_targets` already does. Then fix the test fixture to exercise the real join path instead of hand-inserting `room_memberships`.
- Effort: M
- Risk of change: medium

- ID: notifications-2
- Severity: high
- Confidence: certain
- Location: server/chat/chat.html:1858-1864, 1890-1898 (`_toggleMeetupGoing`), server/chat_api.py:1267-1286, contrast with server/chat/chat.html:1870-1888 (`_toggleRoomMembership`)
- Finding: The meetup list bell button is explicitly labeled "Get notified" / "Mute meetup" and wired to `_toggleMeetupGoing`, which only calls `POST/DELETE /chat/api/meetups/{id}/join`. Unlike the equivalent room bell (`_toggleRoomMembership`), which also sends a live `wsSend('join_room'|'leave_room', ...)` to update the connected session's state, `_toggleMeetupGoing` sends nothing over the WS. Combined with notifications-1, toggling this bell "on" has literally no effect on whether the user is ever notified — the UI makes an explicit promise it cannot keep, even after the accounting bug is fixed, because the current WS session's `manager.user_badge_rooms` won't include the meetup room until the user separately opens it (which is what actually triggers the `join_room` event that adds it, per chat_ws.py:1399-1400).
- Recommendation: Once notifications-1 is fixed, also emit a WS signal (or reuse `join_room`/`leave_room`) from the "going" toggle so a live session's badge/notification state updates immediately rather than only on the next room open or reconnect.
- Effort: S
- Risk of change: low

- ID: notifications-3
- Severity: high
- Confidence: certain
- Location: server/chat_ws.py:1748-1824 (`create_meetup` handler), contrast with server/chat_ws.py:1146-1200 (`send_message` badge/push targeting)
- Finding: When a meetup is created with a `stage_id`, the invite card is posted via a direct `create_message()` + `manager.broadcast_to_room()` call inside the `create_meetup` WS handler. This completely bypasses the badge/push machinery that every ordinary message goes through in `send_message` (the `badge_update` fan-out loop over `mgr.user_badge_rooms`, and the `_get_room_notification_targets` + `_push_or_defer` push targeting). So even *regular room members* of the target room (not just meetup attendees) get no live badge bump and no push when a meetup invite lands in their room — they'll only see it if they already have the room open, or retroactively on their next reconnect (since `get_unread_counts` re-derives from the `messages` table directly).
- Recommendation: Route the meetup-invite message through the same targeting/push/badge code path as `send_message` (factor it into a shared helper called by both), instead of a bespoke `create_message` + bare broadcast.
- Effort: M
- Risk of change: medium

- ID: notifications-4
- Severity: medium
- Confidence: certain
- Location: server/chat_ws.py:1825-1832 (`meetup_created`), 1834-1866 (`meetup_updated`), 2195-2205 (`meetup_expired`); server/chat/chat.html:1506-1510 (`meetup_expired` client handler)
- Finding: `meetup_created`, `meetup_updated`, and `meetup_expired` are pure `broadcast_to_room` events with no push fallback — they reach only clients with a live, currently-connected WebSocket subscribed to that specific room. An attendee whose app is closed/backgrounded gets no signal when a meetup they're going to is announced, when the attendee list changes, or — most importantly — when their meetup is cancelled/expires and its room+messages are deleted out from under them. On the client, even a foreground user elsewhere in the app gets no toast/snackbar for `meetup_expired`; the handler only dims an invite-card DOM element if that exact card happens to be rendered in the currently-open room, so most users see nothing at all.
- Recommendation: Send a (debounced, reusing `_push_or_defer`'s existing dedup) push to attendees for meetup expiry/cancellation at minimum, since it destroys their room; add a client-side toast for `meetup_expired` that fires regardless of which room is currently open.
- Effort: M
- Risk of change: low

- ID: notifications-5
- Severity: medium
- Confidence: likely
- Location: server/chat_ws.py:827-836 (`check_rate_limit`), :1749 (`create_meetup` WS), server/chat_api.py:1238 (`create_meetup_endpoint`)
- Finding: Meetup creation shares the same generic rate bucket as ordinary chat messages (`check_rate_limit(user_id)`, default 10 events / 10 seconds), both over WS and REST. A meetup is a much heavier, more disruptive unit than a text message — it creates a new room, an attendee record, and (intended to be, per notifications-3) a room-wide broadcast invite card — yet nothing throttles meetup creation more tightly than plain chat. A user can create up to 10 meetups in a 10-second window, each targeting a different `stage_id`, flooding up to 10 invite cards into a room's live view in seconds. There is also no cap on how many meetups a single user can have active concurrently.
- Recommendation: Give `create_meetup` its own, stricter rate limit (e.g. 1 per 30–60s) and/or a small per-user cap on concurrently active meetups, independent of the general chat-message bucket.
- Effort: S
- Risk of change: low

- ID: notifications-6
- Severity: low
- Confidence: speculative
- Location: server/chat/chat.html:1685 (`_toggleRoomMembership` room bell) vs :1864 (`_toggleMeetupGoing` meetup bell)
- Finding: The only way to stop receiving meetup-invite-card noise in a given room is to leave that room entirely via the room-level "mute" toggle (which silences everything, not just meetup invites). There's no finer-grained way to opt out of meetup invites specifically. This is low-impact today only because notifications-1/3 mean meetup content doesn't currently reach push/badges at all — but it becomes a real spam-exposure gap the moment those bugs are fixed, especially combined with the weak per-meetup rate limiting in notifications-5.
- Recommendation: Low priority; revisit if/when notifications-1 and notifications-3 are fixed and meetup push volume becomes a real user complaint. Consider deprioritizing meetup_invite in the push debounce/preview path rather than building a dedicated opt-out.
- Effort: S
- Risk of change: low
