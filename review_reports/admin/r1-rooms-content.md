## Findings

### F1. Read-only rooms have no admin-post bypass — "only admins can post" is not implemented [severity: high] [kind: missing-feature]
Evidence: `server/chat_ws.py:1552-1562` (message-send handler rejects when `send_room["is_read_only"]` with zero exception for identity); `server/chat_api.py:245-268` (`_require_admin` — admin identity is checked only via `X-Admin-Token` header or an admin-email session cookie, and this function is called *only* from the `/chat/api/admin/*` routes, never referenced anywhere in `chat_ws.py`). There is also no `/chat/api/admin/rooms/{id}/messages` POST endpoint or any other admin path that calls `create_message`.
What happens: A read-only room rejects every sender, including a user whose session happens to belong to a `CHAT_ADMIN_EMAILS` address — the WS handler has no notion of "admin" at all. CLAUDE.md documents read-only rooms as "only admins can post," but no code path lets an admin actually post. The only way to get a message into a read-only room today is to flip `is_read_only` off, send, then flip it back on — a manual race-prone workaround.
Proposed change: either add an admin-post endpoint (`create_message` + broadcast, reusing `_moderate_and_broadcast`/skip-moderation semantics) gated by `_require_admin`, or have the WS handler check the connected user's identity against `CHAT_ADMIN_EMAILS` before enforcing `is_read_only`.

### F2. Admin Rooms tab "Members" column is hardcoded to 0 for every room [severity: high] [kind: bug]
Evidence: `server/chat_db.py:1734-1762` `get_room_stats()` returns `"member_count": 0` as a literal, never queried; compare to `get_reachable_member_counts()` (`chat_db.py:1473-1492`), which computes the real reachable-member count and *is* used by the user-facing `/chat/api/rooms` endpoint (`chat_api.py:853,874`) but is never called from the admin path (`chat_api.py:2117-2132` `admin_rooms`).
What happens: every row in the admin Rooms table shows "Members: 0" (`admin.html:376`), regardless of actual membership — the stat is entirely fake. An admin can't use this to judge room reach/audience size.
Proposed change: call `get_reachable_member_counts(db, [r["id"] for r in rooms])` in `admin_rooms` and merge into `get_room_stats`'s output the same way `online_counts` is merged.

### F3. Editing a room whose TTL isn't one of the six preset values silently rewrites it to 30 minutes [severity: medium] [kind: bug]
Evidence: `admin.html:667-677` — the `<select id="rm-ttl">` only has options for 30/60/360/1440/2880/4320/null; none of the `${room?.ttl_minutes === X ? 'selected' : ''}` checks fire for any other value, and there's no catch-all/custom option. `saveRoom()` (`admin.html:703-712`) reads `document.getElementById('rm-ttl')?.value` unconditionally and sends it as `ttl_minutes` in the PATCH body.
What happens: if a room's `ttl_minutes` is e.g. 120 (set via a future feature, a direct `PATCH` API call, or a changed `room_ttl_minutes` default that isn't in the six presets — see F4), opening its edit modal renders no option as selected, so the browser defaults to the first `<option>` (30 minutes). Clicking Save without touching the TTL field silently rewrites the room's TTL to 30m.
Proposed change: add a synthetic option when `room.ttl_minutes` doesn't match any preset (e.g. `<option value="${room.ttl_minutes}" selected>${room.ttl_minutes}m (custom)</option>`), or disable Save-without-touch by diffing against the loaded value.

### F4. No admin UI (or `/admin/settings` support) for `msg_char_limit`, `dm_ttl_minutes`, `room_ttl_minutes`, `meetup_ttl_minutes` — contradicts CLAUDE.md [severity: medium] [kind: missing-feature/inconsistency]
Evidence: `chat_api.py:2018-2040` — `admin_get_settings`/`admin_update_settings` only read/write `room_sort`; `admin.html` has no field referencing any of the other four `chat_settings` keys (grep for `msg_char_limit`/`dm_ttl_minutes`/`room_ttl_minutes`/`meetup_ttl_minutes` in `admin.html` returns nothing). CLAUDE.md ("Chat UI") explicitly states "Change in DB, no deploy needed" for `msg_char_limit`.
What happens: despite the doc's framing, changing the char limit or any default TTL requires a raw `sqlite3` edit to `chat_settings` — there is no admin-panel path at all, and doing so risks re-introducing F3 for future room edits.
Proposed change: add a small "App Settings" panel section with these four fields, backed by extending `admin_update_settings`'s allowlist.

### F5. `PATCH /admin/rooms/{room_id}` and `POST /admin/rooms` accept unvalidated field types and untyped `type`/room ids not restricted to DM/meetup exclusion [severity: medium] [kind: bug]
Evidence: `chat_api.py:2317-2332` (`admin_update_room` forwards raw `await request.json()` into `update_room(db, room_id, **body)` with zero type checks) and `chat_db.py:732-755` (`update_room` only filters by *key name*, not by value type — `ttl_minutes`, `name`, `description`, `position` are inserted as-is). Also `chat_api.py:2279-2314` (`admin_create_room`) takes `body.get("type", "general")` with no allowlist (UI only offers `general`/`stage`, but the endpoint accepts any string, e.g. `"dm"` or `"meetup"`, producing a phantom room row not linked to any `dm_participants`/`meetups` record).
What happens (FACT: no validation exists; INFERENCE: downstream crash) — e.g. sending `ttl_minutes: "soon"` via the PATCH body stores a non-numeric string in an `INTEGER` column (SQLite permits this); the next message sent in that room calls `datetime.now() + timedelta(minutes=ttl_minutes)` in `create_message` (`chat_db.py:951-955`), which raises `TypeError` for a non-numeric value, breaking that room's message sending until manually fixed. Similarly, nothing stops `PATCH /admin/rooms/{dm_or_meetup_id}` (the endpoint itself never checks `room["type"]`) from setting `is_moderated: true` on a DM room — the UI hides the Edit button for DM/meetup rows (`admin.html:381`), but the API has no matching server-side guard.
Proposed change: validate `type` against `{"general","stage"}` in `admin_create_room`; validate `ttl_minutes`/`position` are int-or-None and `name`/`description` are strings in `admin_update_room`; reject PATCH targeting `type in ("dm","meetup")` server-side (mirroring the existing DELETE guard at `chat_api.py:2386-2387`).

### F6. Rogue meetups (offensive titles) cannot be removed from the admin panel — and meetup titles skip all moderation [severity: medium] [kind: missing-feature]
Evidence: `chat_ws.py:1748-1793` (`create_meetup` WS handler) never calls `moderate_message`/word-filter/AI on `title`/`note` before `create_meetup(db, ...)` — only the invite *message* content is moderated implicitly through the normal `create_message`+broadcast flow, and even that (`meetup_invite` type at line 1790) is inserted directly, not routed through `_moderate_and_broadcast`. `chat_api.py:2376-2394` (`admin_delete_room`) explicitly raises 400 for `room["type"] in ("dm", "meetup")` — "DM and meetup rooms are managed automatically." `admin.html:381-382` hides Edit/Delete for `r.type === 'meetup'` rows entirely. No `/admin/meetups*` route exists anywhere in `chat_api.py`.
What happens: a meetup title is never scanned by any of the three moderation layers, and once created it is visible in the admin Rooms list (as a `type=meetup` row, since `get_room_stats` doesn't filter by type) but cannot be edited or deleted — the only way to force it gone is deleting the *creator's entire user account* (`delete_user` in `chat_db.py:514-530` does clean up rooms/meetups created by that user), which is a disproportionate action that also wipes the user's whole chat history/identity. There's no lightweight "delete this meetup" control.
Proposed change: run `title`/`note` through the word filter (and ideally the AI layers) at meetup creation; add an admin meetup list + delete-meetup endpoint that tears down the meetup row + its room, mirroring `delete_room`'s media cleanup.

### F7. No way to remove/inspect a single message or media file from the admin panel [severity: medium] [kind: missing-feature]
Evidence: grep across `chat_api.py` for admin+message routes returns nothing; the only message-delete path is the WS `delete_message` event (`chat_ws.py:1986-2035`), gated by `msg_row["user_id"] == user_id` (sender-only) and a 120s window — no admin override. The only server-controlled bulk removal paths are `delete_user_messages` (deletes *all* of a user's active messages, used by mute/strike/ban/delete-user) and `delete_room` (deletes an entire room's history). There is also no endpoint to browse `chat/uploads/` or unlink a specific file.
What happens: if one image/video slips past moderation, an admin's only levers are "delete every message this user has sent everywhere" or "delete the whole room." There's no scalpel — no way to purge a single message or a single stuck upload without collateral damage to unrelated messages/users in the same room.
Proposed change: add `GET /admin/rooms/{id}/messages` (paginated, admin-only, bypassing the `moderation_status != 'pending'` filter so admins can see everything) and `DELETE /admin/messages/{id}` that reuses the existing `_unlink_media_if_orphaned` cleanup path.

### F8. DM rooms are listed in the admin Rooms tab as indistinguishable "DM" rows with no participant identity [severity: low] [kind: ux]
Evidence: `chat_db.py:1306` `create_room(db, room_id, event_id, "dm", "DM", ...)` — every DM room's `name` is literally the string `"DM"`; `get_room_stats` (`chat_db.py:1734-1762`) doesn't join `dm_participants` to surface who's in it. `admin.html:373` renders `r.name` verbatim.
What happens: an admin scanning the Rooms tab sees N rows all named "DM," type `dm`, member_count 0 (F2), with no way to tell which two users they belong to — functionally useless for oversight, and (correctly, per E2EE) can't be opened/edited/deleted, but the total absence of any identifying info makes the row noise rather than useful. This is expected to some degree given E2EE (server intentionally can't show content), but participant *names* are not secret and could be shown.
Proposed change: either exclude `type='dm'` rows from the admin Rooms table entirely (they can't be acted on anyway), or join in the two `dm_participants` display names for identification-only purposes.

### F9. `admin/rooms/reorder` includes DM/meetup rows in the drag list and has no id/existence validation [severity: low] [kind: inconsistency]
Evidence: `admin.html:370` — every `<tr>` (including `r.type === 'dm'`/`'meetup'`) gets `draggable="true"`; `onDrop` (`admin.html:770-784`) builds `order` from all rows in the tbody, unconditionally. `chat_api.py:2356-2373` `admin_reorder_rooms` loops `for i, room_id in enumerate(order): UPDATE rooms SET position = ? WHERE id = ?` with no check that `room_id` exists (a bad id silently no-ops) and no filtering by type.
What happens: dragging a general/stage room past a DM/meetup row reassigns `position` to DM/meetup rows too. This is currently harmless in practice because the user-facing sort (`get_rooms_by_event`, `chat_api.py:851`) already filters to `type IN ('stage','general')`, so DM/meetup `position` values are dead weight — but it's fragile: any future code that sorts by `position` without that type filter would silently misbehave.
Proposed change: exclude `dm`/`meetup` rows from `draggable` in the UI, and have `admin_reorder_rooms` validate each id belongs to `type IN ('general','stage')` before writing.

### F10. Room-name → room_id slug has no character allowlist [severity: low] [kind: bug]
Evidence: `chat_api.py:2287` `room_id = name.lower().replace(" ", "-")` — only lowercases and swaps spaces for hyphens; no stripping of `/`, `?`, `#`, unicode punctuation, emoji, etc.
What happens: a name like `"Q&A / Help"` slugs to `"q&a-/-help"`, which contains a literal `/`. Since `room_id` is used directly in URL paths (`/chat/r/{id}`) and as an uploads-adjacent filesystem-free identifier, a slash-containing id breaks routing (`/chat/r/q&a-/-help` is ambiguous/malformed) with no server-side rejection.
Proposed change: slugify with an allowlist (`[a-z0-9]+(-[a-z0-9]+)*`), collapsing/stripping everything else, and reject (400) if the result is empty.

### F11. Stats bar mixes pending/unmoderated messages into "Messages," and computes but never displays `total_rooms` [severity: low] [kind: inconsistency]
Evidence: `chat_db.py:1548-1575` `get_admin_stats` — `total_messages_active` is `SELECT COUNT(*) FROM messages WHERE expires_at > ?` with no `moderation_status != 'pending'` filter (unlike every user-facing query, e.g. `get_room_messages` at `chat_db.py:998-999`) and no exclusion of DM content; `total_rooms` is computed but `renderStatsHTML` (`admin.html:230-238`) never renders it.
What happens: the "Messages" stat can be inflated by messages still awaiting moderation (which will never be shown to any user) — mildly misleading as a health metric. `total_rooms` is wasted computation with no UI consumer.
Proposed change: add `AND moderation_status != 'pending'` to the count, and either surface `total_rooms` in the stats bar or drop it from the query.

## Feature ideas (not bugs)
- Admin-post-to-room composer (ties into F1) — a simple text box + send button in the room edit modal for read-only/announcement rooms.
- A "custom TTL" numeric input alongside the preset dropdown, avoiding the F3 trap entirely and letting admins set arbitrary per-room TTLs without DB access.
- Meetup admin tab: list active meetups (title, creator, time, attendee count), with a delete action — directly addresses F6.
- Room detail drill-down: click a room row to see live message count broken down by pending/approved, plus a "view recent messages" pane (addresses F7's read side).

## Uncertain / needs orchestrator verification
- F3's exact browser behavior (no `<option selected>` defaulting to the first option) is standard HTML `<select>` semantics, but I did not run this in a browser — worth a quick manual check with a room whose `ttl_minutes` is set to e.g. 120 directly in the DB.
- F5's crash inference (`TypeError` from `timedelta(minutes=<non-numeric str>)`) is based on reading `create_message`'s ternary — not executed. Worth a targeted `pytest`/manual repro: `PATCH /admin/rooms/general` with `{"ttl_minutes": "bad"}` then send a message in `general`.
- Whether `CHAT_ADMIN_EMAILS` users are expected by design to also be regular chat users (which would make F1's fix straightforward via a `user_providers` lookup) — this overlaps with admin-auth design, which is out of my lane; flagging only the functional gap.
- F9's "harmless in practice" claim rests on `get_rooms_by_event`'s type filter being the only consumer of `position` for sort ordering — I did not exhaustively check every place `rooms.position` might be read (e.g. any client-side code, stress tests, or future admin features).
