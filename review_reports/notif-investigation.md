# Findings

## Symptom 1: no desktop push in backgrounded Brave

- **Code path**: `send_message` → `_moderate_and_broadcast` (`server/chat_ws.py:759`, invoked with `is_moderated=False` for DMs, so it still reaches the push section) → push-eligibility check `server/chat_ws.py:912-919` → `_push_or_defer` (`chat_ws.py:309`) → `_do_send_push` (`chat_ws.py:362`) → `pywebpush`. Idle signal: client `sendBeacon('/chat/api/push/idle')` on `visibilitychange`/`pagehide` (`server/chat/chat.html:4508-4518`) → `POST /chat/api/push/idle` (`server/chat_api.py:1484-1491`) → `manager._last_ws_activity[user_id] = 0`.

- **Defect**: `_last_ws_activity` is a `dict[str, float]` keyed by **`user_id`**, not by connection/device (`chat_ws.py:523`, set in `connect()` at `chat_ws.py:538-542`). Every one of a user's connections shares one timestamp. It is refreshed by: any new WS connect (`chat_ws.py:542`), any user-initiated event on **any** device (`chat_ws.py:1076`), and critically the passive 20s-interval `'visible'` keepalive ping sent by any foregrounded tab (`chat_ws.py:1167-1168`, client side `chat.html:4498-4507`). The push-eligibility test at `chat_ws.py:918` is `uid not in connected_uids or now - last_ws_activity(uid) > 30`. Because Gabbo has two simultaneous connections (Brave + iOS PWA), if the iOS PWA is foregrounded and pinging `'visible'` every 20s (or simply just (re)connected), it continuously resets the **shared** timestamp for `user_id=Gabbo`, masking the fact that Brave individually went idle and called `sendBeacon`. Result: `now - last_ws_activity > 30` evaluates false, Gabbo never enters `push_targets`, and **no push is sent to any of Gabbo's subscriptions**, including Brave's. This is unrelated to the reviewed diff — `chat_ws.py`'s only change in `notif-regression.diff` is the `max_content` length check (diff hunk `@@ -1195,8 +1195,17 @@`); the idle-tracking and push-payload code is untouched. It is a pre-existing architectural gap (per-user, not per-device, activity tracking) that this multi-device E2EE test setup happens to expose clearly.

- **Confidence**: medium. This fully explains the reported behavior *if* Gabbo's iOS PWA had an active/foregrounded WS connection at the same time Brave was backgrounded. It's in tension with the stated premise that the message "arrived live in both Brave and iOS PWA" for symptom 1 while a push also separately arrived on iOS for symptom 2 — those may be two different test passes rather than one simultaneous state, which I can't resolve statically.

- **Falsification test**: reproduce with Brave backgrounded and iOS PWA also present but *not sending `'visible'` pings* (e.g., iOS PWA closed) — push to Brave should then succeed. Reproduce again with iOS PWA open/foregrounded at the same time — push to Brave should fail. Check server log line `[PUSH] targets=%d all=%d connected=%d sender=%s` (`chat_ws.py:920-926`) at message-send time: if `targets=0` while `all=1`, this hypothesis is confirmed (Gabbo was filtered out by the idle check, not absent from `all_targets`).

- **Fix direction**: track `_last_ws_activity` per `(user_id, conn_id)` and compute push eligibility per-connection/subscription rather than per-user; a device should only be excluded from push if *that device's* WS is fresh, not because a sibling device pinged recently.

## Symptom 2: iOS notification tap lands on Line-up

- **Code path**: server push payload built in `_do_send_push` (`chat_ws.py:362-449`), `url = f"/chat/msg/{first_msg_id}"` (`chat_ws.py:436`) → SW `push` handler builds unique `tag` and `showNotification` (`server/static/sw.js:25-57`) → SW `notificationclick` writes target URL to `caches.open('stc-push')` then `postMessage`/`focus()`/`openWindow()` (`sw.js:59-86`) → page-side listener does a **full reload** via `window.location.href` in both `chat.html:4494-4496` and `index.html:392-395` → fresh load runs `route()` (`chat.html:4344`), resolves `pathMsg` via `GET /chat/api/messages/{id}` and `openRoom()` (`chat.html:4381-4394`).

- **Defect**: I could not find any change in `notif-regression.diff` that touches `first_msg_id` construction, the SW, or the navigate listeners — `sw.js` has zero diff hunks, and `_do_send_push`/URL/tag logic is byte-identical before/after. The most plausible remaining cause is the pre-existing iOS quirk already called out in `CLAUDE.md` ("openWindow() silently returns null when a window already exists"): if iOS retains a suspended/killed-but-still-registered client reference for the PWA, `clients.matchAll()` at `sw.js:70` may return an empty list, falling to `openWindow()` (`sw.js:77`), which can silently no-op on iOS, causing the OS to instead cold-launch the PWA at its manifest `start_url` (Line-up) — completely bypassing the SW's own navigation logic. This is a residual gap in the *previously applied* tag-uniqueness fix, not something touched by the E2EE work.

- **Confidence**: low. This is inference from documented iOS platform behavior, not something I can confirm by reading this diff — the diff simply doesn't touch any file in this path.

- **Falsification test**: check `[SWLOG]` `click-done` log lines (`POST /chat/api/swlog`, logged via `sw.js:82-84` → `chat_api.py:1494-1501`) for the failing tap. If the result string is `openwindow-failed:...` or `openwindow:null`, that confirms the `openWindow()` no-op theory. If instead `postmessage:...` appears (meaning a client *was* matched and messaged), the defect is downstream in that client's own state, not `openWindow()` — worth also checking whether `_pendingScrollMsg`/`ctx.room_id` resolution throws (`chat.html:4381-4394`, `catch` branch at 4388-4394 falls back to `renderRoomList()`/main room on **any** failure of `GET /messages/{id}`, e.g. if the encrypted message's DB row was already purged or `expires_at` elapsed faster than expected for E2EE DMs — TTL is unaffected by encryption per `CLAUDE.md`, so this is unlikely but worth ruling out via logs).

- **Fix direction**: instrument (or re-verify) the `openWindow()` fallback path specifically for iOS with a suspended-but-terminated PWA process; consider `clients.matchAll({type:'window', includeUncontrolled:true})` returning a stale/uncontrollable client reference as a known false-positive and add a fallback `openWindow()` retry when `postMessage` produces no visible effect within a timeout.

## Symptom 3: missing unread dot/badge in PWA after cold start

- **Code path**: server sends `badge_counts` right after WS connect/auth (`chat_ws.py` handler, ~line 1040-1054, populated from `get_unread_counts()` in `chat_db.py:675-705`, which correctly includes DM rooms via the `dm_participants` UNION at `chat_db.py:683-686`). Client `route()` (`chat.html:4344`) sequence: `checkAuth` → `_checkPushStatus` → `connectWS()` (`chat.html:4373`) → fire-and-forget `E2EE.init()/uploadPublicKey()` (`chat.html:4374-4376`) → `await loadBlockedUsers()` (`chat.html:4378`) → `await loadRooms()` (`chat.html:4379`) → then resolves the target room/DM list.

- **Defect (primary)**: `loadRooms()` (`chat.html:1547-1558`) fetches `GET /chat/api/rooms`, which server-side only returns rooms of `type IN ('stage','general')` (`chat_db.py:610-614` `get_rooms_by_event`) — **DM and meetup rooms are never included**. `loadRooms()` then prunes `unreadByRoom`:
  ```
  const validIds = new Set(rooms.map(r => r.id));
  for (const rid of Object.keys(unreadByRoom)) {
    if (!validIds.has(rid)) delete unreadByRoom[rid];
  }
  ```
  (`chat.html:1553-1556`). Since DM room ids are never in `validIds`, **any DM unread count already populated by the `badge_counts` WS message is deleted the moment `loadRooms()` runs**, as long as `badge_counts` (pushed immediately on WS connect) arrives before or during the two sequential `await`ed REST calls (`loadBlockedUsers()`, `loadRooms()`) that follow `connectWS()` in `route()`. This is a straight race: whichever arrives second (the WS push vs. the `loadRooms()` prune) wins, matching the reported "unreliable... does not always appear." When the user later opens the DMs tab, `loadDMs()` (`chat.html:1836-1865`) computes the badge purely from `unreadByRoom[dm.room_id] || 0` (`chat.html:1851`) — if it was wiped, no badge renders, and no code re-fetches it.

- **Defect (secondary, compounding)**: even when `unreadByRoom` does hold correct DM counts and a *live* `badge_counts` refresh (e.g. WS reconnect) arrives after the DM list is already rendered, `refreshAllBadges()` (`chat.html:1748-1751`) only queries `document.querySelectorAll('.room-item[data-room-id]')`. DM rows are rendered with class `member-item`, not `room-item` (`chat.html:1856`), so `refreshAllBadges()` never reaches them — only `updateBadge()` called directly from the `badge_update` per-message handler (`chat.html:1498`, no class scoping) can update a DM row in place. This means the bulk `badge_counts` sync path can never repair a stale/incorrect DM badge already on screen.

- **Confidence**: high for the primary defect — it is a clean, reproducible logic bug independent of timing luck once you understand the race, and it precisely matches "unreliable... does not always appear" and "cold start with no dot." Note: neither of these two functions (`loadRooms`, `refreshAllBadges`) appears in `notif-regression.diff`, so this predates the E2EE work and is not something the diff introduced — it's a latent bug that the E2EE-focused testing session happened to surface.

- **Falsification test**: add/inspect `dbg()` timing — confirm whether `[BADGE] initial counts:` (`chat.html:1472`) log fires *before* `[ROOM] loadRooms` completes on a cold PWA launch with an unread DM waiting. If `unreadByRoom[dmRoomId]` is set and then immediately removed by the `loadRooms()` prune loop, this is confirmed. Also confirm server-side: `get_rooms_by_event` (`chat_db.py:612`) query indeed excludes `type='dm'` rows (static, already verified by reading).

- **Fix direction**: don't prune `unreadByRoom` using the `/rooms` (stage/general-only) id set — either fetch a full membership id set (rooms + DMs + meetups) for pruning, or don't prune DM/meetup entries from this generic list at all (they're naturally cleaned up via `mark_read`/room deletion elsewhere). Separately, fix `refreshAllBadges()`'s selector to also cover `.member-item[data-room-id]` (DM rows) and `.meetup-item[data-room-id]` if applicable.
