# Findings: e2ee

## [SEVERITY: HIGH] Uploaded media for E2EE image/video DMs is never deleted from disk
- Where: `server/chat_db.py:858-892` (`purge_expired_messages`), `server/chat_ws.py:1613-1651` (`delete_message` handler)
- Evidence: `purge_expired_messages` extracts the file to clean up via `content = json.loads(msg["content"]); if "url" in content: image_paths.append(content["url"])` (chat_db.py:873-875). A v2 E2EE envelope is `{"e2ee": true, "v": 2, "sd": ..., "ct": ..., "keys": {...}}` — there is no top-level `"url"` key (the real URL lives inside encrypted `ct`), so `"url" in content` is always `False` for encrypted image/video messages. The same pattern repeats in the manual delete handler: `url = json.loads(msg_row["content"]).get("url", "")` (chat_ws.py:1631) resolves to `""` for an E2EE envelope, then `(_UPLOADS_DIR / filename).unlink(missing_ok=True)` with `filename=""` targets `_UPLOADS_DIR` itself (empty path components are absorbed by `pathlib` joins), raises `IsADirectoryError`, and is silently swallowed by the surrounding `except Exception: pass` (chat_ws.py:1640-1641).
- Impact: Every image/video sent in an encrypted DM leaves its actual file (served unauthenticated at `/chat/uploads/{filename}`, see `chat_api.py:2133-2151`) on disk **forever** — neither the 60/1440-minute message TTL purge nor manual "delete message" ever removes it. This (a) causes unbounded disk growth in production, and (b) breaks the app's own ephemerality/privacy model for encrypted media: `docs/e2ee-dev.md` §5.4 explicitly frames the design as "encrypting the URL... prevents third parties with DB access from linking a media file to a conversation" — it does not anticipate the file surviving past the TTL window at all. A URL leaked once (referrer, browser cache, shared screenshot, log line) grants indefinite access to what users believe is an ephemeral, encrypted photo/video.
- Fix: Have the client include the (encrypted) media reference in a way the server can still garbage-collect — e.g., store the plaintext upload URL server-side alongside the encrypted envelope in a dedicated column populated at upload time (server already assigns/handles the file, per docs/e2ee-dev.md §5.4 — it isn't secret from the server), and have both cleanup paths read that column instead of parsing `content["url"]`.

## [SEVERITY: LOW] Ambiguous key-fetch failures are treated as "no key" and silently fall back to plaintext for never-yet-encrypted DMs
- Where: `server/chat/chat.html:4285-4291` (`_fetchDeviceList`), `server/chat/chat.html:2492-2521` (`sendChatMessage`)
- Evidence: `_fetchDeviceList` correctly distinguishes 404 (`return null`) from other failures (`throw new Error(...)`), but the caller in `sendChatMessage`'s `catch` block (line 2511-2521) reacts identically to a thrown error and to `encrypt()` returning `null`: if `_roomEncrypted.has(roomId)` is false, it adds the room to `_unencryptedRooms` and proceeds to send **plaintext** (`outgoingContent` stays as the unmodified `content`).
- Impact: A transient 5xx/network failure on `GET /chat/api/keys/{peer}` — not just a genuine "peer has no key" 404 — can cause the first message of a brand-new DM to go out unencrypted even though the peer does have registered device keys, since the client cannot tell the two cases apart. Narrow window (self-heals on the next message once `_roomEncrypted`/cache succeed, and requires a non-404 failure precisely on the first send), and outside the stated threat model (requires either a flaky server or genuine MITM, both already discounted by the "server operator honest, transport is HTTPS" assumptions) — hence LOW rather than MEDIUM/HIGH.
- Fix: In `sendChatMessage`'s catch, only apply the plaintext-fallback + `_unencryptedRooms` path when the failure is specifically "no key registered" (i.e., propagate the 404/null distinction through the thrown error), and otherwise abort the send with an error toast (same as the `_roomEncrypted` branch).

## [SEVERITY: LOW] Server-side JWK validation does not check the public key lies on the P-256 curve
- Where: `server/chat_api.py:1535-1564` (`_validate_e2ee_jwk`)
- Evidence: Validation checks `kty == "EC"`, `crv == "P-256"`, absence of `d`, and that `x`/`y` decode to exactly 32 bytes — it never verifies `(x, y)` is actually a point on the P-256 curve.
- Impact: A malicious/compromised account could upload an off-curve or invalid point as its own device key. This is only exploitable if a peer's browser `crypto.subtle.importKey('jwk', ...)` fails to reject invalid-curve points during import — modern WebCrypto implementations do perform this validation (BoringSSL/NSS reject non-curve points at import), so real-world exploitability is low, and this mirrors v1's validation scope unchanged (not a v2 regression).
- Fix: Optional defense-in-depth — validate curve membership server-side (e.g., via a lightweight EC point-on-curve check) before accepting a key, so a browser with a laxer WebCrypto implementation isn't the last line of defense.

## [SEVERITY: LOW] `GET /chat/api/keys/{user_id}` has no relationship/authorization scoping
- Where: `server/chat_api.py:1622-1641`
- Evidence: The endpoint only requires `_get_user_from_cookie(request)` to succeed (any authenticated chat user) — it does not check that the caller is actually a DM participant with `user_id`.
- Impact: Any logged-in user can enumerate any other user's device count and device `created_at` timestamps, not just their own DM partners. The design doc accepts device-count metadata as leaked info *from envelope inspection*, but this widens it to open enumeration across the whole user base. Low severity since public keys and coarse device-count metadata are explicitly non-secret in this threat model, and this is inherited unchanged from v1 (not a v2 regression).
- Fix: Optional — restrict to users who share at least one DM/room with the target, if minimizing metadata exposure is desired.

## Verified clean

- **Crypto correctness** (checklist #1): P-256 ECDH + HKDF-SHA256 + AES-256-GCM used correctly on both wrap and content layers; `mk`, `iv1`, and each `ivD` all sourced from `crypto.getRandomValues` (no `Math.random`); GCM tag handling relies on Web Crypto's built-in append/verify, no custom tag logic; self-slot `ECDH(own_private, own_public)` is mathematically valid and exercised on reload.
- **Envelope handling / logging** (checklist #2): grepped all `logger.*` calls touching message content — none log raw `content`; `[MOD] text=%r` logs the already-E2EE-blanked `text_for_moderation`, not the envelope; the one place raw request bodies are logged (`/chat/api/swlog`) is an unrelated SW-navigation diagnostic endpoint.
- **Server content-gating completeness** (checklist #3) for moderation, push preview (both the immediate and debounced-refetch paths), reply snippets, link previews, duplicate-check, and reports: all correctly gate on `_is_e2ee_content` and produce empty/generic values for encrypted DM content. (Media-file cleanup is the one path that was missed — see HIGH finding above.)
- **Key registration authz** (checklist #4): `PUT /chat/api/keys` always writes under the session-derived `user["id"]`, never a client-supplied user id — no cross-user key registration is possible. Device cap/pruning logic cannot evict a device that was just inserted (ordered by `last_seen DESC`, newest kept) and only ever prunes the authenticated user's own rows.
- **Downgrade attacks** (checklist #5): no delete-key endpoint exists; `key_rotated` cannot be spoofed by a client (the WS dispatch loop is a fixed whitelist of event names with no reflect/relay path); an attacker-supplied `v:2` envelope with an empty `keys` map renders the sentinel, not plaintext. Fallback state is clearly surfaced via toast + banner + suppressed lock icon.
- **Cross-device / rotation** (checklist #6): sender always wraps for all of the peer's AND its own devices (including itself), own-device injection covers the upload race, `key_rotated` invalidation is bulk (whole device-list + all KEKs) and simple to reason about, sender-device-pruned messages fail closed to the sentinel via the existing retry guard.
- **localStorage / XSS surface** (checklist #7): decrypted DM plaintext flows through the same `renderMessage`/`_linkify`/`escapeHtml` pipeline as all other messages; text, reply quotes, and meetup titles are HTML-escaped before insertion; the one unescaped interpolation (`location` card lat/lng into an `href`) is type-guarded by `.toFixed()` throwing on non-numeric JSON values before reaching the template, and is unchanged/pre-existing across all message types, not E2EE-specific.
