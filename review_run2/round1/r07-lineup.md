## MEDIUM

- **scraper/render.py:920-922, 2425-2427** — `_link()` (Python) and `_popupLink()` (client JS) build `<a href="...">` using only `esc(href)` (HTML-entity escaping), with no URL-scheme allowlist. Scraped URLs are safe because `scrape.py`'s `is_valid_url()` restricts every scraped link to `http`/`https` (scrape.py:34-44, enforced at every `upsert_artist_link` call site). However, `scraper/db.py`'s `apply_overrides()` (db.py:290-301) writes the `PLATFORM_LINK_FIELDS` value straight into `artist_links.url` with **no scheme check at all**. A `soundcloud = "javascript:alert(document.cookie)"` entry in `scraper/overrides.toml` — a plain-text file checked into the repo and explicitly documented as an editable manual-correction file — renders as a live, clickable `javascript:` link on the public lineup and timetable pages with zero defense at the render layer. Since render.py is the last line of defense before output, it should validate the scheme (reject non-http(s)) regardless of what upstream data hygiene guarantees.

- **scraper/render.py:156-178 (`_render_markdown`)** — Bio HTML sanitization is a hand-rolled blacklist of regexes (strip `script/iframe/object/embed/form` tags, `on\w+=` attributes, `javascript:` hrefs) applied to markdown output that allows raw HTML passthrough (python-markdown doesn't sanitize by default). This is bypassable by design of regex-based sanitization: an HTML-entity-encoded scheme such as `href="&#106;avascript:alert(1)"` is decoded to `javascript:` by the browser at attribute-parse time, but the regex's literal-substring match on `javascript:` never sees it in the raw markdown output, so it survives unstripped. `xlink:href` on SVG `<a>` elements is another vector the regex never inspects. Bio text originates from RA-scraped artist profiles (partially artist-controlled) and from `overrides.toml`, and is injected via `bioText.innerHTML = data.bio` (render.py:2121) into the public bio modal — i.e. stored XSS if a bio survives with a crafted href. This was flagged in the prior round1 review (MEDIUM) and was not touched by any of the 12 fix commits; still present as-is. Recommend an allowlist sanitizer (e.g. `bleach`/`nh3`) instead of regex stripping.

- **scraper/render.py:1191, 2321-2341 (per-day scroll restoration)** — `_savedScrollTop` is keyed only by `period` ("day"/"night"), not by `date + period` — the `.timetable-panel` element only carries `data-period` (render.py:1191), not the date, into the scroll-restore lookup. Switching from e.g. Day 1 Night to Day 2 Night restores Day 1's scroll offset instead of Day 2's own (or top-of-page for a first visit), because both panels share the key `"night"`. This contradicts the documented behavior ("Scroll position: saved per view — switching between lineup and timetable restores where you were") for any multi-day festival with more than one date sharing a period name (which is the common case).

- **scraper/scrape.py:641-729 (`fetch_ra_profile` / `fetch_all_ra`)** — Unlike `fetch_sc_profile` (scrape.py:222-264), `fetch_ig_profile` (scrape.py:400-459), and `fetch_spotify_listeners` (scrape.py:504-522), which each wrap their network/Playwright calls in `try/except Exception` and degrade gracefully per-artist, `fetch_ra_profile`'s `_ra_gql` calls (`page.evaluate` → `fetch`) have no exception handling, and `fetch_all_ra`'s loop body (scrape.py:709) is not wrapped in `try/except` either (only `page.close()` is guaranteed via the outer `finally`). A single RA network timeout, `page.evaluate` failure, or unexpected GraphQL response shape raises uncaught and aborts the entire RA-fetch phase, losing progress on every remaining artist in that batch. (Per-artist DB writes elsewhere already commit immediately — `db.py` commits after every upsert — so this doesn't corrupt state, it only kills the rest of the run.) Flagged in round1 as MEDIUM; not addressed by any of the 12 fix commits; still present.

## LOW

- **scraper/images.py:26-38, 93-122 (`resize_and_crop` / `process_artist_photos`)** — The 20MB byte-size cap on downloaded photos (images.py:117-119) bounds compressed size but not decoded pixel dimensions; `pyvips.Image.new_from_file(raw_path)` has no width×height cap before decode. A crafted/corrupted image well under 20MB compressed can still decode to a very large bitmap (decompression-bomb pattern), causing a memory spike before the per-photo `except Exception` (images.py:126) can catch anything useful. Low severity since photo URLs are scrape/override-sourced rather than directly attacker-submitted.

## Clean areas

- **scraper/timetable_json.py** — No issues found. Timezone is read from the `events` table with a sane fallback (`Europe/Berlin`); slot UUIDs (`uuid.uuid5(uuid.NAMESPACE_URL, card_key)`) are deterministic across rebuilds as long as artist IDs/date/period/stage_id are stable, which they are (push-notification dedup depends on this and it holds).
- **server/static/shared.css** — No issues found. Purely presentational tokens/utilities, no user-data interpolation.
- **server/static/shared.js** — No issues found. `escapeHtml`/`esc` is a correct, complete HTML-entity escaper (`&<>"'`); `storageGet/Set/Remove` are defensively wrapped in try/catch for quota/private-browsing failures; no eval/Function/unsanitized DOM writes.
- **scraper/render.py — WS reconnect / PIN cross-device sync / view-switching** — No correctness issues found beyond the scroll-restoration bug above. WS reconnect uses capped exponential backoff with jitter (render.py:1856, 1891); `reconcile()` and `connectWS()` are properly sequenced (`await reconcile(); connectWS(sessionId)`, render.py:2577) so there's no read/write race between the two; PIN exchange and sync-modal countdown logic are self-contained with no injection points.
- **fetch_videos.py** — No security issues found. Video IDs are YouTube-controlled strings used directly as filenames (`THUMBS_DIR / f"{vid_id}.avif"`, line 173) with no path-traversal guard, but this is not attacker-influenced in practice since IDs come from YouTube's own search API, not user input.

**Summary:** The two previously-CRITICAL XSS findings (`</script>` breakout and unescaped popup/bio `innerHTML`) and the scraper-crashing FK bug are all confirmed fixed and hold up under direct code inspection. The remaining gaps are all MEDIUM-or-lower: a missing URL-scheme allowlist at the render layer that's currently only enforced during scraping (exploitable via a hand-edited `overrides.toml` entry, not from live scraped content), a regex-based (rather than allowlist) bio HTML sanitizer with a concrete entity-encoding bypass, a scroll-restoration state bug on multi-day timetables, and an unguarded RA-fetch phase that can abort mid-run on a single network hiccup. None of these block deploy, but the scheme-allowlist and bio-sanitizer gaps are worth fixing soon since they're genuine (if narrow) stored-XSS surfaces in a public-facing page.
