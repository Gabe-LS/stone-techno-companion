Based on a hunk-by-hunk review of both diffs and the surrounding code they touch, here are my findings.

## HIGH

**[HIGH] server/chat_ws.py:1816-1873 vs :966-1006 — deleted "pending" message can be resurrected by the async moderation task**
`delete_message` has no check on `moderation_status` — a user can delete their own message (including one still `pending`, since moderation of a few-second AI call is often still in flight) within the 120s window. If `_moderate_and_broadcast` is still running when the delete happens, its "moderation passed" path (chat_ws.py:967-1006) doesn't re-check whether the row still exists before broadcasting: it runs `UPDATE messages SET moderation_status='approved' WHERE id=?` (silently a no-op if already deleted), then unconditionally constructs `event_data` from the closure's captured `msg`/`content` and calls `mgr.broadcast_to_room(room_id, event_data, ...)`. The result: a message the sender explicitly deleted reappears live for every other room member moments later (and if it was an image, the file may already be unlinked, so it appears broken). This is a direct interaction bug between the new pending-status flow (1a6aaca) and the pre-existing delete flow — worth a existence/status check (`SELECT ... WHERE id=? AND moderation_status='pending'`) before broadcasting.

**[HIGH] server/chat_ws.py:967-974 + :1278 (get_room_messages filter) — a message stuck in "pending" is invisible forever with no client-side signal**
If the moderation task never completes (process killed by a deploy/restart mid-flight — this is exactly the TRIAGE H15 scenario, still unfixed, just changed shape), the message row is left with `moderation_status='pending'` permanently: no sweep/reconciliation job exists to resolve orphaned pending rows. Because `get_room_messages` now filters `moderation_status != 'pending'` (chat_db.py:896-897), that message vanishes from `room_history` for **everyone, including the sender**, on the next reconnect/reload. But the sender already received `message_acked` (chat_ws.py:1527-1539) which clears the client-side `pending` styling (chat.html:1341 `msgs[idx].pending = false`) — so from the sender's point of view the message looked fully sent, then silently disappeared on refresh with no toast, no `message_removed` event, nothing. Before this fix the message would have stayed visible (unmoderated) rather than vanishing without explanation. There's no client-side timeout tied to moderation completion (only the 10s ack timeout at chat.html:2602-2611, which fires long before the moderation window in a restart scenario would even be detectable).

**[HIGH] server/chat_api.py:214-224 — IP-keyed auth rate limit can lock out an entire venue**
`_check_auth_rate` (20 requests / 5 min per `request.client.host`) is now applied to `/auth/google`, `/auth/google/code`, and `/verify` (magic-link click). This app's documented context is a live festival: attendees on venue WiFi or mobile-carrier CGNAT commonly share a small number of public IPs (confirmed correctly forwarded by Caddy per `--forwarded-allow-ips '*'` in server/Dockerfile:10, so this isn't a proxy-header bug — it's a real shared-IP scenario). A normal sign-in rush (more than 20 people completing Google/email sign-in within 5 minutes from the same NAT'd IP) will 429 every other user behind that IP, including people who made zero requests themselves. Given the stated audience (festival attendees), this is a plausible day-one lockout, not a theoretical edge case.

## MEDIUM

**[MEDIUM] server/api.py:70-75 + server/chat_api.py:1336-1338,1433+ — H13 (unbounded body buffering) only partially closed**
The new `_limit_body_size` middleware only rejects when a `Content-Length` header states a value over 110MB. The actual upload endpoints (`upload_image` chat_api.py:1336, `upload_video` similarly, avatar upload similarly) still do `data = await file.read()` first and check the real size limit (5MB, etc.) afterward. A request with no `Content-Length` (chunked transfer) or one that lies at just under 110MB to a 5MB endpoint still gets fully buffered in memory before the per-endpoint check fires. This narrows the exposure from "unbounded" to "up to 110MB per request," but doesn't fix the underlying pattern the TRIAGE item called out.

**[MEDIUM] server/chat_ws.py:1235-1263 (join_room) vs :976-988 (_moderate_and_broadcast) — profile-edit propagation (b6048b1) is incomplete**
`broadcast_profile_update` only touches rooms in `self.user_rooms[user_id]` — i.e., rooms the editor currently has *joined* via an explicit `join_room` WS event (populated only in `join_room`, not in `connect`, see chat_ws.py:622-626). So: (1) rooms/DMs the editor belongs to but isn't currently viewing keep stale cached identity in `room.user_info` until the editor later joins them; and (2) when the editor *does* later join a new room in the same WS session, `join_room` passes the same per-connection closure variables (`display_name`, `username`, `color_index`, `avatar_url` — captured once at connect time, chat_ws.py:1153-1157) rather than a fresh DB lookup, so the new room gets seeded with the pre-edit identity. Message-sending was explicitly fixed to re-fetch (chat_ws.py:979-988), but `join_room` was not given the same treatment — an inconsistent fix of the same underlying problem (H22).

**[MEDIUM] server/chat_db.py:552-558,1491-1494 (LEFT JOIN) + server/chat/admin.html:293-294 — deleted reporter/reported user renders as literal "null"**
`ceeec3b` correctly changes `get_pending_reports`/`get_user_admin_detail` from INNER to LEFT JOIN so report rows survive user deletion, but `admin.html` wasn't updated to handle the resulting NULL `reporter_name`/`reported_name`. `esc()` is `String(s).replace(...)`, so `esc(null)` renders the string `"null"` directly in the Reports/Users admin table — confusing during live moderation ("null" appearing where a username should be) rather than a placeholder like "[deleted user]".

**[MEDIUM] server/chat_ws.py:170-229 (`_resolve_safe_ip`/`_pinned_preview_get`) — link preview pinned to a single resolved address with no fallback**
The new SSRF-safe fetch resolves the hostname once and connects to `safe[0]` (the first address `getaddrinfo` returns) with no retry across other addresses. Previously, connecting via hostname let the underlying connector try multiple resolved addresses if one failed. For a dual-stack host whose first returned address is an unreachable AAAA record (e.g., IPv6 without a route from inside the Docker network), link previews that worked before this fix will now fail outright. Not a security issue, but a real reliability regression for some target sites.

## LOW

**[LOW] server/chat/chat.html:3985-4029,4581-4582 — first-run notification prompt could stack with a manually-opened Notifications modal**
`_maybePromptNotifications` is scheduled 1.5s after `route()` completes and creates its own `#notif-prompt-modal` overlay without checking whether the user has already opened the Notifications settings modal manually in that window. Low-probability, purely cosmetic (two overlapping `.modal-overlay` divs), no functional break.

**[LOW] deploy.sh:98-111 — WAL checkpoint failure is easy to miss and silently degrades the fix's own guarantee**
If `wal_checkpoint(TRUNCATE)` fails on the VPS (e.g., lock contention with the live app), the script prints `WARN: checkpoint failed ...` but proceeds to back up anyway — potentially with the exact WAL/db split inconsistency this fix was meant to close, buried in verbose ssh output rather than surfaced distinctly.

## Areas reviewed with no issues found
- **deploy.sh atomic `.env` write** (lines 82-90): quoting is correct (`\$` vs `$` used deliberately to control local vs. remote expansion), the size-check-then-`mv` pattern correctly prevents a truncated file from ever being installed, and `set -euo pipefail` correctly aborts the script if the remote check fails.
- **stone_techno_companion.py rsync scoping change** (lines 79-103): scoping `--delete` to only `photos/`/`thumbs/` instead of the whole `server/static/` is a genuine improvement over the prior global-mirror bug; both directories are incrementally accumulated locally in the same way, so this doesn't introduce a new deletion risk beyond what already existed.
- **Push endpoint allowlist** (`_is_valid_push_endpoint`, api.py/chat_api.py): suffix matching correctly covers real FCM/Mozilla/Apple/WNS endpoint hostnames; doesn't block any currently-supported browser.
- **`_push_or_defer`/`_push_sent` fix**: the debounce dict mutation happens synchronously before the first `await`, so there's no race between concurrent calls for the same user/room key; the "only mark sent after a real send" logic is correctly threaded through all of `_do_send_push`'s return paths.
- **Cross-message media_url deletion guard** (chat_ws.py:1842-1863): I could not construct a legitimate-delete-fails-to-unlink scenario — each upload gets a unique 32-hex token, so the `others == 0` check doesn't false-positive for normal single-reference images/videos.

**Summary**: No new critical/deploy-blocking regressions found. The most significant issues are two real interaction bugs in the new moderation-pending flow (a deleted-but-still-moderating message can reappear for the whole room, and a message whose moderation task never completes vanishes for its own sender with no explanation) and an IP-keyed auth rate limiter that's a plausible lockout vector given this app's actual festival/shared-network deployment context. The remaining items are incomplete-but-directionally-correct fixes (body-size capping, profile-edit propagation, deleted-user display in admin) that narrow but don't fully close their original TRIAGE findings, plus a couple of low-priority cosmetic/reliability edge cases.
