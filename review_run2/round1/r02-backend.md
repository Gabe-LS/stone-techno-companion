## Backend REST + Auth Sweep — `server/api.py` + `server/chat_api.py`

Reviewed both files end-to-end against the current (post-fix) state, cross-checked against `review_run/TRIAGE.md` and the two diffs in `review_run2/inputs/` to confirm what was actually fixed vs. only partially addressed.

### CRITICAL
No issues found. All six CRITICAL items from TRIAGE that touched these two files (C5 push-endpoint SSRF, C6 blocking Google OAuth calls) are fixed and verified in the current code (`chat_api.py:1597-1615`, `api.py:846-864` for the SSRF allowlist; `asyncio.to_thread` wrapping in `auth_google`/`auth_google_code`, `chat_api.py:339-344, 388-399, 408-413`).

### HIGH

- **[HIGH] `server/chat_api.py:554-565`** — `POST /chat/api/logout` deletes the session row and clears the cookie but never closes the user's live WebSocket connection(s). Compare to `auth_delete_account` (577-581) and `admin_ban` (1904-1910), which both explicitly iterate `manager.user_conns[user_id]` and force-close every socket. `chat_ws.handle_chat_ws` (`chat_ws.py:1138-1140`) validates the session token only once, at connect — so an already-open tab keeps full send/receive access indefinitely after the user clicks "Log out," e.g. on a shared/public device. This is the same bug class as the already-fixed H23, just on the one endpoint the fix missed.

- **[HIGH] `server/chat_api.py:1671-1678`** — `POST /chat/api/swlog` has no authentication and no rate limit at all (contrast every other endpoint in the file). Any anonymous caller can POST arbitrary JSON repeatedly; each call writes `logger.info("[SWLOG] %s", json.dumps(body)[:500])` straight to the production log. This is an unauthenticated log-injection/log-flood primitive — logs are capped at 10MB×5 via `docker-compose.yml` (per CLAUDE.md), so sustained abuse can rotate away genuine security-relevant lines (moderation FLAGGED entries, VAPID errors, push failures) as cover for a concurrent attack, in addition to letting an attacker write attacker-chosen content into the server's logs.

- **[HIGH] `server/api.py:59-63` vs `server/api.py:797`** — The `_RedactTokensFilter` added to fix "session token in URL gets logged" only redacts `/ws/chat/` (chat WS) and `/chat/v/` (email verify) paths, plus `?token=/?code=/?id_token=` query params. It does **not** match the lineup sync WebSocket route `@app.websocket("/ws/{code}")` (api.py:797), whose `code` path segment is frequently the raw `session_id` — the exact same secret value set in the `stc_session` cookie (`httponly=True`, `secure=True`, 90-day `max_age`). That secret rides unredacted in every `uvicorn.access` log line for that connection, defeating the httpOnly protection and handing out full read/write hijack of a user's picks/schedule for up to 90 days to anyone with log access. This is the same defect class the fix commit (`361a6e1`) was written to close, just missed on this second WS endpoint.

### MEDIUM

- **[MEDIUM] `server/chat_api.py:476-478`** — `POST /chat/api/login` checks `is_banned(db, "email", provider_id)` and, if true, returns `HTTPException(403, f"You have been banned: {ban['reason']}")` **before** any proof the caller owns the address (no click-through required). Anyone can probe an arbitrary email and learn (a) that it's banned and (b) the admin-authored ban reason text — an enumeration/harassment vector distinct from (and not mitigated by) the existing `_email_rate` limiter, which only slows, not blocks, single-IP probing.

- **[MEDIUM] `server/chat_api.py:1239, 1336, 1433`** — The global 110MB Content-Length ceiling added in `api.py:518-530` only bounds the *absolute worst case* (sized for 100MB video); it does not stop `upload_avatar`/`upload_image`/`upload_video` from fully buffering an oversized body into memory (or to a temp file, for video) before the endpoint's real declared limit is enforced. E.g. a 100MB POST to `/chat/api/upload/avatar` (limit: 500KB) passes the middleware and is still fully read via `await file.read()` (line 1239) before the 500KB check rejects it. Combined with the 10-uploads/60s per-user cap, a handful of concurrent requests per user can still force tens of MB of memory pressure well beyond what each endpoint is supposed to allow. TRIAGE's H13 ("unbounded request body buffered before size check") is only partially closed for these three routes.

- **[MEDIUM] `server/chat_api.py:959-974`** — `GET /chat/api/rooms/{room_id}/online` checks DM-participant membership only for `room["type"] == "dm"`; it has no equivalent check for `type == "meetup"`. The sibling endpoint `room_messages` (886-904) gates *both* `dm` and `meetup` types via `dm_participants`/`meetup_attendees`. Any authenticated user (attendee or not) can therefore query the online/member list of a meetup's private chat room, even though the messages in that same room are properly access-controlled.

### LOW

- **[LOW] `server/chat_api.py:856-868`** — `POST /chat/api/rooms/{room_id}/join` never checks `room["type"]`, so it can be called with a `dm` or `meetup` room's id to insert a `room_memberships` row for a room the caller was never invited to. Doesn't leak message content (that stays gated by `dm_participants`/`meetup_attendees` in `room_messages`), but pollutes membership/badge tracking and is inconsistent with the type-aware checks used elsewhere in the same file.

- **[LOW] `server/chat_api.py:676-679`** — `GET /chat/api/check-name` has no auth and no rate limit, unlike `/check-username` (666-673) which requires a session cookie. Lets anonymous callers hammer the display-name word filter (`chat_moderation.get_word_filter().check`) without limit — a low-value oracle/probing surface, but a clear asymmetry with its sibling endpoint.

- **[LOW] `server/chat_api.py:89-98`** — `_upload_rate` (the per-user upload-rate-limit dict) never prunes stale `user_id` keys, unlike `_auth_rate` (194-197) and `_email_rate` (449-452), which both self-prune once they exceed 1000 tracked IPs. Bounded in practice by the number of real registered accounts, but it's a real inconsistency with the pattern applied to the other two in-memory rate-limit dicts in this file.

- **[LOW] `server/chat_api.py:1681-1704`** — `POST /chat/api/push/ack` has neither authentication nor rate limiting; it trusts whatever `endpoint` string is posted and looks up the owning user via `find_user_by_push_endpoint`. Practical exploitability is low (requires already knowing a victim's push subscription endpoint, which isn't exposed to other users through the app), but it's worth at least a light rate limit given it can write `last_seen`/`last_active` for an arbitrary account.

- **[LOW] `server/api.py:1037-1094`** — `/manifest.json`, `/favicon.svg`, `/favicon.png`, `/bios.json`, `/shared.css`, `/shared.js` set no explicit `Cache-Control` header (unlike `/sw.js`, which sets `no-cache`, and the catch-all `index.html`, which sets `no-store`). Since the working-tree diff to `stone_techno_companion.py` now copies `shared.css`/`shared.js` to the VPS on every content deploy, a browser's heuristic cache could keep serving a stale CSS/JS bundle past a deploy meant to fix it, with no cache-busting mechanism to force a refresh.

### Clean areas worth calling out explicitly
- **Magic-link email content injection** (the specific thing flagged in the assignment): `_magic_link_email`'s `site` and `host` parameters are *not* attacker-controlled — `site` comes from `lineup.db` (admin-authored) or a hardcoded fallback, and `host` is derived from the `CHAT_BASE_URL` env var (or the hardcoded prod domain), never from a request header. No header/HTML injection path exists here.
- **SQL injection**: every query across both files uses parameterized `?` placeholders; no string-formatted SQL was found.
- **Path traversal**: the `/chat/uploads/{filename}` route and both static `StaticFiles` mounts are correctly locked down (hex-only filename regex; Starlette's own traversal guards).
- **Push-endpoint SSRF fix**: the `.endswith(".googleapis.com")`-style suffix allowlist in both `api.py:854-864` and `chat_api.py:1605-1615` is correctly anchored with a leading dot and can't be bypassed via naive suffix-confusion domains.

One-paragraph summary: the fix commits closed all previously-flagged CRITICAL items in these two files, and most HIGH items (SSRF, blocking OAuth calls, VAPID TTL, rate limits on verify/meetup) are solid. The remaining gaps cluster around two things the earlier round didn't fully generalize: WebSocket-invalidation-on-auth-change (fixed for delete/ban but missed for plain logout) and URL-embedded-secret redaction (fixed for the chat WS/verify routes but missed the lineup sync WS route) — both HIGH and both straightforward one-line-pattern fixes once flagged. The newly-discovered `/chat/api/swlog` endpoint is a genuine unauthenticated log-injection/flood hole that should be closed or gated before deploy. The MEDIUM/LOW items (ban-reason enumeration via `/login`, incomplete upload-size enforcement, meetup-room online-list exposure, minor rate-limit/cache-header inconsistencies) are real but lower-urgency polish.
