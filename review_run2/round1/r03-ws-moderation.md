# WebSocket + Moderation Review

## CRITICAL

**[CRITICAL] `server/chat_ws.py:896-900` + `server/chat_moderation.py:420-438` — banned/muted users can send unlimited DM messages over any already-open connection.**
`_moderate_and_broadcast` only calls `moderate_message()` (which contains the `is_banned`/`is_muted` checks at `chat_moderation.py:420-438`) `if is_moderated:`. Every DM room is created with `is_moderated=False` (`chat_db.py:1148`, reinforced by the migration `UPDATE rooms SET is_moderated = 0 WHERE type = 'dm'` at `chat_db.py:352`), so for DM sends the code takes the `else: mod_result = {"allowed": True}` branch (`chat_ws.py:900`) and `moderate_message` — and therefore the ban/mute check — never runs. This is not a "DMs can't be content-scanned" limitation (that part is intentional and documented); it's that ban/mute *enforcement* is nested inside the same function as content scanning, so skipping content scanning also skips enforcement. Compounding this, `handle_chat_ws` (`chat_ws.py:1138-1146`) never checks `is_banned` at connect time either — it only validates the session token. Since `ban_user`/`ban_user_all_providers` deletes sessions (blocking *new* connections) but does not close already-open WebSocket connections for the automatic-ban path (see next finding), any banned or muted user with a live connection at the time of the ban can continue sending DMs indefinitely to any non-blocking user. This defeats the ban system for the app's most private, least-moderated surface.

## HIGH

**[HIGH] `server/chat_ws.py:967-1006` — deleting/expiring a message during in-flight moderation doesn't stop it from being broadcast, badged, and pushed.**
`create_message` inserts the row immediately and moderation runs as a detached `asyncio.create_task` (`chat_ws.py:1566-1585`). If the sender calls `delete_message` (`chat_ws.py:1816-1873`) while that task is still awaiting the OpenAI calls, the DB row is deleted — but when the task later resolves as "allowed", it executes `UPDATE messages SET moderation_status='approved' WHERE id=?` (a silent no-op on 0 rows, never checked) and proceeds unconditionally to `broadcast_to_room` the "message" event, run the link-preview fetch, fan out `badge_update`, and schedule push notifications (`chat_ws.py:1006-1097`) — all for content the sender explicitly retracted. There is no re-check that the message still exists before any of this. The same gap applies if the message's TTL purges it mid-moderation. Net effect: other room members (and push recipients) can receive a message the user believed was deleted, with no way to know it happened.

**[HIGH] `server/chat_ws.py:939-946` vs `server/chat_api.py:1904-1910` — automatic bans don't close all of a user's live connections, and non-message WS events are never checked against ban/mute at all.**
The admin-ban path (`chat_api.py:1906-1910`) iterates every `conn_id` in `manager.user_conns[user_id]` and closes all of them. The automatic moderation-ban path in `_moderate_and_broadcast` only closes the single `ws` instance tied to the flagged message (`chat_ws.py:943-946`); any other open connection for the same user (a second tab/device) stays connected and fully functional. Worse, none of `typing`, `add_reaction`, `remove_reaction`, `create_meetup`, `join_meetup`, `leave_meetup`, `block_user`/`unblock_user`, `report_message`, `open_dm`, `mark_read`, or `join_room` (`chat_ws.py:1587-1937`) check `is_banned`/`is_muted` at all — only `send_message` does (indirectly, via `moderate_message`, and only for moderated rooms per the CRITICAL finding above). A banned user's still-open connection retains full ability to react, join meetups, file reports against others, and browse rooms.

## MEDIUM

**[MEDIUM] `server/chat_ws.py:576-591` — `ChatRoom.broadcast()` cleans up room-scoped state on a failed send but leaves `ConnectionManager`-scoped state (zombie connection).**
When `ws.send_text()` raises inside `broadcast()`, the handler pops the connection from `room.connections`/`room.conn_users` (and `user_names`/`user_info` if last connection in that room), but never touches `ConnectionManager.user_conns`, `conn_user`, or `user_rooms`. Those only get cleaned up in `disconnect()`, which runs from the WS receive loop's `finally` block — i.e., only when that connection's own `receive_text()` eventually raises. A half-dead socket (TCP buffer still accepting writes but never producing reads, e.g. a phone that lost network without a clean close) can sit as a "connected" zombie in `user_conns`/`conn_user`/`user_rooms` indefinitely, inconsistent with the room-level presence state that already marked it offline. `send_to_user` (`chat_ws.py:745-752`) will keep attempting (harmlessly swallowed) sends to it, and for roughly the first 30s after this happens the user is wrongly excluded from push targeting (`chat_ws.py:1069-1076`) since `connected_uids` still includes them.

**[MEDIUM] `server/chat_ws.py:617-620`, `1177`, `1187-1190`, `1993-2025` — `manager.rooms` and `manager._room_meta` grow unboundedly for DM rooms with any message history.**
The purge loop only evicts `_room_meta`/`rooms` entries for expired meetups and *empty* DM shells (`last_message_at IS NULL`). A DM room that has ever exchanged a message is never removed from these process-memory dicts for the life of the server, by design (DM threads persist). Over a multi-day event with many concurrent 1:1 conversations, this is unbounded, unrecoverable growth until restart — worth capping or evicting `_room_meta`/`ChatRoom` entries for DMs with no live connections after some idle window, independent of the DB-level purge rule.

**[MEDIUM] `server/chat_ws.py:1086-1097`, `1566-1585`, `389-391` — fire-and-forget `asyncio.create_task` calls with no retained reference.**
Per asyncio's own documented pitfall, the event loop only holds a weak reference to a task; without storing the returned `Task` somewhere (e.g., a module-level `set()` with a done-callback), the task is eligible for GC before completion. This applies to the moderation task, the per-target push tasks, and the debounce flush task. A dropped moderation task leaves a message stuck at `moderation_status='pending'` forever (until TTL purge silently removes it) — the sender never gets a `message_removed` or a broadcast, so the message just vanishes with no feedback.

**[MEDIUM] `server/chat_ws.py:1549-1564` (using `_image_to_data_uri`/`_video_mod_frames`, `chat_ws.py:82-108`) — synchronous file reads in the WS receive loop block the whole event loop.**
Unlike the DNS resolution and `webpush()` calls elsewhere in this file, which are wrapped in `asyncio.to_thread`, the moderation-copy image/video frame reads (`mod_path.read_bytes()`) run synchronously inside the main `send_message` handling path, before the moderation task is even spawned. Since this is a single asyncio event loop shared by every connected user, every image/video send briefly stalls all other connections' I/O.

**[MEDIUM] `server/chat_ws.py:457-471` — deferred/debounced push previews show raw JSON instead of message text.**
`_flush_push_later` always calls `_push_or_defer` with empty `sender_name`/`text_preview` (`chat_ws.py:424`), forcing the fallback lookup in `_do_send_push`. For a non-DM room with `count == 1`, that fallback sets `text_preview = msg_row["content"] or ""` (`chat_ws.py:471`) — the raw stored JSON envelope (e.g. `{"text": "hey everyone"}` for text, or the image/video JSON blob), not the parsed text used everywhere else (`chat_ws.py:1030-1041` parses type-appropriately with "Sent a photo" etc.). Any debounced group-room push — which is the common case, since debounce exists precisely to delay pushes during bursts — will show malformed JSON in the notification body.

**[MEDIUM] `server/chat_moderation.py:123-147` — word filter fully bypassed by spacing out letters.**
`check()` normalizes and tokenizes on whitespace, then requires an exact token-sequence match (`words[i:i+n] == term_words`, `chat_moderation.py:132-136`). A message like `"m d m a"` or `"k e t"` normalizes to multiple single-character tokens that can never equal a multi-character term, so the entire word-filter layer is defeated by this trivial, well-known bypass technique. AI layers 2/3 partially mitigate this, but layer 1 (the instant, free check) is fully bypassed, and both AI layers can be skipped/errored (fail-closed only when `OPENAI_API_KEY` is set).

**[MEDIUM] `server/chat_ws.py:1746-1814` — `add_reaction`/`remove_reaction` don't check `is_blocked` for DM rooms.**
`send_message` explicitly checks `is_blocked` for DM sends (`chat_ws.py:1443-1454`), but reactions on DM messages only verify the actor is a `dm_participant` (`chat_ws.py:1758-1763`, `1793-1798`) — not whether either party has blocked the other. A user who has blocked their DM counterpart (or been blocked) can still see/add reactions from them in that DM thread, and `typing` events have no block check at all in any room type. This is a narrower version of the documented "blocking is client-side for group rooms" design, but for DMs the server otherwise actively enforces the block, making the reaction/typing gap look like an oversight rather than intentional scope.

## LOW

**[LOW] `server/chat_moderation.py:128-137` — TRIAGE item H9 (no substring match within a token) still unaddressed.** None of the 12 fix commits touch this. A term embedded in a longer unbroken token (e.g. `"iloveketamine"`) still isn't caught by the word filter's `check()` (only `check_username` does substring matching, at `chat_moderation.py:149-163`). Confirming this is still open since it wasn't part of the recent fix batch.

**[LOW] `server/chat_db.py:1239-1247` — pending reports are never purged.** `purge_old_reports` only deletes reports with `status IN ('actioned', 'dismissed')`; a report stuck at `status='pending'` (e.g. never reviewed by an admin) lives forever. The `report_message` rate limit added in the recent fixes bounds the *rate* of new reports but not the total retained count over a long-running deployment.

**[LOW] `server/chat_ws.py:1636-1654` — `meetup_invite` broadcasts use stale identity captured at connection time.** Unlike the regular message path, which re-fetches `display_name`/`color_index`/`avatar_url` from the DB right before broadcasting (`chat_ws.py:979-989`) specifically to reflect profile edits made mid-connection, the meetup-invite broadcast still uses the handshake-time locals. A user who edits their profile and then creates a meetup will show the old identity on the invite card, inconsistent with the just-fixed general-message behavior.

## Summary

The most serious finding is that DM rooms' `is_moderated=False` flag short-circuits the *entire* `moderate_message()` call, which is also where ban/mute enforcement lives — so a banned or muted user with any still-open WebSocket connection can keep sending DMs indefinitely, and automatic (non-admin) bans only close the one connection tied to the flagging message rather than all of a user's connections, while no non-message event type checks ban/mute status at all. Separately, `_moderate_and_broadcast` has no re-check that a message still exists before marking it approved and broadcasting/pushing it, so deleting or letting a message expire during its (often multi-second) moderation window doesn't actually retract it for other room members. The remaining findings are more contained: a partial-cleanup gap that can leave zombie connections in `ConnectionManager` state, unbounded in-memory growth of DM room metadata over a multi-day event, un-referenced fire-and-forget tasks that could silently strand a message in "pending," a blocking file read on the hot WS path, a raw-JSON push preview bug for debounced pushes, and a cheap word-filter bypass via letter-spacing. Connection lifecycle, strike escalation math (1/2/3/4-strike ladder, mute-cycling counter, strike TTL/reset), and the purge loop's core invariants (DM-shell preservation, meetup expiry, cascade-deleted reactions/reply nulling) all checked out clean.
