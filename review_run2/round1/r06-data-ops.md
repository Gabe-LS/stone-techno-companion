# Data Layer + Deploy/Ops Review

## HIGH

**[HIGH] .gitignore:1-33 — `server/chat-uploads/` (the real production upload directory) is not covered**
`docker-compose.yml:14` bind-mounts `./chat-uploads:/app/chat/uploads` — on the VPS, real user-uploaded photos/videos live at `server/chat-uploads/`, inside the same git working tree that `deploy.sh` runs `git pull` against. The current `.gitignore` only ignores `server/chat/uploads/` and `server/chat/tmp/` (a different, sibling path — the local-dev-only location when running bare `uvicorn`). CLAUDE.md explicitly claims "`.gitignore` covers the live chat.db and chat/uploads/ so they can't be committed" — that claim is false for the actual production bind-mount path. This repo has already been bitten by exactly this class of gap: the current diff shows `server/static/thumbs` being deleted from git tracking and added to `.gitignore` in the same commit (`.gitignore` diff line 9, `git status` shows `D server/static/thumbs`), meaning that path was previously untracked-but-ignored and got accidentally committed. `server/chat-uploads/` is the same shape of gap, live right now, and holds real (including DM-adjacent, privacy-sensitive) user media — a stray `git add -A`, `git stash`, or a well-meaning `git clean -fd` run to fix an unrelated VPS git snag would either commit or permanently delete live user uploads. Add `server/chat-uploads/` to `.gitignore`.

## MEDIUM

**[MEDIUM] server/chat_db.py:167-171 — `dm_participants` has no index usable by its hottest lookup**
`dm_participants`'s only index is the composite PK `(room_id, user_id)`. Both `find_or_create_dm` (chat_db.py:1129-1133, run on every DM open) and `get_unread_counts` (chat_db.py:765-785, run on every push-eligibility check and on connect for badge counts) filter with `WHERE dp.user_id = ?` / `dp1.user_id = ?` — a leading-column mismatch against the PK, forcing a full table scan of `dm_participants` on every call. Add `CREATE INDEX IF NOT EXISTS idx_dm_participants_user ON dm_participants(user_id)`.

**[MEDIUM] deploy.sh:49-54, 62-69 — required-var check aborts silently instead of showing its own error message**
`val=$(grep "^${var}=" "$LOCAL_ENV" | cut -d= -f2-)` has no `|| true` fallback (unlike the `OPTIONAL_VARS` loop at deploy.sh:71-76, which correctly has one). If a `PROD_VARS` entry is entirely absent from `server/.env` (not just present-but-empty), `grep` exits 1; under `pipefail` that becomes the assignment's exit status even though the trailing `cut` succeeds, and `set -e` then aborts the script immediately — before the intended `"  ERROR: Missing values in $LOCAL_ENV: ..."` message at line 56 ever prints. Not reachable today (every `PROD_VARS` entry is also required for local dev, so it's always present), but it defeats the one piece of diagnostics this block exists to provide, and CLAUDE.md's own Deploy Checklist documents a var rename (`VAPID_SUBJECT` → `VAPID_CLAIMS_EMAIL`) as exactly the kind of change that would trigger it next. Add `|| true` to both loops, matching the optional-vars pattern.

**[MEDIUM] deploy.sh:92-132 — VPS backup never covers `server/chat-uploads/`, and the exposure window just grew**
The download/verify/backup steps only touch `server/data/` (hearts.db, chat.db, vapid key). The bind-mounted `server/chat-uploads/` media directory (docker-compose.yml:14) is never backed up locally or on the VPS. This gap existed before, but it was sized against a 60-minute message TTL; `chat_db.py:233-234` now defaults DM and room TTLs to 1440 minutes (24h), so a VPS disk failure can now lose a full day of live, undeleted user photos/videos instead of ~1 hour.

**[MEDIUM] server/chat_api.py upload_image/upload_video (chat_db.py purge has no coverage) — uploaded-but-never-sent media is never garbage collected**
`upload_image`/`upload_video` write the served file (+ moderation copies) to `chat/uploads/` unconditionally and return the URL, with no DB row linking the file to anything. `purge_expired_messages` (chat_db.py:964-1001) is the only cleanup path for message media, and it only ever looks at rows in `messages` — a file that was uploaded but whose message was never sent (client abandoned it, or crashed before `send_message`) has no message row and is therefore invisible to every purge path in `chat_db.py` forever. Bounded only by the 10/min upload rate limit, this is straightforward unbounded disk growth plus indefinite public hosting of unmoderated content. Needs a `pending_uploads`-style tracking table (or equivalent) with its own sweep, since nothing in the current schema can express "uploaded, not yet attached to a message."

## LOW

**[LOW] deploy.sh:183 — backup-pruning step can abort the script after a successful deploy**
`ls -dt server/data.bak.* 2>/dev/null | tail -n +6 | xargs rm -rf`: GNU `xargs` without `-r`/`--no-run-if-empty` still invokes the command once even on empty input, so when fewer than 6 backups exist yet (true for the first ~5 real deploys against this VPS), this runs a bare `rm -rf` with no operand, which exits 1. Combined with `set -euo pipefail`, this aborts the script at the very last step — after the health check and chat-API check have already passed — so the trailing `"=== Deploy complete ==="` banner never prints, reading as a failed deploy when it wasn't.

**[LOW] server/chat_db.py:65-71, 193-201 — `email_tokens.expires_at` and `strikes.expires_at` have no index, scanned every 30s**
`purge_loop` (chat_ws.py:1976-2007) calls `purge_expired_sessions` (deletes from `email_tokens WHERE expires_at <= ?`) and `purge_expired_strikes` on every 30-second cycle unconditionally. Neither table has an index on `expires_at` (`strikes` only has `idx_strikes_user`; `email_tokens` has no secondary index at all), so each cycle does a full table scan. Low cost at current expected scale, but cheap to fix given how frequently it runs: `CREATE INDEX ... ON email_tokens(expires_at)`, `CREATE INDEX ... ON strikes(expires_at)`.

**[LOW] server/Dockerfile:1-10 — container still runs as root**
No `USER` directive; unchanged from the prior audit (review_reports/predeploy-ops.md). Standard hardening gap, not urgent given container isolation, but still open.

## Verified clean (worth noting explicitly given the assignment's specific questions)
- **deploy.sh WAL-checkpoint heredoc** (deploy.sh:101-111): `$VPS_DIR` *does* expand — it's inside a double-quoted string in the *local* shell, and the `<<'PY'` quoting only governs how the *remote* shell would parse a heredoc, which is irrelevant here since the whole thing is just literal text inside an already-quoted ssh argument. This is correct and necessary (the python script needs the literal path); not a bug.
- **Atomic `.env` write** (deploy.sh:82-89): temp file + byte-count check + `mv`, with the remote `$(wc -c ...)` correctly backslash-escaped so it evaluates on the *remote* side against the *remote* file — a dropped connection can't leave a truncated `.env` in place, and previous prod `.env` is preserved on any failure.
- **Health-check-failure path**: now exits 1 before reaching the backup-pruning step, so pruning genuinely only happens after a confirmed-healthy deploy — the "prunes backups regardless of failure" issue from the earlier audit is fixed.
- **`reports` table cascade fix** (chat_db.py:180-191, 319-350): FK-less `reporter_id`/`reported_user_id` confirmed, with a careful table-rebuild migration (commit before turning `foreign_keys` off, explicit `BEGIN/COMMIT`, re-enable + recommit) — moderation evidence now correctly survives user deletion, matching the "preserve report evidence" fix commit.
- **`delete_user` cascade + media cleanup**: every call site (`chat_api.py:572-573, 2142-2143`, `chat_ws.py:929`) correctly calls `delete_user_messages` (which unlinks media files) before `delete_user` (which cascades the DB rows) — no orphaned files on user deletion.
- **Moderation-hold redesign** (`moderation_status` column): `get_room_messages` correctly filters `!= 'pending'`; rejected messages have their served files unlinked (chat_ws.py:904-917) before the row is deleted; a crashed/killed moderation task just leaves a row stuck `'pending'` until the ordinary TTL purge removes it — bounded, never served. H11/H15 both hold up.
- **scraper/db.py, migrate_db.py**: no issues found. Schema/FK/index shape is appropriate for the lineup DB's read-heavy, non-ephemeral workload; `migrate_db.py`'s one-time table-rebuild sequences the `PRAGMA foreign_keys` toggle correctly around commits and ends with an explicit `PRAGMA foreign_key_check`.
- **Docker layer caching, healthcheck, restart policy, log caps, resource limits**: `requirements.txt` copied/installed before source `COPY`s (correct cache ordering), `mem_limit`/`cpus`/`stop_grace_period`/`logging.max-size`/`max-file` all present and match the H16 fix, healthcheck target matches the exposed port and CMD binding.

## Summary
No CRITICAL issues in this slice. The one HIGH is a real, currently-live gap between `.gitignore` and the actual production upload path, made more credible by an in-repo precedent (`server/static/thumbs`) of exactly this kind of gap causing an accidental commit. The MEDIUMs are two real hot-path index gaps (`dm_participants`), a silent-abort bug in `deploy.sh`'s own validation logic, a backup-coverage gap whose exposure window grew with the recent TTL default change, and a still-open "abandoned upload" growth vector that wasn't part of this round's fix batch. The moderation-hold, report-evidence-preservation, and deploy-safety fixes from the prior 12 commits all check out correctly against their original findings — no regressions found in that fixed code.
