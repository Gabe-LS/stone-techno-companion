## Chat Frontend Sweep — `server/chat/chat.html`, `server/static/shared.js`, `server/static/sw.js`

Read all 4,794 lines of `chat.html` in order, plus the full `shared.js` (97 lines) and `sw.js` (113 lines). Cross-checked server-side validation in `chat_ws.py` wherever a sink's safety depended on it (reactions, media URLs, location coordinates).

### CRITICAL
No issues found.

### HIGH
No issues found. Every `innerHTML`/`insertAdjacentHTML` sink that touches user-controlled data (room/meetup/DM names, message text, reply snippets, link previews, member names, country names) runs through `esc()`/`escapeHtml()` or the attribute-safe `jss()` helper (`chat.html:4260`, correctly double-escapes: JS-string-escape then HTML-escape, in the right order for a single-quoted JS literal inside a double-quoted HTML attribute). The one case that looks unescaped at first glance — the `location` message type's `href="https://maps.google.com/?q=${c.lat},${c.lng}"` at `chat.html:2198-2201` — is actually safe: `c.lat`/`c.lng` are read via `JSON.parse`, so a string payload crafted for injection makes `c.lat.toFixed(5)` throw, which falls into the existing `catch` that re-renders the raw content through `escapeHtml`. Reaction emoji (`chat.html:2260-2266`) are constrained server-side to `ALLOWED_REACTIONS` (`chat_ws.py:1134,1751`), and media URLs rendered into `src`/`onclick` (`chat.html:2189,2192`) are constrained to `_UPLOAD_URL_RE` both server-side (`chat_ws.py:1394-1408`) and client-side for decrypted E2EE media (`_validateDecryptedMedia`, `chat.html:1151-1160`).

### MEDIUM

- **[MEDIUM] chat.html:1978-1989** — `openRoom`'s REST fallback path (taken whenever the WebSocket isn't `readyState 1` yet when a room is opened, e.g. a slow/failed handshake) fetches messages via `api('/rooms/'+roomId+'/messages')` and calls `renderMessages()` directly, **without** ever calling `_decryptMessageContent()`. For an E2EE DM this means the raw `{e2ee:true,...}` envelope is passed to `renderMessage()`, which does `JSON.parse(m.content).text` and gets `undefined` — the message renders as a blank bubble. It self-heals once `connectWS()` completes and `join_room`/`room_history` re-renders with decryption, but on a flaky connection (the exact case this fallback exists for) the blank state can persist for seconds, and it's a confusing failure mode for the encrypted-messaging feature specifically.

- **[MEDIUM] chat.html:2299-2351** — `_markReadRoom`/`_lastSeenTime`/`_markReadTimer` are single global variables, not per-room state, and `openRoom` never flushes a pending mark-read before switching. Sequence: user sees a message in room A → `_debouncedMarkRead()` schedules a 500ms timer for room A. User switches to room B within that window → `_setupReadObserver()` (called from room B's `room_history` handler) overwrites `_markReadRoom = B` and resets `_lastSeenTime = null`. If room B's IntersectionObserver then fires (very likely — messages are visible on load), `_debouncedMarkRead()` runs `clearTimeout(_markReadTimer)`, killing room A's pending send before it ever fires. Room A's read receipt for the message the user actually read is silently dropped — the room can be left with a stale/incorrect unread badge until a genuinely new message arrives there. `_flushMarkRead()` exists and correctly guards this same race for `visibilitychange`, but is never invoked on room switch.

- **[MEDIUM] chat.html:1269 + 2602-2611** — `room_history` fully replaces `messagesByRoom[data.room_id]` (no merge). If a WebSocket reconnect happens right after `sendChatMessage()` pushes an optimistic (`pending: true`) message but before the ack arrives, and the reconnect's `join_room` triggers a fresh `room_history` before the message is visible server-side (e.g. still pending moderation), the optimistic entry is wiped from the array. The 10s "not sent" fallback timeout in `sendChatMessage` then does `msgs.find(m => m.id === tempId && m.pending)`, finds nothing, and silently does nothing — the user's own message disappears from their view with no error toast, indistinguishable from success.

### LOW

- **[LOW] chat.html:454-460** — `avatarHtml()`'s fallback-initials branch interpolates `initials` into the DOM without `esc()`/`escapeHtml()`, unlike every other name-rendering call site in the file. Not currently exploitable: `initials` is built from `name.split(' ').map(w => w[0]).join('').slice(0, 2)`, so at most 2 raw characters of attacker-controlled text ever reach the DOM — too little to construct a working tag/attribute injection. Still, it's the one spot that breaks the file's otherwise-consistent escaping discipline, so a future change (longer initials, different truncation) could turn it into real stored XSS.

- **[LOW] chat.html:806/813-830 and 3598/3605-3621** — `openCountryPicker()`/`openEditCountryPicker()` attach a `document`-level `click` listener (`_countryOutsideClick` / `_editCountryOutsideClick`) that is only ever removed by the corresponding `closeCountryPicker()`/`closeEditCountryPicker()`. If the picker's containing modal/prompt is torn down another way (e.g. `submitProfile()` succeeds and replaces `#app` while the country list is still open), the listener is never removed and stays attached to `document` for the rest of the page's life. Currently harmless (the handler's `if (!picker) return` no-ops once the element is gone) but is a genuine unbounded listener leak of the kind the assignment flagged.

**Summary:** No exploitable XSS was found in `chat.html` — every sink handling names, messages, room/meetup titles, reply snippets, and link previews is escaped, and the two sinks that looked unescaped at a glance (`location` coordinates, reaction emoji) are protected by server-side type/allowlist constraints. `sw.js`'s push/notificationclick/pushsubscriptionchange handling matches the documented local-first-ordering and unique-tag invariants with no changes needed. The real findings are three functional state-machine bugs around the E2EE/room-switch/reconnect edges (blank E2EE render on the REST fallback, a room-switch race that drops a pending read receipt, and a reconnect race that can silently vanish an unacked outgoing message) plus two low-impact code-hygiene gaps (one unescaped-but-unexploitable text sink, two small listener leaks). None block deploy; the two MEDIUM state-machine bugs are worth fixing soon since they degrade trust in the encrypted-DM and unread-badge features specifically.
