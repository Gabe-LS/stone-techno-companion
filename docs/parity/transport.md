# Transport Page: Behavioral Parity Specification

Source of truth for the current implementation:
- `services/companion/static/pages/transport.html` (the whole SPA, markup + inline CSS + inline JS)
- `services/companion/api.py` (`/api/transport/departures`, `/api/transport/walk`, `/public-transport`, `/timetable-transport.json`, the clean-URL static-page catch-all)
- `services/companion/static/timetable-transport.json` (generated data)
- `services/data/transport/capture-api.mjs` (the offline generator for that JSON)
- `tests/test_transport.py` and the four standalone Playwright checks

This file is the exhaustive, testable checklist for the Next.js port of `/transport`. The port ships only when every box below is checked against a real running instance of the port, not against this document alone. Every item states an input and the exact observable output; where the current code and `CLAUDE.md`'s prose summary disagree, this file follows the code (verified by reading and by the test suite) and the disagreement is flagged explicitly in "Notes on documentation drift" at the end.

The Python endpoints described here (`/api/transport/departures`, `/api/transport/walk`) stay in the existing FastAPI service. The Next.js port only consumes them over HTTP; it must not reimplement the EFA/VRR/OSRM proxying, the stop-id pinning, or the rate limiting in the new stack unless a later migration phase explicitly moves the backend too.

## 1. Routing and URL scheme

- [ ] `GET /transport` with no query string renders the Zollverein -> Essen Hbf outbound view (route internally `"zollverein"`, direction `"outbound"`), same as `?route=zollverein-essen`.
- [ ] `?route=zollverein-essen` resolves to `(route="zollverein", direction="outbound")`.
- [ ] `?route=essen-zollverein` resolves to `(route="zollverein", direction="inbound")`.
- [ ] `?route=dus-airport-essen` resolves to `(route="duesseldorf", direction="outbound")`.
- [ ] `?route=essen-dus-airport` resolves to `(route="duesseldorf", direction="inbound")`.
- [ ] Legacy alias `?route=duesseldorf` resolves to `(route="duesseldorf", direction="outbound")`, identical to `dus-airport-essen`.
- [ ] Legacy alias `?route=duesseldorf-essen` resolves identically to `dus-airport-essen` (present in the slug map even though not surfaced in any UI control).
- [ ] Legacy alias `?route=essen-duesseldorf` resolves identically to `essen-dus-airport` (present in the slug map even though not surfaced in any UI control).
- [ ] An unrecognized `?route=` value (anything not in the slug map, e.g. `?route=bogus`) falls back to the default (`zollverein` outbound) exactly like no param at all.
- [ ] `GET /public-transport` returns HTTP 301 with `Location: /transport` (old bookmarked URL kept alive). Verify status code is exactly 301, not 302 or 307.
- [ ] `/public-transport?route=dus-airport-essen` (with a query string) still 301-redirects to `/transport`. Confirm whether the query string is preserved or dropped by testing the actual `Location` header byte-for-byte (current server code passes no query string through: `RedirectResponse("/transport", status_code=301)`, so the query string is dropped and the redirected page lands on the default zollverein-essen view, not on the requested DÃžsseldorf view. This looks like an oversight but is the current, testable behavior; port must match it exactly unless product explicitly asks for the fix).
- [ ] Clean URL routing: `/transport` is served via the generic static-page catch-all (`static/pages/{slug}.html`, slug regex `[a-z0-9][a-z0-9-]*`, no per-route registration needed). The Next.js port's routing for `/transport` must behave equivalently: a single route owns all four directional views, distinguished only by `?route=`, not by four separate paths.
- [ ] The in-page direction-swap icon (`#dir-toggle`) rewrites only the `route` query param via `history.replaceState`, preserving any other query params already on the URL (e.g. test-only `?date=`/`?time=` overrides) and the path (`/transport`). No page reload occurs.
- [ ] After clicking the swap icon on `zollverein-essen`, the URL becomes `/transport?route=essen-zollverein` and clicking again returns to `/transport?route=zollverein-essen` (round-trips exactly, confirmed by `transport_reverse_check.py`).
- [ ] After clicking the swap icon on `dus-airport-essen`, the URL becomes `/transport?route=essen-dus-airport` and back.
- [ ] Reloading the page at a URL produced by the swap (e.g. `/transport?route=essen-zollverein`) reproduces the exact same view as before the reload (shareable/reloadable direction views).
- [ ] Loading the bare `/transport` URL (no `?route=`) does NOT get canonicalized into `?route=zollverein-essen` in the address bar; the URL stays exactly as typed even though the default view renders. Only the swap-icon click (`toggleDirection`) ever calls `history.replaceState`. (Contrast with the lineup page's own `/` -> `/line-up` canonicalization, which does not apply here.)
- [ ] Desktop command-bar buttons for "Zollverein" and "DUS Airport" only ever navigate to the two `*-essen` outbound slugs (`zollverein-essen`, `dus-airport-essen`). Full page navigation via `location.href`, not client-side routing. Never to the two `essen-*` inbound slugs. Only the in-page swap icon or the mobile submenu can reach the inbound slugs directly.
- [ ] Desktop command-bar "active" state (`.active` class / `[data-route]`) reflects only the itinerary (`zollverein` vs `duesseldorf`), not the direction: viewing `essen-zollverein` still highlights the "Zollverein" cmd-bar button, not a hidden fourth state.
- [ ] Mobile hamburger submenu "active" state (`[data-slug]`) reflects the exact 4-way slug, so `essen-zollverein` highlights only that one row, distinct from `zollverein-essen`.

## 2. Static schedule data (`timetable-transport.json`)

- [ ] `GET /timetable-transport.json` returns HTTP 200 with `Content-Type: application/json` and header `Cache-Control: no-cache` (so a regenerated schedule via `git pull` is picked up without a hard cache, but still allows conditional revalidation). Confirm the port serves this exact header on whatever route/mechanism replaces it.
- [ ] Top-level JSON shape: `{ route, stop, days, reverse, duesseldorf }`.
  - `route`: `{ from, to, fromId, toId }` for the Zollverein outbound leg (e.g. `from: "Zollverein"`, `to: "Essen Hbf"`, `fromId: "20009206"`, `toId: "20009289"`).
  - `stop`: `{ lat, lng }`. The outbound (Zollverein) stop coordinates, `{51.486095, 7.046062}`.
  - `days`: array of day objects for the Zollverein outbound board.
  - `reverse`: `{ route, stop, days }` for the Zollverein inbound (Essen Hbf -> Zollverein) board. Note `reverse` has NO nested `reverse` key (it is a leaf).
  - `duesseldorf`: `{ route, stop, days, reverse }`. Same shape one level down, for the DUS Airport itinerary; its own `reverse` is `{ route, stop, days }`.
- [ ] Each day object: `{ day, date, departures }` where `day` is an English weekday name (`"Thursday"`, `"Friday"`, `"Saturday"`, `"Sunday"`) and `date` is `DD.MM.YYYY` with DOTS as separators (not slashes. The client converts dots to slashes only for display).
- [ ] The Zollverein board (`days` and `reverse.days`) has exactly 3 day entries: Friday, Saturday, Sunday (festival days only).
- [ ] The DÃžsseldorf board (`duesseldorf.days` and `duesseldorf.reverse.days`) has exactly 4 day entries: Thursday, Friday, Saturday, Sunday. Thursday is the extra "arrival day" that only the airport itinerary has.
- [ ] Each departure object for the Zollverein board: `{ dep, arr, line, direction, duration, platform }` plus an optional `nextDay: true`. `line` is `"107"` or `"NE2"`. There is no `badge` field on Zollverein departures (the client derives the CSS badge class from `line` directly: `l-107` / `l-NE2`).
- [ ] Each departure object for the DÃžsseldorf board additionally carries `badge` (`"re"` for any `RE*`/`RB*`/plain numeric line, `"s"` for any line starting with `S` including the literal value `"SEV"` which is a rail-replacement bus, not an S-Bahn. This is a pre-existing data-generation quirk in `capture-api.mjs`'s `badge: /^S/.test(line) ? 's' : 're'` rule; reproduce it as-is, do not "fix" the SEV classification during the port).
- [ ] `direction` on the DÃžsseldorf board is the train's shortened headsign/terminus (e.g. `"Hamm"`, `"Dortmund"`, `"MÃžnster"`, `"KÃķln"`), produced at generation time by `shortenHeadsign()` (strips a trailing parenthetical and a trailing `Hauptbahnhof|Hbf|Bahnhof|Bhf|Bf`), never the literal string `"Essen"` or a bare `"Hbf"`.
- [ ] Each calendar day's `departures` array is midnight-to-midnight for that literal calendar date (`day`/`date`), PLUS up to 5 extra entries copied from the START of the NEXT calendar day's own departures, each flagged `nextDay: true`. These are for late-night/early-morning continuity display, not for a different calendar date's own tab. The last day in each `days` array (Sunday) has no such overflow (nothing follows it).
- [ ] A `nextDay: true` departure's own `dep` clock time is still whatever the next real calendar day's early-morning time is (e.g. `00:09`, `01:09` for NE2 owl service). The client, not the data, is responsible for treating it as "1440 minutes later" for sort/highlight purposes (see section 7).
- [ ] `platform` may be an empty string (seen on some SEV entries). The port's rendering must treat `""` the same as absent (no "Pl. " sub-line segment emitted), matching the client's `if (plat) subParts.push(...)` truthiness check.
- [ ] The generator (`services/data/transport/capture-api.mjs`) is a manually-run Node 18+ script, not a live pipeline step: regenerating the schedule for new event dates means editing its `dates` / `duesDates` arrays and re-running it, then deploying the output JSON via git pull (no rebuild). The port does not need to replicate the generator logic, only accept whatever shape it produces (as specified above).

## 3. Realtime layer: server endpoints (existing FastAPI, consumed as-is)

### `GET /api/transport/departures`

- [ ] Required params: `date` (FastAPI path/query required, so a missing `date` returns HTTP 422, not 400) and `time` (validated manually inside the handler, so a missing or malformed `time` returns HTTP 400, not 422. This asymmetry between the two "required" params must be reproduced exactly if the port re-implements or proxies this contract).
- [ ] `date` must match `^\d{2}\.\d{2}\.\d{4}$` (e.g. `10.07.2026`). Anything else (missing dots, wrong digit counts, `2026-07-10`) is HTTP 400 with body `{"detail": "date (DD.MM.YYYY) and time (HH:MM) required"}`.
- [ ] `time` must match `^\d{1,2}:\d{2}$` (colon required, e.g. `14:00` or `9:05`). **Pin this explicitly: `time=1400` (no colon) MUST 400.** This exact regression (a client sending `HHMM` instead of `HH:MM`) silently broke ALL realtime for a period in production; the port's client must never construct this parameter without the colon, and the port's test suite must include a case that would have caught it.
- [ ] Optional `route` param: `"zollverein"` (default when absent) or `"duesseldorf"`. Any other value falls through to the Zollverein (EFA departure-monitor) branch. It is not rejected, just silently treated as the default (worth flagging: there is no explicit 400 for an unrecognized `route` value on this endpoint, unlike `dir`).
- [ ] Optional `dir` param: `"outbound"` (default) or `"inbound"`. Any other value (e.g. `sideways`) is HTTP 400, `{"detail": "dir must be 'outbound' or 'inbound'"}`. Validated against `_TRANSPORT_DIRECTIONS` for the Zollverein branch and `_TRANSPORT_DUES_TRIP` for the DÃžsseldorf branch.
- [ ] **Zollverein branch** (`route` absent or `"zollverein"`): pinned stop id is `20009206` (outbound, Zollverein) or `20009289` (inbound, Essen Hbf). Never derived from client input. Queries `https://efa.vrr.de/vrr/XSLT_DM_REQUEST` with `stateless=1, type_dm=stop, mode=direct, useRealtime=1, limit=30`. Result is filtered to `servingLine.number` in `("107", "NE2")` only, and to entries whose `servingLine.direction` contains one of the direction's allowed destination substrings (outbound: `"Bredeney"` or `"Hauptbahnhof"`; inbound: `"Hanielstr"` or `"Gelsenkirchen"`).
- [ ] Mapped Zollverein departure entry fields: `line`, `direction`, `platform`, `scheduled` (`HH:MM`), `scheduledDate` (`DD.MM.YYYY`), `realtime` (bool, from `servingLine.realtime === "1"`), and conditionally `real` (`HH:MM`, from `realDateTime`), `delay` (int minutes, from `servingLine.delay`), `status` (from `realtimeTripStatus`, e.g. `"MONITORED"` or `"CANCELED"`), `countdown` (int, minutes to arrival at the stop per EFA).
- [ ] **DÃžsseldorf branch** (`route=duesseldorf`): pinned origin/destination ids from `_TRANSPORT_DUES_TRIP`. Outbound `20018488` (D-Flughafen) -> `20009289` (Essen Hbf); inbound the reverse. Queries the VRR journey planner `XML_TRIP_REQUEST2` with `maxChanges=0`, `useRealtime=1`, `routeType=LEASTTIME`, up to 3 paginated calls advancing by the last journey's departure + 1 minute, stopping once 6 results are collected or no more journeys are returned.
- [ ] DÃžsseldorf realtime filtering: only journeys with `interchanges === 0` (direct trips) are kept; only lines matching `^(RE|RB|S)` (regex on `disassembledName` or `number`) are kept. This excludes ICE/IC long-distance trains, which surface as bare non-RE/RB/S train numbers/classes in the upstream feed.
- [ ] DÃžsseldorf realtime timestamps: upstream `departureTimePlanned`/`departureTimeEstimated`/`arrivalTimePlanned`/`arrivalTimeEstimated` are ISO-8601 UTC; the server converts each to Europe/Berlin `HH:MM` + `DD.MM.YYYY` via `_berlin_hhmm_date`. Verify a UTC timestamp during CEST (summer, UTC+2) converts +2h, e.g. `2026-07-10T12:03:00Z` -> `14:03` on `10.07.2026` (this is exactly what `test_route_maps_trip_realtime` asserts).
- [ ] Mapped DÃžsseldorf entry fields: `line`, `scheduled`, `scheduledDate`, `realtime` (bool: whether an estimated departure time was present at all), `platform` (from `origin.properties.platform`), `trainNumber` (from `transportation.properties.trainNumber`), `arr` (planned arrival `HH:MM`), and conditionally `real`/`delay` (departure) and `arrReal`/`arrDelay` (arrival), each delay computed in minutes via `round((estimated - planned).total_seconds() / 60)`.
- [ ] DÃžsseldorf entries have **no** `direction`, `status`, or `countdown` fields. The client's destination text for these rows comes from the STATIC schedule (`dep.direction`, the pre-shortened headsign), never from the realtime payload.
- [ ] "Realtime only for now": the journey-planner query is always anchored at the literal `date`+`time` params passed in; requesting a FUTURE festival date returns whatever the journey planner has scheduled for that date with `useRealtime=1`, but since the upstream service does not carry live estimates for non-today dates, `real`/`delay`/`arrReal`/`arrDelay` will be absent and every entry renders as scheduled-only (no red delay style, no green/red rt-dot, live-indicator stays hiden because `isToday` gates the whole realtime fetch client-side. See section 4).
- [ ] Response shape (both branches): `{"departures": [...], "ts": "<ISO-8601 UTC timestamp of THIS response>"}`, plus `"stale": true` only when the upstream call failed and a previously cached (but expired) result was served instead.
- [ ] **Cache**: keyed by `(date, dir)` for Zollverein or `(date, "dues:" + dir)` for DÃžsseldorf. **There is no minute or per-route (beyond the `dues:` prefix) component in the key, and no `time` component at all.** A cache entry is reused verbatim for 90 seconds from when it was first populated (`now - cached[0] < 90`), confirmed by code and by `test_cache_collapses_upstream_calls` ("same date+dir with a different time still hits the cache, no time in key"). Note that a cache hit still returns a freshly-generated `ts` field each time (see below), so equal `ts` values do not indicate two separate cache hits. **This differs from `CLAUDE.md`'s prose claim of "a shared 55 s cache per (date, minute, route:dir) bucket": the actual, tested behavior is a 90-second TTL keyed by (date, dir) with no minute bucketing.** The port must match the CODE (90s, no minute key), not the stale prose; see "Notes on documentation drift" below.
- [ ] Cache eviction: when `_transport_cache` exceeds 50 entries, the single oldest entry (by cache-write time) is evicted. An LRU-by-age cap, not a TTL sweep, so a stale (>90s) entry can still be served as the "stale" fallback on a subsequent upstream failure until it ages out of the 50-slot cap.
- [ ] Concurrency: a per-key `asyncio.Lock` collapses concurrent requests for the same `(date, dir)` bucket into a single upstream call (double-checked cache read before and after acquiring the lock).
- [ ] Upstream failure (non-2xx or exception) with NO usable cache entry present: HTTP 502, `{"detail": "Departure service unavailable"}`.
- [ ] Upstream failure WITH a cache entry present (even if expired past 90s): HTTP 200, the cached `departures` array, `"stale": true`, and a freshly-generated `ts` (the port must not assume `stale: true` implies an old `ts`. The timestamp always reflects the moment of THIS response, not the age of the data).
- [ ] Rate limit: 30 requests per rolling 60-second window, keyed by client IP (`request.client.host`, `"unknown"` if absent). The 31st request within the window returns HTTP 429. **This counter is a single per-IP bucket SHARED across `/api/transport/departures` (limit checked at 30) and `/api/transport/walk` (limit checked at 10). Calls to one endpoint consume the same underlying hit list as the other,** even though each call site enforces a different numeric `limit`. A client that has already made 25 walk-endpoint calls in the current window has only 5 departures-endpoint calls left before it starts getting 429s from THAT endpoint's check (since the same `_transport_rate[ip]` list is shared, and the walk endpoint's own check would already have 429'd it at its own limit=10 well before that). Reproduce this shared-bucket behavior exactly, or explicitly decide (and document) that the port intentionally separates the two limiters. But the current system does not separate them.

### `GET /api/transport/walk`

- [ ] Required params: `lat`, `lng` (floats).
- [ ] Optional `route` (`"zollverein"` default or `"duesseldorf"`) and `dir` (`"outbound"` default or `"inbound"`).
- [ ] Walk target (destination fed to OSRM) is resolved SERVER-SIDE only, never from client input beyond the `route`/`dir` enum selectors:
  - `route` absent/`"zollverein"`, `dir=outbound`: target = Zollverein stop `(51.486095, 7.046062)`.
  - `route` absent/`"zollverein"`, `dir=inbound`: target = Essen Hbf `(51.449732, 7.012213)`.
  - `route=duesseldorf`, `dir=inbound` (Essen -> airport, walker departs Essen Hbf): target = Essen Hbf `(51.449732, 7.012213)`.
  - `route=duesseldorf`, `dir` absent or `outbound` (walker departs the airport): target = D-Flughafen Bf `(51.291368, 6.787158)`.
  - Any other `route`/`dir` combination not covered above: HTTP 400, `{"detail": "invalid route/dir"}`.
- [ ] Bounds check BEFORE hitting OSRM: `abs(lat - 51.486095) > 0.35` OR `abs(lng - 7.046062) > 0.5` -> HTTP 400, `{"detail": "Out of service area"}` (this bound is centered on the Zollverein stop regardless of which route/dir was requested. E.g. Munich `(48.13, 11.58)` is rejected; the DUS Airport coordinates themselves, `(51.291368, 6.787158)`, are comfortably inside this box since Essen-area bounding is intentionally generous).
- [ ] Rate limit: 10 requests per rolling 60-second window per client IP, sharing the same underlying bucket as `/api/transport/departures` (see above).
- [ ] Proxies to FOSSGIS's real foot-profile OSRM (`https://routing.openstreetmap.de/routed-foot/route/v1/foot/{lng},{lat};{tgtLng},{tgtLat}?overview=false`). Never the `project-osrm.org` demo server, which only hosts a car profile and would silently mis-route a 50-minute walk as a 9-minute drive.
- [ ] Success: HTTP 200, `{"distanceM": <meters, float>, "durationS": <seconds, float>}` (bare object, no wrapper key).
- [ ] `code !== "Ok"` or empty `routes` from OSRM: HTTP 502, `{"detail": "No route found"}`.
- [ ] Any network/timeout exception talking to OSRM: HTTP 502, `{"detail": "Routing service unavailable"}`.
- [ ] Coordinates are used transiently only: never written to a database, never logged, per the code's own docstring. Verify the port's replacement (if any) upholds the same "never stored, never logged" invariant for GPS coordinates.

## 4. Realtime layer: client behavior

- [ ] On initial load and after every day-tab click and every direction toggle, the client calls `fetchRealtime()` immediately (not just on a timer).
- [ ] A `setInterval` re-calls `fetchRealtime()` every **90000 ms (90 seconds)**, not 30 seconds. **Pin this precisely: the client-side `POLL_INTERVAL` constant is `90000`. This differs from `CLAUDE.md`'s prose claim of "polls ... every 30 s". See "Notes on documentation drift" below.**
- [ ] A SEPARATE `setInterval` fires every exactly 30000 ms (30 seconds) and, only when `data` is loaded AND realtime is currently NOT active (`!realtimeActive`), calls `renderPanel()` (a pure re-render, no network call) so that past/next/too-late row highlighting keeps advancing with the wall clock even when no live data is available (e.g. viewing a non-today date, or the last realtime fetch failed). This is the actual mechanism behind any "updates every 30s" impression; the port must keep it distinct from the 90s network poll.
- [ ] `fetchRealtime()` is a no-op (does nothing, does not clear existing state) whenever the currently active day tab's `date` is not today's real calendar date (compared via `getNow()`, see below). Realtime is only ever fetched for the day tab matching today.
- [ ] When the active day IS today, the request is built as `GET /api/transport/departures?date=<DD.MM.YYYY>&time=<HH:MM>&dir=<outbound|inbound>&route=<zollverein|duesseldorf>&limit=5`. **The `limit=5` query param sent by the client is dead: the server endpoint does not read a `limit` param at all (it hardcodes `limit=30` upstream to EFA for Zollverein, and `want=6` internally for DÃžsseldorf). Reproducing the exact request string (including the inert `limit=5`) is optional for parity, but the port must NOT assume only 5 entries come back:** up to 30 (post-filter) Zollverein entries or up to 6 DÃžsseldorf entries can be returned and are all cached client-side.
- [ ] `getNow()` reads `?date=DD.MM.YYYY&time=HH:MM` from the URL query string (both must be present) to construct a synthetic "current moment" for ALL time-relative logic (today-detection, past/next/too-late, the realtime request's own `date`/`time` params); absent either param, it uses the real `new Date()`. This override exists specifically to make the standalone Playwright checks deterministic and must be preserved by the port (it is exercised directly by `transport_duesseldorf_realtime_check.py`).
- [ ] On a successful fetch (response has a `departures` array, even if empty): the live indicator becomes visible (`.active` class added to `#live-indicator`), and `#live-updated`'s text becomes `"Updated DD/MM/YYYY at HH:MM"` computed from `result.ts` in the BROWSER's local time zone (via native `Date` parsing, no explicit TZ conversion in the client). Verify this is Europe/Berlin-equivalent in the deployed environment, not literally UTC.
- [ ] Each returned departure is written into an in-memory `realtimeCache` keyed by `line + "|" + scheduledTime + "|" + scheduledDate` (exact string join with pipe separators), so it can later be matched against the static schedule's `dep.line`/`dep.dep`/`dayData.date` triple for the currently rendered day panel.
- [ ] Any returned departure whose `scheduledDate` differs from the request's own `dateStr` AND that has a truthy `real` field is additionally pushed into a `realtimeSpillover` list (previous-day service running late into today). This spillover list is REPLACED (not appended) on every successful fetch.
- [ ] On fetch failure (network error, non-JSON, thrown exception): the live indicator's `.active` class is removed (dot/timestamp hidden) and `realtimeActive` is set false; the previously-rendered departure list is left untouched (no re-render is forced, no error message shown to the user).
- [ ] A `stale: true` field in the response is NOT specially handled by the client: a stale response is treated identically to a fresh one (live indicator still shown as active, `Updated ...` timestamp still uses the response's `ts`, which is always "now", not the age of the underlying cached data). The port must not assume the UI ever visibly distinguishes fresh vs. stale realtime data.
- [ ] Switching direction (`toggleDirection`) unconditionally WIPES `realtimeCache` and `realtimeSpillover` and deactivates the live indicator BEFORE re-rendering and re-fetching. A stale cache entry from the other direction must never bleed into the new direction's rows (same line+time+date could otherwise coincidentally collide).
- [ ] DÃžsseldorf-route realtime spillover entries carry an empty destination string on their synthetic day-panel row (`sp.direction || ""`), because the DÃžsseldorf realtime payload has no `direction` field at all. Unlike Zollverein spillover rows, which do carry a real direction string from the realtime payload. This asymmetry is inherent to the upstream data shape and must be reproduced, not "fixed", unless a product decision changes the DÃžsseldorf realtime payload shape.

## 5. Walk time (Locate me / geolocation)

- [ ] Initial banner state on load: text `"Walk time unknown"`, button label `"Locate me"`, button not disabled.
- [ ] Clicking Locate me with no `navigator.geolocation` in the browser: banner text becomes `"Geolocation not supported"`; button stays `"Locate me"` (not disabled, no browser API call attempted).
- [ ] Clicking Locate me (geolocation available): button label becomes `"..."` and the button is disabled for the duration of the request; `navigator.geolocation.getCurrentPosition` is called with `{ enableHighAccuracy: true, timeout: 10000 }`.
- [ ] Geolocation permission denied / error callback: button re-enabled, label becomes `"Retry"`; banner text becomes `"Location access denied"`.
- [ ] Geolocation success: calls `GET /api/transport/walk?lat=<lat>&lng=<lng>&dir=<direction>&route=<route>` (note: params are NOT URL-encoded beyond default string interpolation. Verify the port encodes them safely for arbitrary float representations, e.g. negative coordinates).
- [ ] While that request is in flight: banner text becomes `"Calculating route..."`.
- [ ] Walk endpoint success (`distanceM` present): `walkMinutes = durationS ? ceil(durationS / 60) : ceil((distanceM/1000) / 4.5 * 60)` (4.5 km/h fallback walking speed used only if `durationS` is falsy). Banner text becomes `"<dist> / <mins> min to <curRoute().from>"` where `<dist>` is `fmtDist(distanceM)` and `<mins>` is `fmtNum(walkMinutes)`. Button label becomes `"Update"`.
- [ ] `fmtDist(meters)`: if `< 1000`, `round(meters)` formatted via `Number.toLocaleString("de-DE")` + `" m"`; else `(meters/1000).toFixed(1)` parsed back to float, formatted via `toLocaleString("de-DE")` + `" km"` (German locale grouping/decimal conventions. Verify with a value like 1234.5 m yields `"1,2 km"` under `de-DE`, not `"1.2 km"`).
- [ ] `fmtNum(n)`: `n.toLocaleString("de-DE")`. Applies to the minutes figure too.

## 11. Security invariants (stay server-side; port only consumes them)

- [ ] Every EFA/VRR stop id or origin/destination pair is a fixed, code-level constant (`_TRANSPORT_DIRECTIONS`, `_TRANSPORT_DUES_TRIP`, `_TRANSPORT_STOP_COORDS`) selected only by the small, validated `dir`/`route` enum. The client NEVER supplies a stop id, station name, or arbitrary origin/destination directly. This must remain true if/when the port introduces its own API layer in front of these endpoints: it must not add a code path that accepts an arbitrary stop id from the client, even for convenience/testing.
- [ ] The walk endpoint's OSRM target is likewise chosen only from the same small server-side enum, never from client-supplied coordinates (the client's `lat`/`lng` are the WALKER's position only, always paired with a server-chosen destination).
- [ ] The Essen-area bounding box on `/api/transport/walk` (`|lat-51.486095| <= 0.35`, `|lng-7.046062| <= 0.5`) exists specifically so the endpoint cannot be abused as a general-purpose routing proxy; a request from far outside the festival area is rejected before any upstream call is made.
- [ ] Both proxy endpoints exist because the upstream EFA API has no CORS headers; the client can only ever reach these upstreams through the server-side proxy, never directly from the browser.
- [ ] Rate limiting (30/min departures, 10/min walk, shared per-IP bucket as documented in section 3) exists to protect the public VRR/OSRM infrastructure from hundreds of festival phones polling simultaneously; the port must not remove or bypass it by, e.g., calling the upstream directly from a Next.js edge function without an equivalent limiter.
- [ ] GPS coordinates sent to `/api/transport/walk` are never persisted or logged server-side; verify the port's own logging (access logs, APM, etc.) does not incidentally capture raw `lat`/`lng` query params in a way the current system avoids.
- [ ] These endpoints, their pinned constants, and their rate limiting stay in the existing Python/FastAPI service for this phase of the migration; the Next.js port is a pure HTTP consumer of `/api/transport/departures` and `/api/transport/walk` and of the static `/timetable-transport.json`, and must not reimplement or fork the upstream-calling logic into the new stack.

1. **Realtime cache TTL and key.** `CLAUDE.md` says "a shared 55 s cache per (date, minute, route:dir) bucket". The code (`services/companion/api.py`, both the Zollverein and DÃžsseldorf branches of `/api/transport/departures`) uses a 90-second TTL (`now - cached[0] < 90`) keyed by `(date, dir)` (or `(date, "dues:"+dir)`), with NO minute component at all. Confirmed by `test_cache_collapses_upstream_calls`'s explicit assertion that a different `time` with the same `date`+`dir` still hits the cache. Whoever wrote the `CLAUDE.md` line either described an earlier version of the cache or a slightly different mental model of the bucketing; either way, 90 seconds / (date, dir)-only is what a byte-for-byte port must match.
2. **Client poll interval.** `CLAUDE.md` says the page "polls ... every 30 s when viewing today". The client's actual `POLL_INTERVAL` constant is `90000` (90 seconds) for the real network re-fetch of realtime data. There IS a genuinely separate 30-second `setInterval`, but it only triggers a local re-render (`renderPanel()`, no network call) and only when realtime is not currently active. A port that polls the network every 30 seconds would be denser than the current production behavior (3x the request rate) and would need a product decision, not just a bugfix, before shipping that as an intentional change.

One more finding worth flagging even though it isn't a `CLAUDE.md` contradiction: `transport_duesseldorf_realtime_check.py` Part B references CSS classes (`.dep-time-scheduled`, `.dep-time-real`, `.dep-arr-real`) that do not exist in the current `transport.html` markup at all. This test currently cannot pass as written. It is called out as its own checklist item in section 12 rather than buried here, since it blocks calling that script "gating" without a fix.
