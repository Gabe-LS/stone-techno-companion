## 2. Static schedule data (`timetable-transport.json`)

- [ ] `GET /timetable-transport.json` returns HTTP 200 with `Content-Type: application/json` and header `Cache-Control: no-cache` (so a regenerated schedule via `git pull` is picked up without a hard cache, but still allows conditional revalidation). Confirm the port serves this exact header on whatever route/mechanism replaces it. 200 status confirmed via the dev rewrite; response headers (Content-Type/Cache-Control) weren't separately asserted this pass -- the header is set by the unchanged backend and passed through by the Next.js dev rewrite unmodified, but this wasn't directly checked.
- [x] Top-level JSON shape: `{ route, stop, days, reverse, duesseldorf }`.
  - `route`: `{ from, to, fromId, toId }` for the Zollverein outbound leg (e.g. `from: "Zollverein"`, `to: "Essen Hbf"`, `fromId: "20009206"`, `toId: "20009289"`).
  - `stop`: `{ lat, lng }`. The outbound (Zollverein) stop coordinates, `{51.486095, 7.046062}`.
  - `days`: array of day objects for the Zollverein outbound board.
  - `reverse`: `{ route, stop, days }` for the Zollverein inbound (Essen Hbf -> Zollverein) board. Note `reverse` has NO nested `reverse` key (it is a leaf).
  - `duesseldorf`: `{ route, stop, days, reverse }`. Same shape one level down, for the DUS Airport itinerary; its own `reverse` is `{ route, stop, days }`.
  (Directly inspected the real `services/companion/static/timetable-transport.json` via `python3 -m json`, and this exact shape is what `lib/transport/types.ts`/`logic.ts` are written against and what rendered correctly through the whole app.)
- [x] Each day object: `{ day, date, departures }` where `day` is an English weekday name (`"Thursday"`, `"Friday"`, `"Saturday"`, `"Sunday"`) and `date` is `DD.MM.YYYY` with DOTS as separators (not slashes. The client converts dots to slashes only for display).
- [x] The Zollverein board (`days` and `reverse.days`) has exactly 3 day entries: Friday, Saturday, Sunday (festival days only).
- [x] The Düsseldorf board (`duesseldorf.days` and `duesseldorf.reverse.days`) has exactly 4 day entries: Thursday, Friday, Saturday, Sunday. Thursday is the extra "arrival day" that only the airport itinerary has.
- [ ] Each departure object for the Zollverein board: `{ dep, arr, line, direction, duration, platform }` plus an optional `nextDay: true`. `line` is `"107"` or `"NE2"`. There is no `badge` field on Zollverein departures (the client derives the CSS badge class from `line` directly: `l-107` / `l-NE2`). Field shape confirmed by inspection and by rendering (`l-107`/`l-NE2` badges visible in screenshots); the `nextDay` overflow behavior specifically wasn't exercised this pass.
- [x] Each departure object for the Düsseldorf board additionally carries `badge` (`"re"` for any `RE*`/`RB*`/plain numeric line, `"s"` for any line starting with `S` including the literal value `"SEV"` which is a rail-replacement bus, not an S-Bahn. This is a pre-existing data-generation quirk in `capture-api.mjs`'s `badge: /^S/.test(line) ? 's' : 're'` rule; reproduce it as-is, do not "fix" the SEV classification during the port). Confirmed visually: the DUS Airport board's `SEV` rows render with the same green `l-s` badge style as real S-Bahn lines, not the magenta `l-re` style.
- [x] `direction` on the Düsseldorf board is the train's shortened headsign/terminus (e.g. `"Hamm"`, `"Dortmund"`, `"Münster"`, `"Köln"`), produced at generation time by `shortenHeadsign()` (strips a trailing parenthetical and a trailing `Hauptbahnhof|Hbf|Bahnhof|Bhf|Bf`), never the literal string `"Essen"` or a bare `"Hbf"`. Confirmed visually (Münster, Dortmund, Hamm, Kassel Bahnhof Wilhelmshöhe rows in the desktop screenshot).
- [ ] Each calendar day's `departures` array is midnight-to-midnight for that literal calendar date (`day`/`date`), PLUS up to 5 extra entries copied from the START of the NEXT calendar day's own departures, each flagged `nextDay: true`. These are for late-night/early-morning continuity display, not for a different calendar date's own tab. The last day in each `days` array (Sunday) has no such overflow (nothing follows it). Not exercised this pass.
- [ ] A `nextDay: true` departure's own `dep` clock time is still whatever the next real calendar day's early-morning time is (e.g. `00:09`, `01:09` for NE2 owl service). The client, not the data, is responsible for treating it as "1440 minutes later" for sort/highlight purposes (see section 7). Implemented in `lib/transport/logic.ts`'s `buildRows()` (the `+1440` offset) but not exercised by a live-data test this pass.
- [ ] `platform` may be an empty string (seen on some SEV entries). The port's rendering must treat `""` the same as absent (no "Pl. " sub-line segment emitted), matching the client's `if (plat) subParts.push(...)` truthiness check. Implemented (`if (plat) subParts.push(...)` ported verbatim in `DepartureList.tsx`) but not exercised against a real empty-platform row this pass.
- [x] The generator (`services/data/transport/capture-api.mjs`) is a manually-run Node 18+ script, not a live pipeline step: regenerating the schedule for new event dates means editing its `dates` / `duesDates` arrays and re-running it, then deploying the output JSON via git pull (no rebuild). The port does not need to replicate the generator logic, only accept whatever shape it produces (as specified above). (Generator untouched; port only consumes its output.)

## 3. Realtime layer: server endpoints (existing FastAPI, consumed as-is)

All items in this section describe the **unchanged** backend (`services/companion/api.py`), which the Next.js port only calls over HTTP via the dev rewrite / production reverse proxy. Checked off on the strength of `tests/test_transport.py`'s 20/20 passing tests (part of the 280-test `pytest tests/ -q --ignore=tests/test_notifications.py` canary run this pass) -- exactly the tests this section's own gating checklist (section 12) names as the verification method for these behaviors -- plus direct inspection that the Next.js client (`TransportBoard.tsx`) only ever sends the documented `date`/`time`/`dir`/`route` query params, never a raw stop id or coordinate pair beyond the walker's own lat/lng.

### `GET /api/transport/departures`

- [x] Required params: `date` (FastAPI path/query required, so a missing `date` returns HTTP 422, not 400) and `time` (validated manually inside the handler, so a missing or malformed `time` returns HTTP 400, not 422. This asymmetry between the two "required" params must be reproduced exactly if the port re-implements or proxies this contract).
- [x] `date` must match `^\d{2}\.\d{2}\.\d{4}$` (e.g. `10.07.2026`). Anything else (missing dots, wrong digit counts, `2026-07-10`) is HTTP 400 with body `{"detail": "date (DD.MM.YYYY) and time (HH:MM) required"}`.
- [x] `time` must match `^\d{1,2}:\d{2}$` (colon required, e.g. `14:00` or `9:05`). **Pin this explicitly: `time=1400` (no colon) MUST 400.** This exact regression (a client sending `HHMM` instead of `HH:MM`) silently broke ALL realtime for a period in production; the port's client must never construct this parameter without the colon, and the port's test suite must include a case that would have caught it. The Next.js client's own request builder (`fetchRealtime` in `TransportBoard.tsx`) always formats `time` as `${pad2(hours)}:${pad2(minutes)}`, so it structurally cannot regress into the `HHMM` bug.
- [x] Optional `route` param: `"zollverein"` (default when absent) or `"duesseldorf"`. Any other value falls through to the Zollverein (EFA departure-monitor) branch. It is not rejected, just silently treated as the default (worth flagging: there is no explicit 400 for an unrecognized `route` value on this endpoint, unlike `dir`).
- [x] Optional `dir` param: `"outbound"` (default) or `"inbound"`. Any other value (e.g. `sideways`) is HTTP 400, `{"detail": "dir must be 'outbound' or 'inbound'"}`. Validated against `_TRANSPORT_DIRECTIONS` for the Zollverein branch and `_TRANSPORT_DUES_TRIP` for the Düsseldorf branch.
- [x] **Zollverein branch** (`route` absent or `"zollverein"`): pinned stop id is `20009206` (outbound, Zollverein) or `20009289` (inbound, Essen Hbf). Never derived from client input. Queries `https://efa.vrr.de/vrr/XSLT_DM_REQUEST` with `stateless=1, type_dm=stop, mode=direct, useRealtime=1, limit=30`. Result is filtered to `servingLine.number` in `("107", "NE2")` only, and to entries whose `servingLine.direction` contains one of the direction's allowed destination substrings (outbound: `"Bredeney"` or `"Hauptbahnhof"`; inbound: `"Hanielstr"` or `"Gelsenkirchen"`).
- [x] Mapped Zollverein departure entry fields: `line`, `direction`, `platform`, `scheduled` (`HH:MM`), `scheduledDate` (`DD.MM.YYYY`), `realtime` (bool, from `servingLine.realtime === "1"`), and conditionally `real` (`HH:MM`, from `realDateTime`), `delay` (int minutes, from `servingLine.delay`), `status` (from `realtimeTripStatus`, e.g. `"MONITORED"` or `"CANCELED"`), `countdown` (int, minutes to arrival at the stop per EFA).
- [x] **Düsseldorf branch** (`route=duesseldorf`): pinned origin/destination ids from `_TRANSPORT_DUES_TRIP`. Outbound `20018488` (D-Flughafen) -> `20009289` (Essen Hbf); inbound the reverse. Queries the VRR journey planner `XML_TRIP_REQUEST2` with `maxChanges=0`, `useRealtime=1`, `routeType=LEASTTIME`, up to 3 paginated calls advancing by the last journey's departure + 1 minute, stopping once 6 results are collected or no more journeys are returned.
- [x] Düsseldorf realtime filtering: only journeys with `interchanges === 0` (direct trips) are kept; only lines matching `^(RE|RB|S)` (regex on `disassembledName` or `number`) are kept. This excludes ICE/IC long-distance trains, which surface as bare non-RE/RB/S train numbers/classes in the upstream feed.
- [x] Düsseldorf realtime timestamps: upstream `departureTimePlanned`/`departureTimeEstimated`/`arrivalTimePlanned`/`arrivalTimeEstimated` are ISO-8601 UTC; the server converts each to Europe/Berlin `HH:MM` + `DD.MM.YYYY` via `_berlin_hhmm_date`. Verify a UTC timestamp during CEST (summer, UTC+2) converts +2h, e.g. `2026-07-10T12:03:00Z` -> `14:03` on `10.07.2026` (this is exactly what `test_route_maps_trip_realtime` asserts).
- [x] Mapped Düsseldorf entry fields: `line`, `scheduled`, `scheduledDate`, `realtime` (bool: whether an estimated departure time was present at all), `platform` (from `origin.properties.platform`), `trainNumber` (from `transportation.properties.trainNumber`), `arr` (planned arrival `HH:MM`), and conditionally `real`/`delay` (departure) and `arrReal`/`arrDelay` (arrival), each delay computed in minutes via `round((estimated - planned).total_seconds() / 60)`. Also directly exercised end-to-end client-side via a mocked response with all these fields (`tests/web/transport_nextjs_check.py` round-2 coverage): platform + trainNumber sub-line and delayed arrival both rendered correctly.
- [x] Düsseldorf entries have **no** `direction`, `status`, or `countdown` fields. The client's destination text for these rows comes from the STATIC schedule (`dep.direction`, the pre-shortened headsign), never from the realtime payload. (`RowSource.direction` in `lib/transport/logic.ts` is always taken from the static `dep`, never from `rt`.)
- [x] "Realtime only for now": the journey-planner query is always anchored at the literal `date`+`time` params passed in; requesting a FUTURE festival date returns whatever the journey planner has scheduled for that date with `useRealtime=1`, but since the upstream service does not carry live estimates for non-today dates, `real`/`delay`/`arrReal`/`arrDelay` will be absent and every entry renders as scheduled-only (no red delay style, no green/red rt-dot, live-indicator stays hiden because `isToday` gates the whole realtime fetch client-side. See section 4).
- [x] Response shape (both branches): `{"departures": [...], "ts": "<ISO-8601 UTC timestamp of THIS response>"}`, plus `"stale": true` only when the upstream call failed and a previously cached (but expired) result was served instead.
- [x] **Cache**: keyed by `(date, dir)` for Zollverein or `(date, "dues:" + dir)` for Düsseldorf. **There is no minute or per-route (beyond the `dues:` prefix) component in the key, and no `time` component at all.** A cache entry is reused verbatim for 90 seconds from when it was first populated (`now - cached[0] < 90`), confirmed by code and by `test_cache_collapses_upstream_calls` ("same date+dir with a different time still hits the cache, no time in key"). Note that a cache hit still returns a freshly-generated `ts` field each time (see below), so equal `ts` values do not indicate two separate cache hits. **This differs from `CLAUDE.md`'s prose claim of "a shared 55 s cache per (date, minute, route:dir) bucket": the actual, tested behavior is a 90-second TTL keyed by (date, dir) with no minute bucketing.** The port must match the CODE (90s, no minute key), not the stale prose; see "Notes on documentation drift" below. The Next.js client's own 90s `POLL_INTERVAL_MS` matches this TTL by construction.
- [x] Cache eviction: when `_transport_cache` exceeds 50 entries, the single oldest entry (by cache-write time) is evicted. An LRU-by-age cap, not a TTL sweep, so a stale (>90s) entry can still be served as the "stale" fallback on a subsequent upstream failure until it ages out of the 50-slot cap.
- [x] Concurrency: a per-key `asyncio.Lock` collapses concurrent requests for the same `(date, dir)` bucket into a single upstream call (double-checked cache read before and after acquiring the lock).
- [x] Upstream failure (non-2xx or exception) with NO usable cache entry present: HTTP 502, `{"detail": "Departure service unavailable"}`.
- [x] Upstream failure WITH a cache entry present (even if expired past 90s): HTTP 200, the cached `departures` array, `"stale": true`, and a freshly-generated `ts` (the port must not assume `stale: true` implies an old `ts`. The timestamp always reflects the moment of THIS response, not the age of the data). The client treats `stale` identically to a fresh response by construction (`fetchRealtime` never reads `result.stale`).
- [x] Rate limit: 30 requests per rolling 60-second window, keyed by client IP (`request.client.host`, `"unknown"` if absent). The 31st request within the window returns HTTP 429. **This counter is a single per-IP bucket SHARED across `/api/transport/departures` (limit checked at 30) and `/api/transport/walk` (limit checked at 10). Calls to one endpoint consume the same underlying hit list as the other,** even though each call site enforces a different numeric `limit`. A client that has already made 25 walk-endpoint calls in the current window has only 5 departures-endpoint calls left before it starts getting 429s from THAT endpoint's check (since the same `_transport_rate[ip]` list is shared, and the walk endpoint's own check would already have 429'd it at its own limit=10 well before that). Reproduce this shared-bucket behavior exactly, or explicitly decide (and document) that the port intentionally separates the two limiters. But the current system does not separate them. (Unchanged backend; the Next.js port makes no independent server-side calls to the upstream, so this invariant is untouched.)

### `GET /api/transport/walk`

- [x] Required params: `lat`, `lng` (floats).
- [x] Optional `route` (`"zollverein"` default or `"duesseldorf"`) and `dir` (`"outbound"` default or `"inbound"`).
- [x] Walk target (destination fed to OSRM) is resolved SERVER-SIDE only, never from client input beyond the `route`/`dir` enum selectors:
  - `route` absent/`"zollverein"`, `dir=outbound`: target = Zollverein stop `(51.486095, 7.046062)`.
  - `route` absent/`"zollverein"`, `dir=inbound`: target = Essen Hbf `(51.449732, 7.012213)`.
  - `route=duesseldorf`, `dir=inbound` (Essen -> airport, walker departs Essen Hbf): target = Essen Hbf `(51.449732, 7.012213)`.
  - `route=duesseldorf`, `dir` absent or `outbound` (walker departs the airport): target = D-Flughafen Bf `(51.291368, 6.787158)`.
  - Any other `route`/`dir` combination not covered above: HTTP 400, `{"detail": "invalid route/dir"}`.
- [x] Bounds check BEFORE hitting OSRM: `abs(lat - 51.486095) > 0.35` OR `abs(lng - 7.046062) > 0.5` -> HTTP 400, `{"detail": "Out of service area"}` (this bound is centered on the Zollverein stop regardless of which route/dir was requested. E.g. Munich `(48.13, 11.58)` is rejected; the DUS Airport coordinates themselves, `(51.291368, 6.787158)`, are comfortably inside this box since Essen-area bounding is intentionally generous).
- [x] Rate limit: 10 requests per rolling 60-second window per client IP, sharing the same underlying bucket as `/api/transport/departures` (see above).
- [x] Proxies to FOSSGIS's real foot-profile OSRM (`https://routing.openstreetmap.de/routed-foot/route/v1/foot/{lng},{lat};{tgtLng},{tgtLat}?overview=false`). Never the `project-osrm.org` demo server, which only hosts a car profile and would silently mis-route a 50-minute walk as a 9-minute drive.

- [x] `code !== "Ok"` or empty `routes` from OSRM: HTTP 502, `{"detail": "No route found"}`.
- [x] Any network/timeout exception talking to OSRM: HTTP 502, `{"detail": "Routing service unavailable"}`.
- [x] Coordinates are used transiently only: never written to a database, never logged, per the code's own docstring. Verify the port's replacement (if any) upholds the same "never stored, never logged" invariant for GPS coordinates. (The Next.js client passes `lat`/`lng` straight through to the existing proxy endpoint and never stores or logs them itself.)

## 4. Realtime layer: client behavior

- [x] On initial load and after every day-tab click and every direction toggle, the client calls `fetchRealtime()` immediately (not just on a timer). (`TransportBoard.tsx`: the initial-data effect, `onTabClick`, and `toggleDirection` each call `fetchRealtime()` directly, not just via the interval.)
- [x] A `setInterval` re-calls `fetchRealtime()` every **90000 ms (90 seconds)**, not 30 seconds. **Pin this precisely: the client-side `POLL_INTERVAL` constant is `90000`.** Ported as `POLL_INTERVAL_MS = 90000` in `TransportBoard.tsx`.
- [x] A SEPARATE `setInterval` fires every exactly 30000 ms (30 seconds) and, only when `data` is loaded AND realtime is currently NOT active (`!realtimeActive`), calls `renderPanel()` (a pure re-render, no network call) so that past/next/too-late row highlighting keeps advancing with the wall clock even when no live data is available. Ported as `LOCAL_RENDER_INTERVAL_MS = 30000` driving a `tick` state bump (forces a re-render; no network call), gated by the same `!realtimeActive` check via a ref.
- [x] `fetchRealtime()` is a no-op (does nothing, does not clear existing state) whenever the currently active day tab's `date` is not today's real calendar date (compared via `getNow()`, see below). ## Notes on documentation drift (not checklist items, context for whoever ships the port)

Two claims in `CLAUDE.md`'s "Public transport page" section do not match the code as actually read and tested, and this spec follows the code:

1. **Realtime cache TTL and key.** `CLAUDE.md` says "a shared 55 s cache per (date, minute, route:dir) bucket". The code (`services/companion/api.py`, both the Zollverein and Düsseldorf branches of `/api/transport/departures`) uses a 90-second TTL (`now - cached[0] < 90`) keyed by `(date, dir)` (or `(date, "dues:"+dir)`), with NO minute component at all. Confirmed by `test_cache_collapses_upstream_calls`'s explicit assertion that a different `time` with the same `date`+`dir` still hits the cache. Whoever wrote the `CLAUDE.md` line either described an earlier version of the cache or a slightly different mental model of the bucketing; either way, 90 seconds / (date, dir)-only is what a byte-for-byte port must match.
2. **Client poll interval.** `CLAUDE.md` says the page "polls ... every 30 s when viewing today". The client's actual `POLL_INTERVAL` constant is `90000` (90 seconds) for the real network re-fetch of realtime data. There IS a genuinely separate 30-second `setInterval`, but it only triggers a local re-render (`renderPanel()`, no network call) and only when realtime is not currently active. A port that polls the network every 30 seconds would be denser than the current production behavior (3x the request rate) and would need a product decision, not just a bugfix, before shipping that as an intentional change.

One more finding worth flagging even though it isn't a `CLAUDE.md` contradiction: `transport_duesseldorf_realtime_check.py` Part B references CSS classes (`.dep-time-scheduled`, `.dep-time-real`, `.dep-arr-real`) that do not exist in the current `transport.html` markup at all. This test currently cannot pass as written. It is called out as its own checklist item in section 12 rather than buried here, since it blocks calling that script "gating" without a fix. This is unaffected by the Next.js port (which doesn't reuse this test or its class-name scheme).
