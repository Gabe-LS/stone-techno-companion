# Timetable View: Behavioral Parity Checklist

Scope: the TIMETABLE VIEW only, as generated by `services/data/scraper/render.py` (`render_output_html`, the `has_timetable` branches). Source of truth is that file; where it disagrees with `CLAUDE.md`, this checklist follows the code. Shared mechanics with the list view (hearts sync, modals, push notifications, sync/share) are included only where the timetable view depends on or diverges from them.

All class names, ids, CSS variables, and JS function names below are copied verbatim from `services/data/scraper/render.py`.

## Boot, Routing and View Class

- [ ] An inline `<head>` script (before any other JS) sets `document.documentElement.className` to `view-list` or `view-timetable` based on: URL path `/timetable` -> `timetable`, URL path `/line-up` -> `list`, otherwise `localStorage.stc_view` (`'timetable'` selects `timetable`, anything else `list`).
- [ ] That same script writes `localStorage.stc_view` back with the resolved value (it means "last view SEEN", not "last toggled": a direct `/timetable` or `/line-up` visit updates it too) and calls `history.replaceState` to rewrite the URL to the canonical `/timetable` or `/line-up` path.
- [ ] `body { opacity: 0 }` is set inline in `<head>` (first `<style>` block) so no view flashes before JS runs; it is set back to `1` only after `Promise.race([_initP, 5000ms timeout])` resolves (a hung `fetch()` cannot leave the page permanently invisible).
- [ ] CSS-only immediate view switch (works before JS finishes): `.view-list #timetable-view{display:none}` and `.view-timetable #list-view{display:none}`.
- [ ] CSS-only title/label swap for the pre-JS paint: `.view-timetable #page-title{font-size:0}` with `.view-timetable #page-title::after{content:'Timetable';font-size:var(--font-2xl)}` (and `var(--font-xl)` under `max-width:768px`); `.view-timetable #view-label{font-size:0}` with `::after{content:'Timetable';font-size:var(--font-sm)}`. The real `<h1 id="page-title">` and `<span id="view-label">` DOM text still say "Line-up" at this point: only the CSS pseudo-element paints "Timetable".
- [ ] Later, JS (`switchView`, and the deferred init IIFE for the initial view) performs the real text replace: `h1.textContent = h1.textContent.replace('Line-up', 'Timetable')`, so the actual DOM text (not just the CSS overlay) reflects the active view within one frame.
- [ ] `.view-timetable #btn-list{color:#aaa!important}` / `.view-timetable #btn-timetable{color:#fff!important}` (and mirrored for `.view-list`) tint the cmd-bar toggle buttons for the active view.
- [ ] `.view-list #dd-list{background:var(--gray-700)}` / `.view-timetable #dd-timetable{background:var(--gray-700)}` highlight the matching hamburger-dropdown row.
- [ ] `.view-timetable #btn-schedule{display:inline-block!important}` forces the "Show My Schedule" cmd-bar button visible only in timetable view (it is otherwise not shown; `switchView` also toggles its inline `display` directly).
- [ ] There is exactly one `<h1 id="page-title">` and one `<span id="view-label">` in the DOM, shared and re-labeled between list and timetable views (not two separate headings).

## Timetable Panel Data Model (server-rendered, per date+period)

- [ ] Each timetable "panel" corresponds to one `(date, period)` combination with at least one artist that has both `start_time` and `end_time` set; if none, no panel/section is emitted for that date+period.
- [ ] Floors within a panel are ordered by a canonical fixed list: `eisbahn, grand-hall, koksofenbatterie, listening-floor, mischanlage, salzlager, werksschwimmbad`, then any other floor ids present (not in that list) appended in encounter order.
- [ ] For `period == 'night'` panels, any start/end time before 12:00 (noon) is treated as after-midnight and shifted +1440 minutes (24h) for grid math (so a 01:00 end time sorts after a 23:00 start time).
- [ ] The grid's start minute (`grid_start`) is floored to the top of the hour (`(min_start // 60) * 60`); `grid_end` is the max end time (adjusted for night wraparound).
- [ ] Panel container: `<div class="timetable-panel" data-date="..." data-period="..." data-grid-start="N" data-grid-end="N" data-is-night="0|1" id="panel-{date}-{period}">`. Only one panel has class `active` at a time (`.timetable-panel{display:none}`, `.timetable-panel.active{display:block}`).
- [ ] A per-slot artist id (`slot_uuid`, defined once in `services/data/scraper/timetable_json.py` and reused by `render.py`) is `uuid5(NAMESPACE_URL, artist_ids_joined + ":" + date + ":" + period + ":" + floor_id)`, EXCEPT when the same `(artists, date, period, floor)` group has more than one distinct `(start,end)` pair: the earliest-starting slot keeps that base id, and only the later, colliding slot(s) get a disambiguated id with `:{start_time}:{end_time}` appended before hashing. This is the single id shared by every artist-block/table-cell rendering of that same slot (grid version and mobile-table version resolve to the identical id).
- [ ] A separate per-artist-per-card id exists for hearts: `uuid5(NAMESPACE_URL, f"{artist_id}:{date}:{period}:{floor_id_or_empty}")`, where the floor component is included ONLY when the period is `night` (`loc_for_id = fid if is_night else ''`); for `day` periods the per-artist heart id omits the floor.

## Desktop / Tablet Layout (CSS Grid, >480px)

- [ ] Filter bar (day/period tabs) is `<div class="filter-bar">` containing `<div class="day-tabs" id="day-tabs">` and `<div class="period-tabs" id="period-tabs">`; it is `position: sticky; top: var(--sticky-top-h2, 96px); z-index: 20;` with `min-height: calc(1.5 * var(--font-xl) + 19px)` (chosen so it equals the height of the list view's sticky `h2`, since they occupy the same sticky slot in their respective views).
- [ ] `--sticky-top-h1/h2/h3/h4` are runtime CSS custom properties computed by JS function `setStickyTops()` (cmd-bar height, plus h1 height, plus h2 height, plus h3 height, cumulatively) and recomputed on every window `resize` (debounced via `requestAnimationFrame`). Fallback literal values in the CSS (`28px`, `96px`, `150px`, `190px`) are used only before that JS runs.
- [ ] `.filter-bar::after` is an absolutely-positioned fade strip (`height: var(--space-sm)`, `background: var(--color-bg)`) covering the gap directly below the sticky bar so scrolling content never shows through.
- [ ] Floor header bar: `<div class="floor-header-bar">` pinned directly below the filter bar via `top: calc(var(--sticky-top-h2, 96px) + 1.5 * var(--font-xl) + 19px)`, `position: sticky`, `z-index: var(--z-sticky)`, using CSS grid `grid-template-columns: 40px repeat({num_floors}, var(--col-w, 1fr))`.
- [ ] `.floor-header-bar::after` is a 36px fade-gradient strip that is `opacity:0` by default and `opacity:1` when the bar has class `.stuck` (toggled by an `IntersectionObserver` watching a zero-size sentinel placed just before the bar, offset by `-{top}px`; `stuck` = the sentinel's `intersectionRatio === 0`, i.e., the bar has pinned).
- [ ] Each floor header cell: `<div class="floor-header floor-{fid}">` containing `<span>{floor name}</span>` and, if a curator/host note exists for that date+floor, `<span class="floor-curator">{text}</span>` immediately after (see "Floor Annotations" below). `.floor-header{background:none!important}` (forces no background on the outer cell: only the inner `<span>` is colored).
- [ ] `.floor-header > span:first-child` is the visible colored pill: `font-weight:700; font-size:var(--font-sm); padding:var(--space-sm) var(--space-md); border-radius:var(--radius-pill); white-space:nowrap; overflow:hidden; text-overflow:ellipsis`.
- [ ] The main grid: `<div class="timetable" data-num-floors="N" style="grid-template-columns: 40px repeat(N, var(--col-w, 1fr)); grid-template-rows: auto repeat({total_minutes}, 2px)">`, i.e. exactly 2px of row height per minute of the panel's time range.
- [ ] Column 1 is reserved for time labels/grid-lines gutter; floor columns start at grid column 2. `.grid-line{grid-column: 2 / -1}` (spans all floor columns, not column 1).
- [ ] Hour lines: for every whole hour inside `[grid_start, grid_end]`, a `<div class="time-label">{HH}:00</div>` at `grid-column:1` and a `<div class="grid-line hour">` (`border-top: 1px solid var(--color-line-hour)`) at the same grid row; a half-hour line (`<div class="grid-line half">`, `border-top: 1px dashed var(--color-line-half)`) is added 30 rows later if it still falls inside the panel.
- [ ] `.time-label{font-size:var(--font-xs); color:var(--color-muted); text-align:right; padding-right:var(--space-sm); position:relative; top:calc(-0.5em + 1px)}` (nudged up half a line so the label straddles its hour boundary rather than sitting below it).
- [ ] `.tt-block` cards are absolutely-positioned via CSS grid placement: `grid-column:{floor col index}; grid-row:{row_start} / {row_end}` where `row_start = (start_min - grid_start) + 2` and `row_end = (end_min - grid_start) + 2` (the `+2` accounts for the 1 header row plus 1-indexing).
- [ ] `.tt-block{border-radius:6px; margin:5px 3px var(--space-xs); padding:var(--space-sm) 10px; border:1px solid var(--color-border); cursor:pointer}` and is a flex column via `.tt-text`.
- [ ] A `data-now-line` element (`<div class="now-line" style="grid-row:2; display:none">`) exists once per panel; JS (`updateNowLine`, see "Now Line" below) shows/positions it. `.now-line{grid-column:2/-1; border-top:2px solid var(--color-accent); z-index:8}` with a `::before{content:'NOW'; position:absolute; left:-48px; top:-8px}` label in accent color.
- [ ] Tablet tweak at `max-width:768px`: `html,body{overscroll-behavior:none}`, floor pill font shrinks to `var(--font-xs)` with `padding:6px 2px`, and `.tt-block{font-size:var(--font-xs); padding:6px 7px; margin:2px; gap:5px}`. The grid layout (not the mobile table) is still in effect at this width: the table swap only happens at 480px (see below).
- [ ] The mobile `<table>` wrapper is hidden on desktop/tablet: `.tt-table-wrap{display:none}`.

## Mobile Layout (<=480px): HTML Table Replaces the Grid

- [ ] At `max-width:480px`: `.floor-header-bar{display:none!important}` and `.timetable{display:none!important}` (the desktop grid is fully removed from layout, not just visually hidden), while `.tt-table-wrap{display:block!important; min-height:300px}` takes over.
- [ ] The document itself stops scrolling: `.view-timetable{overflow:hidden; overflow:clip}` on the root element (clip, not just hidden, so even scripted scrolling of the viewport is blocked: a plain `overflow:hidden` on `body` alone would still let a stray 4px of overflow shift the page).
- [ ] `.view-timetable body{height:100vh; height:100dvh; overflow:hidden; overflow:clip; display:flex; flex-direction:column}`: the page is a fixed-height flex column tracking the dynamic viewport height (Safari's collapsing toolbar).
- [ ] The flex chain down to the scroller is: `.view-timetable main{flex:1 1 auto; min-height:0; display:flex; flex-direction:column}` -> `.view-timetable #timetable-view{flex:1 1 auto; min-height:0; display:flex; flex-direction:column}` -> `.view-timetable .timetable-panel.active{flex:1 1 auto; min-height:0; display:flex; flex-direction:column}` -> `.view-timetable .tt-table-wrap{flex:1 1 auto; min-height:0}`.
- [ ] Every other direct child along that chain is pinned non-shrinkable: `.view-timetable body > :not(main), .view-timetable main > :not(#timetable-view), .view-timetable #timetable-view > :not(.timetable-panel), .view-timetable .timetable-panel.active > :not(.tt-table-wrap) { flex: none; }` (so the cmd-bar/title/tabs chrome can never be compressed by the panel's large intrinsic content height).
- [ ] Because the document cannot scroll, all normally-sticky chrome is forced static: `.view-timetable .cmd-bar, .view-timetable #page-title, .view-timetable .filter-bar { position: static; }` (document-sticky offsets would otherwise push the tabs down once the collapsing title raises their flow position).
- [ ] `.view-timetable .filter-bar::after{display:none}`: the sticky scroll-fade strip (meant for the document-scrolling list view) is turned off here because its absolute positioning would otherwise escape the clipped root and reintroduce 4px of scrollable overflow.
- [ ] The ONLY scrolling surface is `.tt-v-scroll{height:100%; overflow:auto; scrollbar-width:none; -ms-overflow-style:none; overscroll-behavior:none}` (plus a `::-webkit-scrollbar{display:none}` rule) inside the active panel's `.tt-table-wrap`. It scrolls BOTH axes (vertical for time, horizontal for floor columns since the table can exceed viewport width).
- [ ] Both the floor-name `<thead>` row and the time-label column stick against THAT `.tt-v-scroll` scroller, not the document: `.tt-table thead th{position:sticky; top:0; z-index:2}`; `.tt-table thead th:first-child{position:sticky; left:0; z-index:3}` (the corner cell needs the higher z-index to sit above both the sticky header row and the sticky time column); `.tt-table tbody td.tt-time-td{position:sticky; left:0; z-index:1}`.
- [ ] `.tt-table{border-collapse:separate; border-spacing:0; table-layout:fixed; width:calc(40px + var(--num-floors) * 40vw)}`: each floor column is exactly `40vw` wide (via `<colgroup>` `<col style="width:40vw;min-width:40vw">`) regardless of floor count, and the time-label column is a fixed `40px` (`<col style="width:40px;min-width:40px">`). With 3+ floors the table is wider than the viewport, requiring horizontal scroll on `.tt-v-scroll`.
- [ ] The 12px page padding (body's own `padding: 0 var(--space-md)` at this width) is what gives the scroller its side inset: there is deliberately NO separate gutter/inset element (a prior 12px pinned gutter strip design was rejected and must not be reintroduced).
- [ ] Table rows use 5-minute granularity: `step = 5`; the table starts 5 minutes BEFORE `grid_start` (`table_start = grid_start - step`) purely so the first hour/half-hour label can be vertically centered via `rowspan="2"` straddling the boundary; `total_rows = (total_minutes + step) // step + 1`.
- [ ] Row height is the CSS variable `--row-h`, set as a single global rule `.tt-table { --row-h: {10 or 14}px; }` inserted once, site-wide (applies to every panel/date, not computed per-panel). The value is `14` if ANY slot in the ENTIRE timetable dataset (any date, any floor) has `duration_minutes < 30 * (number_of_artists_in_that_slot + 1)`; otherwise `10`. (E.g. a single-artist 45-minute slot needs `45 < 60` -> true -> 14px rows apply everywhere; a single 60+ minute slot alone would not trigger it.)
- [ ] `.tt-table tbody tr{height:var(--row-h)}`; the first and last `<tr>` are half-height (`calc(var(--row-h)/2)`) to match the 5-minutes-early start padding.
- [ ] Hour/half-hour gridlines on mobile are a repeating CSS background on `<tbody>` (`repeating-linear-gradient`, one pass per `--row-h * 12` for hour lines in `--color-line-hour`, one per `--row-h * 6` for half-hour lines in `--color-line-half`), not per-row DOM elements.
- [ ] Time label cells appear only on the row whose NEXT 5-minute boundary lands on `:00` (class `tt-line-hour`, text `HH:00`) or `:30` (class `tt-line-half`, text `HH:30`), each with `rowspan="2"`; all other rows get an empty `<td class="tt-time-td">`.
- [ ] Floor header `<th>` uses the same curator-note structure as desktop: `<th class="tt-floor-th floor-{fid}"><span>{name}</span><span class="floor-curator">{text}</span></th>` when a curator note exists, or without the second span when it does not (functionally identical rendered output to desktop, but the Python branches on presence/absence rather than always emitting an empty string).
- [ ] Artist blocks in the table are placed via `<td rowspan="{N}">{block html}</td>` at the correct `(floor column, 5-min row)` cell; cells covered by another block's rowspan, or by a time-label's rowspan, are skipped entirely (no filler `<td>`: the covering cell's rowspan handles it) and empty cells otherwise render as `<td></td>`.
- [ ] `.tt-table .tt-block{position:absolute; top:1.5px; left:1px; right:1px; bottom:2.5px}`: the block fills its host `<td>` with a ~1-2px inset.
- [ ] Mobile `.tt-popup` sizing: `width:calc(100vw - var(--space-xl)); max-width:none; left:var(--space-md)!important` (overrides the desktop 320px fixed-width popup so it never overflows the viewport width), and `.popup-photo`/`.popup-photo-placeholder` shrink to `64px` (from 80px on desktop).
- [ ] Mobile filter-bar shrinks: `padding:6px 0; margin:0 0 var(--space-xs); min-height:calc(1.5 * var(--font-lg) + 13px)`; its fade strip shrinks to `height:var(--space-xs)`.
- [ ] `.tt-photo`/`.tt-photo-placeholder` shrink to `30px` (from 34px) at this width.

## Scroll-Driven Title Compaction (mobile timetable only)

- [ ] Wrapped in `@supports (animation-timeline: scroll())`, itself nested inside the `max-width:480px` media query. This is the feature-detection mechanism: browsers without CSS scroll-driven animation support simply never match the `@supports` block and get NO animation at all, so the title stays statically sized, full-size, permanently (the "static-title fallback").
- [ ] `.view-timetable body{timeline-scope:--tt-vscroll}` exposes a custom named scroll timeline from `body` so a descendant OUTSIDE the scroller (the `<h1>`, which lives above `.tt-v-scroll` in the DOM) can reference it.
- [ ] `.view-timetable .timetable-panel.active .tt-v-scroll{scroll-timeline:--tt-vscroll block}`: only the ACTIVE panel's scroller declares the timeline (so switching panels never leaves two competing owners of the same named timeline).
- [ ] `.view-timetable #page-title{animation:tt-title-compact linear both; animation-timeline:--tt-vscroll; animation-range:0px 16px}` drives the title's compaction directly off that scroller's scroll offset, over its first 16 scrolled pixels, 1:1 (not time-based).
- [ ] `@keyframes tt-title-compact { to { margin-top: 0; } }`: the only animated property is `margin-top` (starting from the base `h1{margin-top:21px}` rule), so the title's text never disappears or resizes; only the whitespace above it collapses, then the title stays pinned at `margin-top:0` for the remainder of the scroll (`both` fill mode holds both the 0%- and 100%-of-range states outside the active range).
- [ ] This is purely CSS-driven (no JS scroll listener involved) and is a per-panel behavior: switching to a different panel via `switchDay`/`renderPeriodTabs` re-associates the timeline to the newly-active panel's `.tt-v-scroll`.

## Scroll Position Persistence

- [ ] View-level scroll (list vs. timetable): an in-memory JS object `_viewScrollPos = { list: 0, timetable: 0 }` (NOT persisted to `localStorage`/`sessionStorage`: lost on a full page reload, only preserved across in-session `switchView()` calls). `switchView()` records `window.scrollY` into `_viewScrollPos[currentView]` before switching, and restores via `window.scrollTo(0, _viewScrollPos[view] || 0)` after switching.
- [ ] Panel-level scroll (per date+period, mobile only): an in-memory JS map `_savedScrollTop` keyed by the string `"{date}|{period}"`, storing `{top, left}`. `showPanel(date, period)` saves the PREVIOUS active panel's scroll position into that map (reading `.tt-v-scroll.scrollTop/scrollLeft` when `matchMedia('(max-width:768px)')` matches, else `window.scrollY/scrollX`) before switching, then restores the NEW panel's saved position (or `{0,0}` if none recorded) on the next animation frame.
- [ ] A one-shot override, `_carryScroll`, takes priority over `_savedScrollTop` exactly once (then is cleared to `null`): set by `switchDay(date, btn)` to `{top:0, left:0}` when re-clicking the SAME day that is already current, or to the CURRENT scroller's live `{top, left}` when switching to a genuinely DIFFERENT day (i.e. changing day deliberately carries the current scroll depth forward into the new day's first panel, rather than resetting to the top).
- [ ] `_ttScroller(panel)` resolves the scroll target: it returns `null` (meaning "use `window` scroll") unless `matchMedia('(max-width:480px)').matches`, in which case it returns that panel's `.tt-v-scroll` element. This 480px threshold matches the CSS breakpoint that swaps in the mobile `<table>`.
- [ ] None of the panel-level or view-level scroll state is written to `localStorage`: only the ACTIVE VIEW itself (`stc_view`: `'list'` or `'timetable'`) persists across reloads; the scroll pixel offset within a view does not.

## Popup Behavior (grid and table share the same popup)

- [ ] A single popup element, `<div class="tt-popup" id="tt-popup" role="dialog" aria-label="Artist details">`, is shared across all panels/blocks; it contains `<div class="popup-meta" id="popup-meta">` and `<div id="popup-artists">`.
- [ ] `.tt-popup{position:fixed; visibility:hidden; opacity:0; pointer-events:none}`; `.tt-popup.open{visibility:visible; opacity:1; pointer-events:auto}`: no CSS transition is declared on those properties (instant show/hide).
- [ ] Clicking a `.tt-block` (anywhere on the card EXCEPT `.tt-photo-heart`, `.tt-cal`, or `.tt-ics`, which `stopPropagation`/handle their own click) triggers exactly one of two behaviors: (a) if the popup is currently open (from any previous block), the click just CLOSES the popup and does nothing else: it does NOT open the newly-clicked block's popup in that same click; (b) if the popup is currently closed, it opens for the clicked block via `openBlockPopup(block, clientX, clientY)`.
- [ ] `openBlockPopup` reads the block's artist array from the global `TT_ARTISTS[artistId]` lookup (JSON embedded once per page as `<script>var TT_ARTISTS={...}</script>`), sets `#popup-meta` text to `"{time} · {floor}"` (using the block's `data-time`/`data-floor` attributes), and renders one `.popup-artist` row per artist in the group (photo or `.popup-photo-placeholder`, name, and a social-links row using `PLATFORM_SVG` icons for `instagram/soundcloud/spotify/linktree/youtube/ra`).
- [ ] Each artist's photo (`<img class="popup-photo" ...>`) and name (`<div class="popup-name">`) carry `data-bioclick="1" data-oid="{id}" data-name="{name}"` attributes.
- [ ] Popup positioning: initially placed off-screen (`left:-9999px; top:-9999px`) to measure size, then placed at `(clientX+12, clientY+12)`, clamped so it never overflows the viewport (`left` clamped to `[8, viewportWidth - popupWidth - 8]`; if it would overflow the BOTTOM the popup flips to open ABOVE the cursor instead: `top = clientY - popupHeight - 12`, then clamped to a minimum of `8`).
- [ ] Popup close triggers: (1) clicking anywhere outside `.tt-popup` while it is open (guarded by a one-frame `_popupJustOpened` flag so the very click that opened it does not immediately close it); (2) any `scroll` event on `document` or `window` (both listeners registered, passive); (3) the `Escape` key, but ONLY when no `.modal-overlay.open` modal is present (so Escape closing a bio/sync/share modal takes priority and does not also close the popup).
- [ ] Bio hand-off: clicking an element with `[data-bioclick]` inside `#popup-artists` (the photo or the name) calls `e.stopPropagation()`, then `closePopup()`, then `openBioById(oid, name)`, i.e. the popup is fully closed BEFORE the bio modal (`#m-bio`, opened via `openDialog('m-bio')`) appears; there is no state where both are simultaneously visible. `openBioById` lazy-fetches `bios.json` once (cached in `_biosCache`), falling back to `{name, photo:'', bio:''}` (rendering "No biography available") if the id is missing or the fetch fails.
- [ ] The popup itself has NO heart button for any artist (hearts only exist on the underlying `.tt-photo-heart` inside the grid/table card, not inside the popup UI).
- [ ] `#popup-artists` uses one delegated click listener (not per-row listeners), so it works identically for popups opened from the desktop grid or the mobile table.

## B2B Sets (Multiple Artists, One Card)

- [ ] Artists are grouped into one card per distinct `(floor, start_time, end_time)` combination; if 2+ artists share that exact triple they render as ONE `.tt-block` (or one `<td>` in the mobile table) rather than one card each.
- [ ] The card's `data-name` attribute is the joined string `"{artist1} b2b {artist2} b2b ..."` (literal `" b2b "` separator, lowercase).
- [ ] The card's `data-artist-id` (used for the calendar/schedule toggle and ICS export) is the single `slot_uuid` computed over ALL artist ids in the group jointly, i.e. schedule state and ICS export are shared/per-CARD for a B2B set, not per-artist.
- [ ] Inside the card, EACH artist in the group gets its own `<div class="tt-artist-row" data-artist-id="{per-artist uuid}">` wrapping its own photo (`.tt-photo`/`.tt-photo-placeholder`) and its own `<button class="tt-photo-heart" data-artist-id inherited from the row>`, i.e. HEART state is tracked per-artist-row (each artist can be favorited independently) even though the surrounding card's schedule/ICS state is shared.
- [ ] This split works via DOM structure, not extra JS logic: `toggleHeart`'s `btn.closest('[data-artist-id]')` finds the nearest `.tt-artist-row` ancestor (per-artist id) because that div is closer than the outer `.tt-block`; `toggleSchedule`'s `btn.closest('[data-artist-id]')` (from the `.tt-cal` button, which lives in `.tt-time-row`, a sibling of the artist rows, not a descendant of any of them) resolves to the outer `.tt-block` (the shared slot id).
- [ ] Consequently the "Show My Picks" filter (`.filter-active .tt-block:not(.hearted):not(:has(.tt-artist-row.hearted))`) dims a B2B card only when NONE of its member artists are hearted (the block's own `.hearted` class is checked too, but in practice a block's own `data-artist-id`: the slot uuid: never matches an entry in `localPicks`, since hearts are only ever written against per-artist-row/per-list-item ids; the `:has(.tt-artist-row.hearted)` clause is what actually does the work for timetable cards).
- [ ] The "Show My Schedule" filter (`.filter-schedule .tt-block:not(.scheduled)`) operates purely at the card level (one shared state for the whole B2B group) since `.scheduled` is toggled on `.tt-block` itself, matching the schedule id being the shared slot id.
- [ ] The ICS "Add to calendar" text button (`.tt-ics`) and its `downloadICS()` handler likewise operate on the whole card (`block.dataset.artistId`, the shared slot id): one ICS file per B2B card, not per artist.

## Schedule (Calendar Icon) and ICS Export

- [ ] Every card has a calendar toggle button: `<button type="button" class="tt-cal" aria-label="Add to schedule" aria-pressed="false"><svg><use href="#i-cal"/></svg></button>`, positioned in the card's top-right (`.tt-time-row{display:flex; justify-content:space-between}`).
- [ ] Clicking `.tt-cal` is intercepted by a document-level delegated listener that calls `e.stopPropagation()` (so it never also triggers the card's own click-to-open-popup handler) and then `toggleSchedule(cal)`.
- [ ] `toggleSchedule` optimistically toggles the local `localSchedule` Set (keyed by the card's shared slot id), toggles `.active`/`aria-pressed` on the button and `.scheduled` on the card immediately, persists to `localStorage` (`stc_schedule`, `stc_synced_schedule`), THEN syncs to the server via `POST`/`DELETE /api/session/{sessionId}/schedule/{id}` (creating a session first via `ensureSession()` if none exists yet). A 404 response triggers a full session reset (clears `stc_session_id`/`stc_share_token`, re-creates a session). Any other non-OK/204 response rolls the optimistic UI change back.
- [ ] Per-id request chaining: `_scheduleSyncChain` (a `Map<id, Promise>`) serializes overlapping POST/DELETE calls for the SAME id so a rapid double-tap cannot race a POST against a DELETE with no ordering guarantee; the optimistic UI update itself is never delayed by this chaining.
- [ ] `.tt-cal.active svg{color:var(--color-schedule)}` (icon recolors) and `.tt-block.scheduled{box-shadow: inset 0 0 0 2px var(--color-schedule)}` (whole card gets an inset outline) reflect "on my schedule" state.
- [ ] `.filter-schedule .tt-block:not(.scheduled){opacity:0.15}` is the "Show My Schedule" dim-filter, toggled by the cmd-bar button `#btn-schedule` ("Show My Schedule", only shown/inline-block while in timetable view) via `toggleScheduleFilter()`, which flips the `filter-schedule` class on `<body>`. Switching back to the list view (`switchView('list', ...)`) forcibly resets `scheduleFilterActive = false` and removes both the body class and the button's `.active` class.
- [ ] ICS export: `<button type="button" class="tt-ics">Add to calendar</button>`, absolutely positioned bottom-left of the card (`.tt-ics{position:absolute; bottom:var(--space-xs); left:var(--space-sm)}`). A document-level delegated click listener calls `e.stopPropagation()` then `downloadICS(ics.closest('[data-ics-start]'))`.
- [ ] `downloadICS(block)` tracks the analytics event `ics-export` (with `{artist: block.dataset.name}`) then navigates the browser to `/ics/{block.dataset.artistId}` (a full page navigation via `window.location.href`, not a fetch+blob download): the shared slot id is passed, so ICS export for a B2B card exports the combined slot, not one artist.
- [ ] Each card also carries `data-ics-start` and `data-ics-end` attributes (the raw ISO start/end time strings) on the outer block element, alongside `data-artist-id`, `data-name`, `data-time` (formatted display string), and `data-floor` (floor display name): all of it present on both the desktop-grid rendering and the mobile-table rendering of the same slot.

## Floor Annotations (`stage_notes` "curated by" / "hosted by")

- [ ] Curator/host text is looked up per render call by the exact key `"{date}.{floor_id}"` (e.g. `"2026-07-11.koksofenbatterie"`) against the `stage_curators` dict passed into `render_output_html`, i.e. it is scoped to one specific calendar date AND one specific floor; the same floor can show a different (or no) note on a different day.
- [ ] Desktop floor header: when a note exists, `<span class="floor-curator">{text}</span>` is appended directly after the floor-name `<span>` inside `.floor-header`; when absent, no extra span is emitted at all (not an empty one).
- [ ] Mobile floor `<th>`: identical text and placement (`<span class="floor-curator">`), with the same presence/absence branching, keeping desktop and mobile visually consistent per date+floor.
- [ ] `.floor-curator{font-style:italic; font-weight:normal; font-size:var(--font-xs); color:var(--color-muted); margin-top:2px; white-space:nowrap; overflow:hidden; text-overflow:ellipsis}` (desktop); on the mobile `<th>` it additionally gets `display:block; padding:1px 0 2px; margin:0` via `.tt-floor-th .floor-curator`.
- [ ] The floor-curator text is HTML-escaped (`esc()`), so markdown/HTML in the source `stage_notes.note` value is not rendered, only literal text.

## Artist Schedule Notes and "Also" Cross-References

- [ ] IMPORTANT DIVERGENCE FROM LIST VIEW: the `_format_artist_schedule()` helper (which produces the `.artist-schedule` "floor, time" line and the `.artist-also` "Also X · Y · Z" cross-reference line described for the LIST view) is called ONLY from `render_artist_card()`, the list-view artist-card renderer. It is never called anywhere in the timetable-panel rendering code (neither the desktop-grid block builder nor the mobile-table block builder). A timetable `.tt-block`/table-cell therefore never renders an `.artist-also`-equivalent "Also playing..." note on the card face, regardless of how many other slots that artist has.
- [ ] What a timetable card DOES show per slot: the slot's own start-end time as `<span class="tt-time">{start}–{end}</span>` (baked directly from that slot's `start_time`/`end_time`, not derived from `_format_artist_schedule`), and floor identity implicitly via the card's column position (grid) or table column (mobile) and its `.floor-{fid}` background color class: the floor's display NAME is not printed on the card itself, only surfaced textually in the popup's `#popup-meta` line (`"{time} · {floor name}"`) and in the `data-floor` attribute.
- [ ] There is consequently no client-side or server-side code path that cross-references a timetable card to that artist's OTHER slots (same day or other days): that cross-referencing behavior is list-view-only. Any Next.js parity work that assumes "Also" notes appear on timetable cards (as a literal reading of `CLAUDE.md`'s Timetable View section might suggest) would be adding behavior that does not exist in `render.py`.
- [ ] For contrast, the list-view format (for reference, defined in `_format_artist_schedule`, `services/data/scraper/render.py` lines ~142-164): the CURRENT slot's label is `"{floor}, {time}"` (parts joined with `", "`, empty parts dropped); every OTHER slot for that artist becomes one entry in `"Also " + " · ".join(entries)`, where each entry is `"{floor}, {time}"` for a slot on the SAME date, or `"{Weekday, Month D}, {floor}, {time}"` (via `_format_short_date_abbr`) for a slot on a DIFFERENT date.

## Day Tabs, Period Tabs, Floor Pills, Stage Colors

- [ ] Day tabs are rendered server-side, one `<button class="day-tab">{abbreviated date, e.g. "Fri 11"}</button>` per date in `dates_seen` (via `_format_date_tab`, format `"%a %-d"`-equivalent i.e. `"Fri 11"`), inside `<div class="day-tabs" id="day-tabs">`. The FIRST day tab (index 0) is given class `active` at render time; there is no logic anywhere (server or client) that selects the tab matching today's real-world date: the default is always simply the first date present in the dataset.
- [ ] Clicking a day tab calls `switchDay(date, btn)`: tracks analytics `day-switch`, computes `_carryScroll` (see Scroll Position section), sets `currentDate`, moves the `active` class to the clicked tab, and calls `renderPeriodTabs(date)`.
- [ ] Period tabs are NOT server-rendered (the container `<div class="period-tabs" id="period-tabs">` starts empty): they are built entirely client-side by `renderPeriodTabs(date)` from the `TT_SECTIONS` JS array (`[{date, period, key}, ...]`, embedded once as `const TT_SECTIONS = [...]`).
- [ ] If a date has 1 (or 0) periods, `renderPeriodTabs` leaves the period-tabs container empty (no buttons rendered) and just calls `showPanel(date, period)` directly with that single period (or the literal fallback string `'day'` if none exist).
- [ ] If a date has 2+ periods, one `<button class="period-tab">` is created per period; the VISIBLE LABEL TEXT is `p === 'day' ? 'Day' : 'Night'`: a strict two-way ternary on the literal string `'day'`, meaning any period value that is not exactly the string `"day"` (e.g. a hypothetical `"afterhours"` period) would still display the button text "Night". The currently-active period is kept across a day switch when the new day also has that period, otherwise it falls back to the new day's first period in list order.
- [ ] `showPanel(date, period)` is the actual panel-switch function: it saves the OUTGOING active panel's scroll position (see Scroll Position section), removes `.active` from all `.timetable-panel` elements, adds `.active` to `#panel-{date}-{period}`, restores scroll on the next animation frame, calls `truncateNames()`, and calls `updateNowLine()`.
- [ ] Floor pills get their color from the `stage_colors` dict (sourced from `event_stages.color`, an "R, G, B" string) passed into `render_output_html`. Two dynamically-generated CSS rules per known floor id: `.floor-{fid} { background: rgba({rgb}, 0.88); }` (applied to the `.tt-block` card body) and `.floor-header.floor-{fid} > span:first-child { background: rgb({rgb}); }` / `.tt-floor-th.floor-{fid} > span:first-child { background: rgb({rgb}); }` (applied to the desktop and mobile floor-name pill respectively), i.e. the CARD uses a 0.88-alpha rgba of the stage color, the PILL uses the fully opaque rgb.
- [ ] Unknown/missing floor (no `location_id`, internally bucketed under the id `"unknown"`) falls back to `.floor-unknown { background: rgba(243, 244, 246, 0.88); }` (light gray) for the card body only; because no floor id `"unknown"` normally exists in `stage_colors`, there is no dynamically-generated pill-color rule for it, so its floor-name pill renders with no explicit background (transparent/inherited) unless the DB happens to actually define a stage color under that literal id.
- [ ] The dynamic floor-color CSS block is inserted once per page load (a single `<style>` block appended after the built-in timetable CSS), keyed only by floor id: colors are NOT re-derived per date/period, so a floor keeps the same color across every day/period panel it appears in.

## Now Line

- [ ] `updateNowLine()` runs once at load, once whenever the active panel changes (`showPanel`, `switchView` into timetable), and on a `setInterval(updateNowLine, 60000)` (every 60 seconds) thereafter.
- [ ] It hides EVERY `[data-now-line]` element first (`display:none`), then computes whether "now" falls inside the CURRENTLY ACTIVE panel only (it does not update or reveal now-lines on inactive/hidden panels).
- [ ] Match logic: for a `day`-period panel, "now" matches only if the panel's `data-date` equals today's real calendar date (`YYYY-MM-DD`, locale-independent manual formatting) AND the current minute-of-day is within `[grid_start, grid_end]`. For a `night`-period panel (`data-is-night="1"`), it additionally matches if the panel's date equals YESTERDAY's date and the current minute-of-day is before 12:00 noon (in which case 1440 minutes is added to the "now" minute value to align with the night grid's post-midnight numbering).
- [ ] On a match, the line's `display` is cleared and its `grid-row` is set to `(nowMin - gridStart) + 2` (same `+2` header-row offset convention as artist blocks).
- [ ] There is NO auto-scroll to bring the now-line into view anywhere in the code: the line renders/repositions automatically, but the viewport/scroller position is never programmatically adjusted to it. (Per the task's instruction not to invent behavior: no "jump to now" or "auto-scroll to current time" exists.)
- [ ] The now-line is rendered identically for both the desktop grid (`.timetable [data-now-line]`) and would only visually appear there since the mobile table's own copy of the panel markup does NOT include a `[data-now-line]` element at all: the mobile `<table>` structure has no now-line element in the HTML the code emits (only the desktop `.timetable` div gets one `<div class="now-line" data-now-line>` per panel).

- [ ] The hamburger button (`.hamburger`, inside `#cmd-bar`) only opens the dropdown (`toggleMenu()`) when `matchMedia('(max-width:768px)').matches`: the click handler on `#cmd-bar` checks this and no-ops otherwise (desktop clicks on the cmd-bar do nothing).
- [ ] Crossing the 768px breakpoint (a `matchMedia('(max-width:768px)')` `change` event) force-closes the dropdown (`closeMenu()`), so a dropdown left open cannot survive a resize into desktop layout.
- [ ] Dropdown rows relevant to the timetable: `#dd-list` ("Line-up") and `#dd-timetable` ("Timetable"); clicking either calls `switchView(...)` then `closeMenu()`. `#dd-schedule` ("Show My Schedule") exists only when `has_timetable` is true, starts with inline `style="display:none"`, and its visibility is kept in sync with the cmd-bar's `#btn-schedule` button (`ddSched.style.display = btnSched.style.display`) by `syncDropdownState()`, i.e. it is only visible in the dropdown while the timetable view is active, mirroring the cmd-bar button's own visibility rule.
- [ ] The `#dd-view-options` container (which wraps `#dd-filter` "Show My Picks" and, when present, `#dd-schedule`) is physically RELOCATED in the DOM by `syncDropdownState()` to sit directly after whichever of `#dd-list`/`#dd-timetable` is currently active (`ddCur.after(ddOpts)`): so the "Show My Picks"/"Show My Schedule" toggle rows visually move to directly under the Line-up row or the Timetable row depending on which view is active.

## Name Truncation

- [ ] `truncateNames()` runs on every `.tt-name` element (the artist name span inside a `.tt-artist-row`): if the element's own `clientWidth` is 0 (not currently laid out/visible) it is skipped entirely; otherwise it restores the FULL original text (cached in `dataset.full` on first run) and, if `scrollWidth > clientWidth`, binary-searches the longest prefix (via repeated `slice` + `…` ellipsis) that fits, then sets that truncated text.
- [ ] `truncateNames()` is called: once at load, inside `showPanel()`'s `requestAnimationFrame` callback (i.e. every day/period switch), inside `switchView('timetable', ...)`'s `requestAnimationFrame`, and via a `ResizeObserver` on `document.body` (debounced with `cancelAnimationFrame`/`requestAnimationFrame`) so a window resize or dynamic-viewport change re-measures every name.
- [ ] Because it operates on live `clientWidth`/`scrollWidth`, this behavior is inherently layout-dependent (grid vs. table, floor-column width, `--row-h`) and must be reproduced pixel-for-pixel (same ellipsis character, same binary-search truncation point) for a faithful port, not merely approximated with CSS `text-overflow:ellipsis` (the CSS ellipsis IS also present as a fallback via `.tt-block .tt-name{overflow:hidden; text-overflow:ellipsis; white-space:nowrap}`, but the JS actively rewrites `textContent`, so the two mechanisms run together).

## Filters Shared With List View But Timetable-Relevant

- [ ] "Show My Picks" (`#btn-filter`, body class `filter-active`) affects the timetable via `.filter-active .tt-block:not(.hearted):not(:has(.tt-artist-row.hearted)) { opacity: 0.15 }` (see B2B section for exactly how `.hearted` vs `:has(...)` interact); it does NOT hide timetable cards (only dims them), unlike the list view where `.filter-active .artist-item:not(.hearted) { display: none }` fully hides non-picked artists.
- [ ] "Show My Schedule" (`#btn-schedule`, body class `filter-schedule`) only exists/is togglable while in timetable view; it dims (does not hide) non-scheduled cards via `.filter-schedule .tt-block:not(.scheduled) { opacity: 0.15 }`. It has no equivalent effect on the list view (the list view has no `.scheduled` concept).
- [ ] Both filter states are plain `<body>` classes, so toggling them while in one view leaves them applied (inert, since the other view's container is `display:none`) if the user switches views, EXCEPT `switchView` explicitly force-resets `scheduleFilterActive` to `false` and removes `filter-schedule`/the button's `.active` class whenever switching TO the list view (there is no equivalent auto-reset for `filter-active`/"Show My Picks", which persists across a view switch).
