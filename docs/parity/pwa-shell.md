# PWA shell behavioral parity spec

Scope: everything the Next.js-served front end must reproduce exactly so that every
already-installed PWA and every already-stored push subscription survives the cutover
with zero user-visible change. This is the highest-risk surface in the whole migration:
get the service worker, the manifest, or the push wiring wrong and every push subscriber
is silently orphaned (no error, no exception, notifications just stop arriving).

Source of truth for every item below: `server/static/sw.js`, `server/static/manifest.json`,
`server/static/shared.js`, `server/api.py`, `server/chat/chat.html`,
`pipeline/scraper/render.py`, and the "Push Notifications" / "Page Load Flash Prevention"
sections of `CLAUDE.md` at the repo root. Line references are to the current (pre-port)
files, so the reviewer can diff the new implementation against them directly.

Convention: every checkbox is meant to be verifiable by inspection or by running the
named test. None of them are aspirational; they describe what the current code already
does.

---

## 1. Serving contract

- [ ] `GET /sw.js` is served from the root of the origin (`/sw.js`, not `/some/path/sw.js`), because a service worker's default max scope is the directory it is served from. Current: `server/api.py` `serve_sw()`.
- [ ] `GET /sw.js` response carries header `Service-Worker-Allowed: /` (belt-and-suspenders alongside root-scope serving). Current: `server/api.py` line ~1436.
- [ ] `GET /sw.js` response carries `Cache-Control: no-cache` (never `no-store`, never a long max-age) so browsers still revalidate on every page load and pick up a new SW version promptly, but a conditional GET can still 304. Current: `server/api.py` line ~1436, media type `application/javascript`.
- [ ] `GET /manifest.json` is served at `/manifest.json` with `Cache-Control: no-cache` and media type `application/manifest+json`. Current: `server/api.py` `serve_manifest()`.
- [ ] `GET /shared.css` and `GET /shared.js` are served with `Cache-Control: no-cache` (not `no-store`); a content deploy that ships new shared bundles must be picked up without a hard refresh. The SPA catch-all `index.html` intentionally uses `no-store` instead; do not conflate the two.
- [ ] There is exactly ONE manifest for the whole app (`manifest.json`), referenced by `<link rel="manifest">` on both the lineup/timetable page and the chat page, with `start_url: "/"` and `scope: "/"`.
- [ ] **Do not re-introduce a second, chat-specific manifest.** This was tried (commit `62e882b`, "dedicated PWA manifest and home-screen icon for chat") and reverted (commit `a86d5aa`, "single-app PWA resume ... Replace the two-manifest approach"). Reason: a second manifest is a second web-app identity in iOS's eyes, which means a second Home Screen icon, a second storage partition (its own cookies/localStorage), and therefore its own independent sign-in state and its own independent push subscription; splitting the "one browser origin, one push subscription" invariant that the whole shared-subscription design (section 4) depends on.
- [ ] `manifest.json` icon set is unchanged: `favicon.svg` (`sizes: any`, `image/svg+xml`) and `favicon.png` (512x512, `purpose: any maskable`). `background_color: #ffffff`, `theme_color: #111111`, `display: standalone`.
- [ ] Every page (`lineup.html`, `chat.html`, `transport.html`, any future page) sets `<meta name="mobile-web-app-capable" content="yes">` and `<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">`; dropping these on any page silently changes that page's standalone chrome on iOS.

## 2. Head-script contract (must run before first paint, in this order)

Both the lineup/timetable page and the chat page load-flash-prevent via `body { opacity: 0 }` declared in an inline `<style>` that is the FIRST element inside `<head>` (before any external stylesheet or script can execute), so nothing ever flashes unstyled before JS finishes initializing. Current: `pipeline/scraper/render.py` lines ~272-289 (lineup), `server/chat/chat.html` line 4 (chat).

- [ ] Lineup/timetable reveal mechanism: `Promise.race([initPromise, 5s-timeout])` flips `document.body.style.opacity = '1'` when the async init IIFE settles (success OR failure) OR after a hardcoded 5000 ms failsafe; a hung `fetch()` must never leave the page permanently blank. Current: `pipeline/scraper/render.py` lines ~2970-2978.
- [ ] The deep-link action runner polls this same opacity flag as its "init done" gate, with its own give-up timeout of 6000 ms; deliberately longer than the 5000 ms reveal timeout so it never gives up before the failsafe reveal fires. Current: `pipeline/scraper/render.py` line ~1738 (`document.body.style.opacity === '1' || ++tries > 60`).
- [ ] Head script 1 (lineup/timetable only), **standalone-launch resume redirect**: runs only when `location.pathname === '/'` AND the app is running standalone (`navigator.standalone` truthy OR `matchMedia('(display-mode: standalone)').matches`) AND `localStorage.last_section === 'chat'`; in that case `location.replace('/chat')` and returns immediately. In every other case (normal browser tab, or standalone but not at `/`, or `last_section` not `chat`), it falls through and sets `localStorage.last_section = 'lineup'`; i.e. any explicit lineup/timetable page load reclaims the flag for next launch. **A normal browser tab hitting `/` must NEVER redirect to `/chat`**; the standalone gate is load-bearing. Current: `pipeline/scraper/render.py` lines ~248-256.
- [ ] Head script 1's counterpart on the chat page unconditionally sets `localStorage.last_section = 'chat'` on every load (no standalone gate needed there; visiting chat always means chat was the last section). Current: `server/chat/chat.html` line 6.
- [ ] Head script 2 (lineup/timetable only), **view-class + canonical-URL rewrite**: computes the view (`timetable` or `list`) from, in priority order, the current pathname (`/timetable` or `/line-up`) then `localStorage.stc_view` (defaulting to `list`); stamps `localStorage.stc_view` with that value (semantics: "last view SEEN", not "last toggled"; a bare `/line-up` or `/timetable` visit updates it too); sets `document.documentElement.className = 'view-' + v`; and if the current pathname doesn't match the canonical path for that view, rewrites it via `history.replaceState` to `/line-up` or `/timetable` (never a full navigation). This must run and complete before body content is parsed, because CSS rules keyed to `.view-list` / `.view-timetable` control which view/buttons/menu-items are visible. Current: `pipeline/scraper/render.py` lines ~257-271.
- [ ] Head script 3 (both pages), **`pwa-standalone` class**: `if (navigator.standalone || matchMedia('(display-mode: standalone)').matches) document.documentElement.classList.add('pwa-standalone')`, run before any layout so standalone-only CSS (safe-area bottom padding, see section 7) applies from first paint. Current: `server/chat/chat.html` line 5; lineup applies the equivalent check inline in its bell/notification logic (`_isStandalone` at `pipeline/scraper/render.py` line ~2411) but the class-stamping pattern is chat's; when porting, both pages must stamp the class in a head script, not compute it late in a body script.
- [ ] `history.scrollRestoration = 'manual'` is set as an early inline script on chat (`if ('scrollRestoration' in history) history.scrollRestoration = 'manual'`); must be preserved so the SPA controls its own scroll position instead of the browser.

## 3. Service worker: `push` event

File: `server/static/sw.js`. `SW_VERSION = 'v10'`; bump this on any behavioral change to the SW so `tests/test_notifications.py::test_sw_version_v10`-style assertions and diagnostic logging stay meaningful; carry the version-bump discipline into the ported SW.

- [ ] `install` calls `self.skipWaiting()` unconditionally (no waiting for old clients to close) and `activate` calls `event.waitUntil(self.clients.claim())`; a new SW takes over immediately, matching the existing deploy model (content deploys are frequent, no user-visible "update available" prompt exists).
- [ ] Payload is parsed defensively: `event.data.text()` then `JSON.parse`, falling back to `{}` on parse failure (never throws, never lets the whole handler reject).
- [ ] Notification title defaults to `'Stone Techno Companion'` when `data.title` is absent; body defaults to `''`.
- [ ] **Tag uniqueness, `push_id` preferred over `push_index`**: `tag = 'stc-' + (data.room_id || '') + '-' + (data.push_id || data.push_index || Math.random().toString(36).slice(2))`. Why this order matters: `push_index` is an in-process counter that resets to 0 on every server restart, so after a restart a new push can collide with the tag of a notification still sitting in iOS Notification Center from before the restart. `push_id` is `secrets.token_hex(8)` minted fresh per push server-side (both `chat_ws._do_send_push` and the lineup scheduler in `api.py`), so it can never collide across a restart. The `Math.random()` fallback exists only for a payload that omits both (should not happen in practice, but the handler must not throw on it).
- [ ] **Why tag uniqueness is critical**: iOS silently drops `notificationclick` for any notification that *replaced* an earlier one (same tag); the tap opens the app at `start_url` with no event and no error, which was the diagnosed root cause of "notification click lands on line-up" (room-tag reuse before `push_id` existed). This is a platform quirk with no workaround other than never reusing a tag for two different pushes the user might still see.
- [ ] Notification options: `icon: '/favicon.png'`, `badge: '/favicon.png'`, `data: { url, roomId, count }`, `silent: data.silent || false`.
- [ ] Before showing the new notification, `self.registration.getNotifications()` is queried and any existing notification whose `data.roomId === data.room_id` is closed; collapses multiple pending pushes for the same room into one visible notification (the debounce/silent-escalation logic lives server-side; the SW's job is just not to stack redundant native notifications for the same room).
- [ ] After `showNotification` resolves: if `data.total_unread` is present and `navigator.setAppBadge` exists, call `setAppBadge(data.total_unread)`.
- [ ] Logging/ack, LAST: `Promise.all([swlog('push-received', data.url), ackPush('delivered', data.url)])`; network calls happen only after the local notification work is done, never before it (see section 4's "local-work-first" principle, which applies here too even though `push` has no client-navigation race).
- [ ] `swlog` POSTs to `/chat/api/swlog` with `{src: 'sw', v: SW_VERSION, step, detail}`, swallowing all errors (`.catch(function(){})`); a diagnostics call must never break the push flow.
- [ ] `ackPush(action, url)` first calls `self.registration.pushManager.getSubscription()`; if no subscription, no-ops; otherwise POSTs to `/chat/api/push/ack` with `{endpoint, action, v, url}`, again swallowing errors.

## 4. Service worker: `notificationclick`

- [ ] `event.preventDefault()` and `event.notification.close()` run first, synchronously.
- [ ] Target URL resolution: `event.notification.data.url` or `'/'`, resolved to an absolute URL via `new URL(targetUrl, self.location.origin).href`.
- [ ] **Local-work-first ordering, exact sequence** (iOS may kill the SW immediately after the app foregrounds, so anything after the local-storage write risks never running):
  1. `caches.open('stc-push')` then `cache.put('/_push_navigate', new Response(targetUrl))`; write the RELATIVE target URL (not the absolute one) to Cache Storage. Wrapped in `.catch(() => {})` so a Cache Storage failure doesn't abort the click handler.
  2. `self.clients.matchAll({ type: 'window', includeUncontrolled: true })`.
  3. If any window client exists: `postMessage({ type: 'navigate', url: fullUrl })` to the FIRST client, then `try { client.focus() } catch {}`. **Never combine `client.navigate()` with `postMessage` in the same click**; two racing navigations abort each other (this was tried and reverted, commit `f08b388` "Revert client.navigate() in SW; breaks iOS cache fallback"). The page itself performs the actual `window.location.href` assignment on receipt of the `navigate` message (or via the Cache Storage poll fallback, section 5); the SW only signals intent.
  4. If NO window client exists: `self.clients.openWindow(fullUrl)`. `openWindow()` silently returns `null` when a window already exists, which is exactly why step 3 is tried first; calling `openWindow` unconditionally would sometimes silently no-op instead of focusing the existing tab.
  5. Acks/logging LAST, after the above resolves: `Promise.all([swlog('click-done', result), ackPush('clicked', targetUrl)])`.
- [ ] Push URL for chat messages includes the message id (`/chat/msg/{id}`) so the client can scroll-to-message on click, not just open the room.

## 5. Service worker: `notificationclose` and `pushsubscriptionchange`

- [ ] `notificationclose` → `event.waitUntil(ackPush('dismissed'))`. No other side effects.
- [ ] `pushsubscriptionchange`: no-ops if `!event.oldSubscription`. Otherwise re-subscribes with `self.registration.pushManager.subscribe(event.oldSubscription.options)` (reusing the SAME options the dead subscription had, i.e. the same `applicationServerKey`), then POSTs the new endpoint + keys to `/chat/api/push/subscribe` (chat's endpoint, user-scoped). Wrapped in `.catch(() => {})`.
- [ ] **Known limitation, must be preserved as-is, not silently "fixed" during the port**: the SW handler re-registers ONLY the chat-side record (`chat_push_subscriptions`, user-scoped), because a service worker cannot read the lineup session id out of `localStorage` (SW has no DOM/localStorage access). The lineup side has no equivalent SW-driven repair; it self-heals its own record on its NEXT page load instead (section 6, lineup re-sync on load). Do not add a lineup POST here without also solving how the SW would obtain a session id; that's a bigger design change, out of scope for parity.

## 6. Service worker: scoped cache-first `fetch` handler (offline map cache)

- [ ] Cache name is exactly `stc-map-v1` (`MAP_CACHE`).
- [ ] The handler only engages for `GET` requests (`if (req.method !== 'GET') return;`; no `respondWith`, request passes through untouched).
- [ ] URL parse failures return early untouched (`try { url = new URL(req.url) } catch (e) { return; }`).
- [ ] The **exact** match set for "is this a map asset":
  - [ ] `url.pathname` starts with `/vendor/maplibre/` (the self-hosted MapLibre GL library).
  - [ ] `url.pathname` starts with `/dop/` (the self-hosted NRW aerial tile proxy, `server/api.py` `dop_tile()`).
  - [ ] `url.hostname === 'tiles.openfreemap.org'` (streets/labels vector tiles, exact hostname match, not a prefix/suffix check).
- [ ] Anything that does not match any of the three above returns early with **no `respondWith` call**; every other request (API calls, HTML, other static assets) passes through to the network completely unmodified, exactly as if this handler didn't exist. This must never be broadened to a blanket cache-first strategy for the whole origin.
- [ ] For a matching request: cache-first, no revalidation; `cache.match(req)`, return the hit if present; otherwise `fetch(req)`, and if the response is `ok` or `type === 'opaque'` (cross-origin no-cors responses, relevant for the OpenFreeMap host), `cache.put(req, res.clone())` (best-effort, `.catch(() => {})`), then return the response. Tiles are treated as effectively immutable; there is no cache-busting or TTL logic.

## 7. Client-page push contract (both surfaces)

The core invariant: **lineup and chat share exactly ONE browser push subscription**, because both register `/sw.js` at the same root scope on the same origin, and a browser holds exactly one `PushSubscription` per (origin, scope). The lineup record lives in `push_subscriptions` (hearts.db, keyed by `session_id`, endpoints under `/api/session/{code}/push/subscribe`); the chat record lives in `chat_push_subscriptions` (chat.db, keyed by `user_id`, endpoints under `/chat/api/push/*`); but both rows must reference the SAME `endpoint`.

- [ ] **Enable flows never unsubscribe before subscribing.** Both `_subscribePush()` (chat, shared by `_enableAllNotifications` and `_repairPushSubscription`) and `enableNotifications()` (lineup) call `reg.pushManager.getSubscription()` first and only call `subscribe(...)` if none exists; reusing an existing subscription rather than rotating it. Rotating the endpoint on one surface silently orphans the OTHER surface's stored record (its next push → 410 → auto-pruned → that surface silently stops receiving push with no error surfaced to the user).
- [ ] `_subscribePush()` (chat) reports success (`return true`) ONLY when BOTH the browser `subscribe()`/`getSubscription()` call succeeded AND the follow-up `POST /chat/api/push/subscribe` returned `ok`. A non-ok POST must be treated as a failed enable, not silently ignored. Current: `server/chat/chat.html` lines ~4595-4626.
- [ ] `enableNotifications()` (lineup) has the equivalent contract: it does not set `localStorage.stc_push = '1'` unless the subscribe POST (`/api/session/{code}/push/subscribe`) returned `ok`. Current: `pipeline/scraper/render.py` lines ~2425-2469.
- [ ] **Disable flows delete only the caller's own server-side record**, and only call the browser's `sub.unsubscribe()` when the OTHER surface is ALSO disabled, checked via the shared-origin localStorage flags:
  - [ ] Chat's `_disableAllNotifications()` always does `DELETE` against `/chat/api/push/subscribe`; it calls `sub.unsubscribe()` only if `localStorage.stc_push !== '1'` (lineup's flag); i.e. if lineup is still enabled, chat disable leaves the shared browser subscription alone and just removes its own DB row.
  - [ ] Lineup's `disableNotifications()` always does `DELETE` against `/api/session/{code}/push/subscribe`; it calls `sub.unsubscribe()` only if `localStorage.push_enabled !== '1'` (chat's flag); same symmetry, reversed.
  - [ ] This symmetry means disabling one surface can NEVER silently break the other surface's push.
- [ ] **Repair-on-load is gated by the surface's own enabled flag, never overriding an explicit disable**:
  - [ ] Chat's `_repairPushSubscription()` returns immediately unless `localStorage.push_enabled === '1'` AND `Notification.permission === 'granted'`. If there's no active subscription, it calls `_subscribePush()` to recreate one; if a subscription DOES already exist, it just resyncs the endpoint to the chat backend via `POST /chat/api/push/subscribe` (since lineup's enable flow shares the same subscription and may have rotated/created it without chat's backend ever finding out). Current: `server/chat/chat.html` lines ~4628-4667.
  - [ ] Lineup's equivalent re-sync-on-load (inside the page's init IIFE) is gated on `localStorage.stc_push === '1'`; if a subscription exists it POSTs to `/api/session/{code}/push/subscribe`; if `serviceWorker` supports push but NO subscription exists, it clears `localStorage.stc_push` and updates the bell UI (self-heals a purged/rotated subscription rather than leaving stale "enabled" UI state). Current: `pipeline/scraper/render.py` lines ~2899-2911.
- [ ] Brave-specific UX: if `subscribe()` throws `AbortError` and `navigator.brave` is present, lineup opens a dedicated dialog instructing the user to enable "Use Google services for push messaging" in `brave://settings/privacy`; chat surfaces the same guidance via a toast on subscribe failure. Preserve this messaging; Brave silently killing push (via "Forget me when I close this site" or a revoked site permission) unsubscribes at the FCM level with no exception the app sees until the next push 410s.
- [ ] iOS-specific gating on lineup: `enableNotifications()` short-circuits to an "add to home screen" dialog when `_isIOS && !_isStandalone` (push requires standalone on iOS Safari), and to a Brave-on-iOS-specific "switch browser" dialog when `_isIOS && navigator.brave`.

## 8. Push click navigation (page-side listeners)

- [ ] Both pages register `navigator.serviceWorker.register('/sw.js')` and add a `message` listener: on `{ type: 'navigate', url }`, `window.location.href = url` (a real navigation, not a SPA route change; simplest and most reliable given the SW can't know the SPA's internal router state).
- [ ] Both pages independently poll Cache Storage as a FALLBACK to the postMessage path (needed because the message event can be missed if the page wasn't fully loaded/listening at the moment the SW posted): `caches.open('stc-push')` → `match('/_push_navigate')` → if present, read the text, `cache.delete('/_push_navigate')`, and if the stored URL differs from `location.pathname`, navigate to it.
- [ ] The poll runs on `window.addEventListener('focus', ...)` and `window.addEventListener('pageshow', ...)`, each firing the check at three retry delays: **0 ms (immediate), 300 ms, and 1000 ms**; not a single check, because the cache write can race the focus event.
- [ ] Lineup additionally polls on `document.visibilitychange` when the tab becomes visible (`!document.hidden`); chat wires the same visibility check directly into its broader visibility handler.
- [ ] **Navigation latch**: a `_pushNavigating` boolean is set `true` for exactly 3000 ms after a navigation is triggered from the cache-poll path, to prevent a second retry firing mid-navigation from double-triggering `window.location.href`. This is a TIMEOUT, not a permanent latch; if the navigation is aborted (e.g. user backs out) the latch self-clears after 3 s and a later retry can still fire.
- [ ] Lineup's cache-poll additionally guards against self-navigation loops: it skips navigating if the stored URL `.includes('line-up')` (i.e. a push that targets the page you're already effectively on doesn't force a reload). Chat's does not carry this extra guard (its equivalent guard is `url !== location.pathname`); preserve each page's own guard as-is rather than trying to unify them, since chat has room-scoped URLs where the simple pathname-equality check is what matters.
- [ ] Chat message push URLs are `/chat/msg/{id}` specifically so the click lands on a message permalink that resolves to the room AND scrolls to/highlights that message, not just the room's default view.

## 9. Server-side contract the shell depends on (stays in Python, pinned here so the new front end's expectations match reality)

- [ ] **`dict(vapid_claims)` per webpush call, never a shared claims dict across a subscription loop.** `pywebpush` mutates the claims dict it's given, stamping the FIRST endpoint's origin into `aud`. A shared dict across a loop over multiple endpoints poisons every later push to a DIFFERENT push service: FCM (all Chromium browsers) strictly validates that `aud` matches its own origin and returns 403 for a mismatched `aud` (e.g. an Apple `aud` leaking into a Chrome push), while Apple and Mozilla do not enforce this binding; meaning Safari/Firefox subscribers keep working and mask the breakage for Chrome/Brave subscribers. This is a real incident (July 2026, cost a full afternoon to diagnose) and both call sites already guard it: `chat_ws._do_send_push` and the lineup scheduler in `server/api.py` (line ~400, `vapid_claims=dict(vapid_claims)`).
- [ ] **`ttl=300` on every `webpush()` call**, both lineup scheduler and chat sender. `pywebpush` defaults to `ttl=0`, which means "deliver right now or discard"; a momentarily unreachable client (locked phone, brief network blip) loses the push entirely at `ttl=0`.
- [ ] **VAPID key pair consistency checked at server startup**, via `_check_vapid_key_consistency()` in `server/api.py`, called from the `lifespan` context manager. It derives the public key from `VAPID_PRIVATE_KEY` and compares it (base64url, stripped of padding) against `VAPID_PUBLIC_KEY`. Logs exactly one of: `"VAPID key pair verified: private key matches public key"` (info) or a loud mismatch error naming both derived and configured key prefixes (error). This log line must be checked after ANY change to VAPID keys or `.env`, including a front-end migration that might touch env plumbing.
- [ ] **410 (and 404) pruning**: on `WebPushException` with `e.response.status_code in (404, 410)`, the lineup scheduler deletes the row from `push_subscriptions` by endpoint and commits, immediately, per-endpoint (not batched at the end of the loop); a raw network exception (timeout, ConnectionError, not wrapped in `WebPushException`) is caught separately and logged without deleting anything, so a transient network blip never gets misclassified as a dead endpoint.
- [ ] **Pending-moderation exclusion**: messages with `moderation_status = 'pending'` are excluded from `get_unread_counts` and from the push-preview fallback query; an unmoderated (or soon-to-be-rejected) message must never appear in a push notification body or inflate a badge count.
- [ ] **Idle detection, two layers**:
  - [ ] Instant/primary: the client `sendBeacon`s `POST /chat/api/push/idle` on `visibilitychange` going hidden AND on `pagehide`. This sets the user's last-activity marker to 0 server-side, making them immediately push-eligible. Verified on iOS to fire on lock, home-swipe, and force-close.
  - [ ] 30-second fallback/safety-net: if the beacon never arrives, the server treats a user as idle once 30 seconds pass with no user-initiated WS event (`send_message`, `typing`, `add_reaction`, etc.; passive events like `join_room`/`mark_read` do NOT reset this timer).
- [ ] **Cross-device badge sync**: `mark_read` broadcasts `badge_update` with `count: 0` to ALL of a user's active connections (not just the one that called mark_read); reading a room on the phone must clear the badge on desktop in the same session.
- [ ] `/chat/api/swlog` and `/chat/api/push/ack` are unauthenticated (by necessity; the SW has no session context) and therefore per-IP rate-limited (swlog 30/min, ack 60/min) to prevent log-flood abuse. The new front end's serving layer (whatever proxies these through to the Python backend) must not strip or bypass this rate limiting.

## 10. Standalone-mode specifics (iOS PWA chrome)

- [ ] **Keyboard handling via `visualViewport`, not automatic repositioning**: iOS does not auto-reposition input bars when the on-screen keyboard opens in standalone mode, so `chat.html` listens to `window.visualViewport`'s `resize` (and `scroll`, re-centering via `window.scrollTo(0,0)`) events. Logic (`server/chat/chat.html` lines ~5537-5566): track a running `_baseH` (largest viewport height seen); if the current height drops below `75%` of `_baseH` (keyboard presumed open), set `#app`'s height directly to the current visual-viewport height in px; otherwise clear the inline height (back to CSS-driven layout). Debounced via a double `requestAnimationFrame`. On `focusout`, after a 100 ms settle delay, if focus landed on nothing/`document.body`, clear `#app`'s inline height and nudge-scroll (`scrollBy(0,-1); scrollBy(0,1)`) to force iOS to repaint correctly.
- [ ] **No `viewport-fit=cover` in the viewport meta tag.** This was deliberately left out because it produced inconsistent `env(safe-area-inset-*)` values across iOS versions/orientations.
- [ ] **Safe-area bottom padding is a static, fixed 20px value gated by the `pwa-standalone` class**, not computed from `env()`: `.pwa-standalone .input-bar { padding-bottom: 20px; }` (`server/chat/chat.html` line ~166). This value is NOT zeroed or shrunk when the keyboard opens; it stays a constant 20px regardless of keyboard state (a deliberate simplification, not a bug).
- [ ] **The keyboard accessory bar (prev/next/done row above the iOS keyboard) cannot be hidden in a PWA.** This is a documented, accepted iOS platform limitation; only a native app shell (e.g. Capacitor) can suppress it. Do not attempt to "fix" this in the Next.js port; it is not fixable at the web-app level and must not be re-reported as a bug or regression.

## 11. Testing gate before this surface can be considered ported correctly

- [ ] **Every push-affecting change must be verified against a Chromium-family browser** (Chrome, Brave, Edge). FCM (used by all Chromium browsers) is the ONLY push service that strictly validates both the `aud` claim and that the VAPID signing key matches the subscription's `applicationServerKey`. Apple and Mozilla accept any self-consistent JWT and will NOT catch a claims-poisoning or key-mismatch bug; Safari or Firefox passing proves nothing about whether Chrome/Brave still receives push. This is exactly how the July 2026 `vapid_claims` mutation bug went unnoticed for as long as it did.
- [ ] `python tests/notif_e2e/run.py --all` is green (21 scenarios across Stage 1 emission / Stage 2 client behavior / Stage 3 service-worker handlers). Must run outside the command sandbox for `--browser`/`--all` (headless Chromium needs Mach-port access).
- [ ] `python tests/verify_push_both.py` is green: after enabling notifications on both lineup and chat in one Chromium profile, both the `push_subscriptions` (hearts.db) and `chat_push_subscriptions` (chat.db) rows report a LIVE endpoint via a real WebPush send, AND the two stored endpoints are IDENTICAL (the single-shared-subscription invariant, section 7). A "different endpoints" or a dead/`Gone` result means the unsubscribe-before-subscribe collision (or an equivalent regression) is back.

## Cutover checklist: moving `sw.js` / `manifest.json` serving from FastAPI to the new front end

Ordered so that at every intermediate step, every already-installed PWA and every already-registered push subscription keeps working with no user action required. The overriding constraint: **the push subscription endpoint URL is chosen by the browser at `subscribe()` time and is never re-derived from scratch by anything server-side**; as long as origin, scope, and the VAPID public key used to subscribe stay identical, existing subscriptions remain valid indefinitely, regardless of which backend process answers `/sw.js` or `/manifest.json`.

1. [ ] Confirm the new front end will be served from the SAME origin as today (same scheme, same host, same port as seen by the browser); a different origin is not a "migration", it is a full re-subscription event for every user and is out of scope for a zero-visible-change cutover.
2. [ ] Stand up the new front end serving `/sw.js` byte-for-byte identical to `server/static/sw.js` (same `SW_VERSION`), at the SAME path (`/sw.js`, root scope) with the SAME headers (`Service-Worker-Allowed: /`, `Cache-Control: no-cache`, `application/javascript`). Verify with a direct request diff (`curl` old vs new, byte-compare) before any traffic cutover.
3. [ ] Stand up `/manifest.json` identically (same content, same `Cache-Control: no-cache`, same media type); `start_url`/`scope` must remain `"/"` unchanged.
4. [ ] Verify `/shared.css` and `/shared.js` are also served with `Cache-Control: no-cache` (not `no-store`) from the new front end, since chat/lineup pages both depend on these updating promptly on deploy.
5. [ ] Confirm the SAME `VAPID_PUBLIC_KEY` value is available to whichever backend answers `GET /api/push/vapid-key` (lineup) and `GET /chat/api/push/vapid-key` (chat) after cutover; these two endpoints likely stay on the Python API regardless of front-end framework, but if the new front end proxies or caches them, the proxied value must be byte-identical to what's in `.env` today. A silently different public key here breaks every NEW subscribe attempt (existing subscriptions are unaffected until they need to resubscribe, e.g. via `pushsubscriptionchange`).
6. [ ] Before cutting any real traffic to the new serving path, run `python tests/verify_push_both.py` against the OLD (current FastAPI) serving path and record the two endpoint strings (lineup, chat) as a baseline.
7. [ ] Cut over `/sw.js` and `/manifest.json` serving to the new front end (same origin, same paths, same headers; steps 2-3). Do NOT touch `/api/push/vapid-key`, `/chat/api/push/*`, `/api/session/{code}/push/*`, `/chat/api/swlog`, or `/chat/api/push/ack` routing in the same change; those stay pointed at the Python backend.
8. [ ] Immediately after cutover, run `python tests/verify_push_both.py` again from the SAME browser profile/session used in step 6 (do not re-enable notifications, do not clear storage) and confirm: both endpoints are still LIVE, and both endpoints are IDENTICAL to the step-6 baseline (proves no re-subscription was silently forced by the swap).
9. [ ] Manually verify on at least one real Chromium-family device/browser (not just the automated harness) that an existing installed PWA, opened WITHOUT reinstalling or re-adding to home screen, still receives a live push after the cutover; the automated suite proves the wire protocol; this step proves the browser's on-disk SW registration didn't get treated as stale by the swap.
10. [ ] Run `python tests/notif_e2e/run.py --all` and `python -m pytest tests/test_notifications.py -v` against the new serving path as a final regression gate before declaring the PWA shell ported.
11. [ ] Only after 8-10 are all green: remove the old FastAPI static routes for `/sw.js` and `/manifest.json` (or leave them as an inert fallback briefly, per the team's own rollback comfort level; this spec does not mandate immediate removal, only that steps 1-10 all pass first).
