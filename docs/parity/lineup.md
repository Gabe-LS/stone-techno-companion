# Lineup Page: Behavioral Parity Checklist

Source of truth: `services/data/scraper/render.py` (the function `render_output_html`, ~2984 lines), read in full line by line. Where this checklist and `CLAUDE.md` disagree on a detail, `render.py` wins. This file covers the **line-up (list) view** and the mechanics it shares with the timetable view (boot contract, modals, push, hearts/schedule sync, design tokens). Timetable-grid-specific behavior (day/period tabs, grid layout, mobile table, now-line, artist popup) is documented in the sibling file `docs/parity/timetable.md`; do not duplicate that scope here beyond what is needed for shared-mechanic context.

## 1. Boot Contract

- [ ] The very first element inside `<head>` is an inline `<script>` (before any other head content) implementing "installed app only" resume: `if (location.pathname==='/' && (window.navigator.standalone || window.matchMedia('(display-mode: standalone)').matches) && localStorage.getItem('last_section')==='chat') { location.replace('/chat'); return; }`.
- [ ] That exact condition requires ALL THREE: pathname is `/`, AND (`navigator.standalone` truthy OR `matchMedia('(display-mode: standalone)').matches`), AND `localStorage.last_section === 'chat'`. In a normal (non-standalone) browser tab this branch can never fire, regardless of `last_section`.
- [ ] The same script unconditionally sets `localStorage.setItem('last_section', 'lineup')` after the standalone check (wrapped in try/catch), reclaiming the flag on every lineup page load so the next home-screen launch returns to lineup unless chat is visited again.
- [ ] A second head script (only emitted when `has_timetable` is true) computes the view class: `v = location.pathname==='/timetable' ? 'timetable' : location.pathname==='/line-up' ? 'list' : (localStorage.getItem('stc_view')==='timetable' ? 'timetable' : 'list')`.
- [ ] That script writes `localStorage.setItem('stc_view', v)` (best-effort, try/catch): this happens on EVERY load including a direct `/line-up` or `/timetable` visit, not only on manual view switches, so `stc_view` tracks "last view SEEN" not "last view toggled."
- [ ] It sets `document.documentElement.className = 'view-' + v` (i.e. `view-list` or `view-timetable`) synchronously in `<head>`, before body content parses.
- [ ] It rewrites the URL via `history.replaceState(null, '', v==='timetable' ? '/timetable' : '/line-up')` whenever `location.pathname` does not already match the canonical path for `v`: this normalizes `/` (and any other path) to `/line-up` or `/timetable`.
- [ ] Inline `<style>` immediately following contains `body{opacity:0}` (verbatim), plus CSS rules keyed to `.view-list`/`.view-timetable` that hide `#timetable-view`/`#list-view`, recolor `#btn-list`/`#btn-timetable`, and swap `#view-label`/`#page-title` text via `font-size:0` + `::after{content:'Timetable'}` tricks (so the correct label/title paints before any JS runs).
- [ ] Body is revealed by `Promise.race([_initP.catch(...), _initTimeout]).then(() => { document.body.style.opacity = '1'; })` where `_initTimeout` is `new Promise(resolve => setTimeout(resolve, 5000))`: the exact reveal failsafe is **5000ms (5s)**.
- [ ] `_initP` is the async init IIFE that restores session state (`/api/me`, `/api/session/{code}`, sync-pin exchange, or `?code=` read-only load), reconciles hearts/schedule, connects the WebSocket, and re-syncs push subscription: reveal happens when this settles (success OR caught failure) OR after the 5s timeout, whichever is first.
- [ ] The deep-link action runner (IIFE reading `window.__deepAction`, captured from `?action=` before the view router strips the query string) polls `document.body.style.opacity === '1'` every 100ms via `setInterval`, giving up after `++tries > 60` (i.e. **60 tries * 100ms = 6000ms / 6s**, deliberately longer than the 5s reveal timeout) and then runs the action 100ms later via `setTimeout(run, 100)`.
- [ ] Supported deep-link actions and their exact handlers: `picks` → `toggleFilter(document.getElementById('btn-filter'))`; `schedule` → `toggleScheduleFilter` on `#btn-schedule` if present; `share` → `openShareModal()`; `sync` → `openSyncModal()`; `notifications` → `toggleNotifications()`.
- [ ] A third head-adjacent `<script>` (still before body) registers `/sw.js` (`navigator.serviceWorker.register('/sw.js')`), listens for `message` events with `e.data.type==='navigate'`, and separately polls `caches.open('stc-push')` for a `/_push_navigate` entry at delays `[0, 300, 800, 1500, 3000, 5000]` ms: each check logs to `/chat/api/swlog` via `fetch` (POST, `{src:'lineup', step, detail}`) through a local `plog()` helper, independent of the later in-body push-navigate poller.
- [ ] Navigation guard `nav(u)`: ignores navigation if `_navigating` is true, `u` is falsy, `u` contains the substring `'line-up'`, or `u === location.pathname`; otherwise sets `_navigating = true` for 3000ms (`setTimeout`) before calling `window.location.href = u`.
- [ ] `<title>` is set to `Line-up &middot; {site_short}` (HTML entity `&middot;`, NOT a literal `·` character, NOT a hyphen) where `site_short` defaults to `"ST26"` unless overridden by the `site_short` render parameter (sourced from `events.short_name`).
- [ ] `<meta name="description">` reads: `Your companion for Stone Techno 2026. Browse the lineup, explore the timetable, plan your schedule, and get notified before your sets start.`
- [ ] Open Graph / Twitter meta tags present: `og:title`, `og:description`, `og:type=website`, `og:url=https://stonetechno.deftlab.dev/`, `og:image` and `twitter:image` both `https://stonetechno.deftlab.dev/favicon.png`, `twitter:card=summary`.
- [ ] Umami analytics script tag is present with `defer`, `data-website-id="8f79ad80-e080-421d-91c6-45b7bfc460d2"`, `data-domains="stonetechno.deftlab.dev"`, `data-auto-track="true"`, `data-performance="true"`, `src="https://analytics.deftlab.dev/script.js"`.
- [ ] Favicon is inlined as a base64 data URI (`data:image/svg+xml;base64,...`) built at render time from `icons/favicon.svg`, not a static file reference.
- [ ] `<link rel="stylesheet" href="/shared.css">` and `<link rel="manifest" href="/manifest.json">` are present; `theme-color` meta is `#111`; `apple-mobile-web-app-status-bar-style` is `black-translucent`; `mobile-web-app-capable` is `yes`; `apple-mobile-web-app-title` equals the page `title` parameter (escaped).

## 2. Artist List Rendering (Line-up view)

- [ ] Artists are grouped by date (`<section class="date-section">`), each with an `<h2>` heading rendered via `_format_date_heading`, exact format `"{Weekday}, {Month} {day}, {year}"` (e.g. "Saturday, July 11, 2026"): no leading zero on day.
- [ ] Within each date, an `<h3 class="period-heading">` (plus `fade-after` class unless the period is `"night"`) shows the period label formatted as `"{Period.capitalize()} ({time_range})"` where `time_range` is the literal string `"23:00 – 07:00"` for period `"night"` and `"12:00 – 23:59"` for anything else (en dash with spaces on both sides).
- [ ] For "night" periods, artists are further grouped by `location_id` under `<h4 class="location-heading fade-after">{floor name}{optional " <small>{description}</small>"}</h4>`: one `<ul class="artist-list">` per floor group. For non-night periods there is a single flat `<ul class="artist-list">` with no floor headings.
- [ ] If a period timestamp key has zero assigned artists, render `<p>No artists found.</p>` in place of the list (no `<ul>` emitted).
- [ ] Each artist card is `<li class="artist-item" data-artist-id="{uuid5}" data-oid="{artist db id}">`. The `data-artist-id` is `uuid.uuid5(NAMESPACE_URL, f"{oid}:{date}:{period}:{loc_id or ''}")`: deterministic per artist+date+period+floor combination, distinct from the raw DB artist id (`data-oid`).
- [ ] Photo: `<img class="artist-photo" src="photos/{photo_file}" alt="{name}" width="120" height="120" loading="lazy" tabindex="0" role="button">` when `photo_file` exists; otherwise `<div class="photo-placeholder"></div>` (no `<img>` tag at all, not a broken-image fallback).
- [ ] `.artist-photo` and `.photo-placeholder` are both `120px x 120px` (desktop), `border-radius: 6px`, shrinking to `72px x 72px` with `border-radius: var(--radius-sm)` under the `max-width: 480px` media query.
- [ ] Artist name is `<span class="artist-name" tabindex="0" role="button">{name}</span>`: keyboard-focusable and clickable, opens the bio overlay (see section 3), same as the photo.
- [ ] If the artist has a slot in the CURRENT date/period, `<span class="artist-schedule">` shows `_format_artist_schedule`'s `current_label`: `", ".join([floor_name, "{start}–{end}"])` (en dash, no spaces): floor name omitted when the section isn't split by floor (non-night periods).
- [ ] "Also" cross-reference: when the artist has OTHER slots (different date/period/floor than the current card), `<span class="artist-also">Also {label1} · {label2} ...</span>`: each label is comma-joined `floor, time` for slots on the SAME date, or `{abbreviated date}, floor, time` for slots on a DIFFERENT date (`_format_short_date_abbr` → `"{Weekday abbr}, {Month} {day}"`), multiple labels joined with `" · "` (middle dot with spaces).
- [ ] Social links render as a `<div class="links">` containing one `<a>` per `artist_links` row that has a URL AND a known platform icon (unknown platforms are silently skipped, not rendered as text). If the artist has zero links at all (`links` empty), render `<span class="missing">No links</span>` instead of an empty `.links` div.
- [ ] Heart button markup: `<button type="button" class="heart-btn" aria-label="Add to favorites" aria-pressed="false"><svg viewBox="0 0 24 24"><use href="#i-heart"/></svg></button>`: the `aria-pressed` attribute and `.active` class are both toggled by client JS (`applyHearts()`), never server-rendered as true.
- [ ] Sticky heading CSS custom properties (desktop defaults, overridden at runtime by JS): `h1` uses `top: var(--sticky-top-h1, 28px)`; `.date-section > h2` uses `var(--sticky-top-h2, 96px)`; `h3.period-heading` uses `var(--sticky-top-h3, 150px)`; `h4.location-heading` uses `var(--sticky-top-h4, 190px)`. All four are recomputed by the JS function `setStickyTops()` (see section 7) rather than left at their CSS fallback values.
- [ ] `h1::before` is an absolutely positioned pseudo-element (`bottom: 100%`, `height: 21px`, `background: var(--color-bg)`) that keeps the margin band above the sticky `h1` opaque as content scrolls beneath it: this exists specifically so the timetable view's `font-size:0` title-collapse trick cannot visually collapse the reserved space.
- [ ] `li.artist-item` background is `var(--gray-50)`, border `1px solid var(--color-surface-hover)`, border-radius `var(--radius-card)` (page-local token, `var(--radius-md)` = 8px), padding `var(--space-md)` (12px), margin-bottom `var(--space-sm)` (8px).
- [ ] At `max-width: 480px`: `li.artist-item` gains `flex-wrap: wrap`, and `.artist-also` gets `margin-left: calc(-72px - 10px)` with `width: calc(100% + 72px + 10px)` so it spans the full card width below the (now 72px) photo column.

## 3. Bio Overlay

- [ ] `bios.json` is fetched lazily, NOT eagerly: `_loadBios()` only calls `fetch('bios.json')` the first time `openBioById`/`openBio` runs, and caches the parsed result in the module-level `_biosCache` variable; every subsequent bio open reuses `_biosCache` with no network call.
- [ ] On fetch failure (network error or non-JSON), `_biosCache` is set to `{}` (empty object) rather than left `null`, so a fallback never re-attempts the network mid-session; a missing `oid` key resolves to `{ name: fallbackName || '', photo: '', bio: '' }`: a name-only overlay (photo placeholder, "No biography available" body).
- [ ] `bios.json` itself is generated server-side (not client-computed) as a dict keyed by artist `id`, deduped across all date/period assignments, each entry: `{name, photo: "photos/{file}" or "", bio: <rendered HTML or "">, videos?: [...]}`: `videos` key is present only if the artist has any entries in the `videos` param passed to `render_output_html`.
- [ ] Bio markdown is rendered server-side via `_render_markdown()`: raw text → `markdown.markdown(text, extensions=['nl2br'])` → custom `HTMLParser`-based sanitizer (NOT bleach or any external sanitizer library).
- [ ] The sanitizer's exact allowed tag set: `p, br, strong, em, b, i, a, ul, ol, li, h1, h2, h3, h4, h5, h6, blockquote, code, pre, hr`. Any tag not in this set is dropped entirely (both its start and end tag are omitted, but text content between them is still emitted/escaped).
- [ ] The sanitizer's exact allowed attribute map: only `<a href="...">` is allowed; no other tag/attribute combination survives (no `class`, `style`, `target`, `id`, etc. on any tag, no attributes at all on non-`a` tags).
- [ ] An `href` on `<a>` survives sanitization only if it matches `^https?://` (case-insensitive): `javascript:`, `mailto:`, relative, and protocol-relative URLs are all stripped (the `href` attribute is dropped from that specific tag, the `<a>` itself is kept).
- [ ] Surviving attribute values are entity-escaped: `"` → `&quot;`, `&` → `&amp;` (via a regex substitution inside the sanitizer, independent of `html.escape`).
- [ ] Booking info is stripped BEFORE markdown rendering by `_strip_booking()`: text is split on blank lines (`\n\n`) into paragraphs; the function keeps paragraphs up to (but not including) the first paragraph where ANY line matches: a case-insensitive prefix regex `^(bookings?|management|press|promos?|agency|contact|licensing|selected performance)\b`, OR contains `@` alongside `>` or `:` (an email-with-brackets/colon pattern), OR is a bare URL (`^https?://\S+$` or `^www\.\S+$`) on its own line.
- [ ] Bio overlay markup: `<div class="modal-overlay" id="m-bio" role="dialog" aria-modal="true" aria-labelledby="bio-name">` wrapping `.modal-box.bio-box` → `.bio-scroll` (max-height `80vh`/`80dvh`, `overflow-y: auto`) containing `.bio-header` (photo + name), `#bio-text`, and `#bio-videos`.
- [ ] Photo element is swapped via `photoEl.outerHTML` on every open: `<img class="bio-photo" id="bio-photo" src="{photo}" alt="{name}">` when a photo exists, else `<div class="bio-photo-placeholder" id="bio-photo"></div>`: both variants re-carry the `id="bio-photo"` so subsequent opens can find the element again.
- [ ] `#bio-text.innerHTML` is set to the pre-rendered bio HTML, or `<span class="bio-empty">No biography available</span>` when `data.bio` is falsy.
- [ ] Sets/videos list (`#bio-videos`): rendered only if `data.videos.length` is truthy; each entry is an `<a class="bio-video" href="{url if it matches /^https?:\/\//i else '#'}" target="_blank" rel="noopener noreferrer">` containing a `<img class="bio-video-thumb" src="thumbs/{video.id}.avif" loading="lazy">`, a `.bio-video-title` (video title, escaped), and `.bio-video-meta` = `"{views} views · {duration} min{ · YYYY-MM-DD if v.date present}"`: views formatted via `_formatViews` (`≥1e6` → `"{n/1e6}M"`, `≥1e3` → `"{n/1e3}K"`, both trimming a trailing `.0`, else the raw integer string).
- [ ] Opening the bio always resets scroll: `document.querySelector('.bio-scroll').scrollTop = 0` runs after every `openDialog('m-bio')` call, so a stale scroll position from a previous artist is never shown.
- [ ] Clicking `.artist-photo` or `.artist-name` (delegated listener on `#list-view` / `main`) calls `openBio(el)`, which reads `el.closest('.artist-item').dataset.oid` and the visible name text (`.artist-name` or `.tt-name`), then calls `openBioById(oid, name)`.
- [ ] Keyboard: pressing `Enter` while focus is on `.artist-photo` or `.artist-name` (delegated `keydown` listener, `e.key !== 'Enter'` guard) also opens the bio: no separate Space-key handling for these two elements.
- [ ] Modal a11y contract (applies to ALL `.modal-overlay` dialogs, not just bio): `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing at the modal's heading id (`bio-name` for the bio overlay).
- [ ] Focus return on close: `openDialog(id)` captures `document.activeElement` into `_modalTrigger` before opening; `closeDialog(id)` calls `_modalTrigger.focus()` (if set) and then nulls it out: so closing any modal returns focus to whatever element triggered it (the artist name/photo for the bio overlay).
- [ ] Tab trapping: a document-level `keydown` listener checks for an open `.modal-overlay.open`; on `Tab` it collects all `button, input, [href], select, textarea, [tabindex]:not([tabindex="-1"])` inside the modal, and wraps focus from last→first (forward Tab) or first→last (Shift+Tab) at the boundaries, via `e.preventDefault()`.
- [ ] `Escape` closes the currently open modal (`closeDialog(modal.id)`): same document-level listener, checked before the Tab-trap branch.
- [ ] Backdrop-click-to-close uses POINTERDOWN tracking, not click-target-only: each `.modal-overlay` has a `pointerdown` listener that records `ov._downOnBackdrop = (e.target === ov)`, and the `click` listener only calls `closeDialog(ov.id)` when BOTH `e.target === ov` AND `ov._downOnBackdrop` are true. This means a text selection or image-pan drag that starts inside `.modal-box` and is released over the backdrop (so the click's common-ancestor target is the overlay) does NOT close the dialog, because pointerdown did not start on the backdrop.
- [ ] Each `.modal-overlay` also has a non-passive `wheel` listener that calls `e.preventDefault()` whenever the wheel event target is not inside `.modal-box`: prevents background scroll-through while a modal is open.
- [ ] Body scroll lock during any open modal: `openDialog` sets `document.body.style.position = 'fixed'`, `top = '-' + savedScrollY + 'px'`, `left = '0'`, `right = '0'` (captured from `window.scrollY` before locking); `closeDialog` clears all four inline styles and calls `window.scrollTo(0, _savedScrollY)` to restore the exact prior scroll position.
- [ ] `.modal-box.bio-box` is `480px` wide (`max-width: 100%`), `max-height: 80vh`/`80dvh`, `border-radius: var(--radius-modal)` (14px), shrinks to `border-radius: 10px` and photo `96px x 96px` under `max-width: 480px`.
- [ ] `_fitToViewport()` (bound to `visualViewport` `resize`/`scroll` while any modal is open) recenters the open modal box against the visual viewport by setting a `translateY` transform on `.modal-box`: compensates for the on-screen keyboard shifting the visual viewport on mobile.

## 4. Social Links

- [ ] `PLATFORM_ICONS` maps exactly: `instagram → i-ig`, `soundcloud → i-sc`, `spotify → i-sp`, `youtube → i-yt`, `ra → i-ra`, `linktree → i-lt`. A platform not in this map is silently skipped in the artist card loop (not rendered, not counted toward "no links").
- [ ] Each rendered link is `<a href="{safe_href}" target="_blank" rel="noopener noreferrer" title="{platform label}">{icon svg} {follower count text}</a>` where the icon is a `<svg><use href="#{icon-id}"/></svg>` referencing a `<symbol>` in the page's inline SVG sprite (`aria-hidden="true"` on the sprite wrapper).
- [ ] Follower count text comes from `format_followers(follower_count)` (imported from `scrape.py`); an empty/`None` count renders as an empty string (icon-only link), never a placeholder like "0" or "N/A".
- [ ] Python-side URL scheme allowlist (`_safe_href` in `render.py`): `h.lower().startswith(("http://", "https://", "mailto:"))`: anything else (including empty string, `javascript:`, relative paths, `//host` protocol-relative) renders as `href="#"`. This applies to every artist social link rendered server-side in the line-up cards.
- [ ] Client-side JS equivalent for the timetable artist-popup social links (`_safeHref` function): `/^(https?:\/\/|mailto:)/i.test(u || '')`: identical allowlist (http, https, mailto), same "#" fallback for anything else.
- [ ] A THIRD, narrower allowlist applies only to bio "Sets" video links (client JS, inline in `openBioById`): `/^https?:\/\//i.test(v.url || '')`: this one does NOT allow `mailto:`, only `http://`/`https://`. Video URLs failing this check render `href="#"`.
- [ ] `@media (hover: hover) { .links a:hover { color: var(--color-text); } }`: link hover color only applies on hover-capable pointers (no sticky hover state on touch devices).
- [ ] `.links a` has `min-width: 72px` and `font-variant-numeric: tabular-nums` (desktop and mobile) so follower counts of varying digit width don't cause layout shift/jitter across a row of links.

## 5. Favorites and Schedule (sync)

- [ ] Heart toggle (`toggleHeart(btn)`): no-ops entirely if `readOnly` is true. Otherwise it's fully optimistic: `localPicks` Set is mutated, `.active` class + `aria-pressed` + `.hearted` class on the card are all flipped SYNCHRONOUSLY and BEFORE any network request, then `saveLocal()` persists to localStorage immediately.
- [ ] Heart button visual states: `.heart-btn svg` default is `fill: none; stroke: var(--color-muted-icon)`; `.heart-btn.active svg` is `fill: var(--color-accent); stroke: var(--color-accent)` (both `var(--color-accent)`, i.e. filled AND stroked the accent red, not two different colors).
- [ ] Per-artist-id request serialization: `_pickSyncChain` and `_scheduleSyncChain` are `Map<id, Promise>`: each `toggleHeart`/`toggleSchedule` call for a given artist id chains its network request (`prev.then(run)`) onto the previous pending request for that SAME id, so a rapid double-tap on one artist can never fire two overlapping POST/DELETE requests out of order; different artist ids are never chained together (fully independent).
- [ ] Server sync request: `POST`/`DELETE {API}/session/{sessionId}/pick/{id}` (heart) or `/schedule/{id}` (calendar). A `404` response clears `sessionId`/`shareToken` (both localStorage keys removed) and calls `ensureSession()` again to mint a fresh session. A non-2xx/non-204 failure ROLLS BACK the optimistic UI change (re-toggles the class/aria-pressed back to the pre-optimistic state) and re-saves local storage. A success (`res.ok` or `204`) adds/removes the id from `syncedPicks`/`syncedSchedule` (the "last known server-confirmed" mirror set) and saves.
- [ ] `ensureSession()` lazily creates a session via `POST /api/session` only when `sessionId` is falsy, deduplicated by a module-level `_sessionPromise` so concurrent callers await the same in-flight request. On success it resets `syncedPicks`/`syncedSchedule` to empty sets (a fresh session has no prior-session deletions to reconcile against) and re-POSTs every existing `localPicks`/`localSchedule` id to the new session.
- [ ] `reconcile()` (runs once at init if a `sessionId` already exists in localStorage, never on every load if not needed): fetches `GET /api/session/{sessionId}`; for each LOCAL pick/schedule id not present on the server, it is either re-POSTed (genuinely new offline change) or silently dropped locally: determined by whether the id is present in `syncedPicks`/`syncedSchedule` (if yes, another device deleted it server-side and a missed WebSocket broadcast is assumed, so the local copy is deleted rather than resurrected via re-POST).
- [ ] `localStorage` key migration on load: if `stc_edit_code` exists and `stc_session_id` does not, it's copied to `stc_session_id` and the old key removed; same pattern for `stc_share_code` → `stc_share_token`.
- [ ] Read-only shared-picks viewing: loading the page with `?code={shareToken}` calls `loadFromServer(code)`, which sets `readOnly = data.readonly` from the server response; when `readOnly` is true, EVERY heart button, `.tt-heart`/`.tt-photo-heart`, and `.tt-cal` button gets `style.pointerEvents = 'none'` (JS-level disable, not a `disabled` attribute), `filterActive` is forced true, `document.body` gets `.filter-active`, and `#btn-filter` is hidden (`style.display = 'none'`) so a visitor viewing someone else's shared link cannot toggle their own filter off.
- [ ] `openShareModal()` requires an existing `shareToken`; if none, `alert('Heart an artist first.')` (native `alert()`, not `showToast`) and the modal never opens. When it does open, `#share-link` is populated with `{location.origin}/?code={shareToken}` and the modal input is `readonly`, clicking it selects the text and copies via `navigator.clipboard.writeText`, showing "Copied!" in the same input for 1500ms before reverting (`.copied` class also toggled for the visual green background).
- [ ] Sync PIN flow: `openSyncModal()` calls `ensureSession()` first (creates a session if none exists, same "heart an artist first" alert gate applies implicitly since a session requires nothing but IS required for the pin), then `POST /api/session/{id}/sync-pin` returns a 6-digit `pin`; a client-side countdown (`_syncTimer`, `setInterval` 1000ms) computes remaining time from a captured `deadline = Date.now() + 300000` (**exactly 300000ms / 5 minutes**), rendering `"Valid for {m} min"` while ≥60s remain or `"Valid for {s}s"` below that; on expiry the PIN display and QR canvas are cleared and replaced with `"QR code and PIN expired. Generate new ones"` (clickable, re-invokes `generateSyncPin()`).
- [ ] The PIN's QR code is rendered client-side via a bundled `qrcode.min.js` library (inlined into the page as a `<script>` at render time from `services/data/scraper/qrcode.min.js`), encoding `{location.origin}/?sync={pin}`, drawn onto a `360x360` canvas (`#sync-qr`) but displayed at `120x120` CSS size; `.qr-wrap` (and thus the QR code) is hidden entirely at `max-width: 480px` (mobile shows only the numeric-entry instructions).
- [ ] PIN entry UI (receive-side): a hidden real `<input class="pin-real" id="pin-input" type="text" inputmode="numeric" maxlength="6">` overlays 6 visual `<span>` boxes (`#pin-boxes`); input is sanitized to digits-only and truncated to 6 chars on every `input` event (`replace(/\D/g, '').slice(0, 6)`); a blinking caret (`@keyframes blink`, 1s step-end infinite) appears on the active empty box only while the wrapper has `.focused` (via a real `focus`/`blur` listener pair) and that box is `.active` and empty.
- [ ] `submitPin()` requires exactly 6 digits (else no-ops), closes the sync modal, clears the input, then calls `exchangeSyncPin(pin)` → `POST /api/sync/{pin}`: on success this replaces `localPicks`/`localSchedule`/`syncedPicks`/`syncedSchedule` wholesale from the response, sets `readOnly`, persists a new `sessionId`/`shareToken` if not read-only, and reconnects the WebSocket.
- [ ] On successful WebSocket `sync_complete` message: the client stops any running sync-PIN countdown timer, hides `#sync-pending` and shows `#sync-done` (green checkmark SVG + "Device synced successfully" text) inside the still-open sync modal: the modal is NOT auto-closed, the user must dismiss it manually (Escape/backdrop/reopen elsewhere).
- [ ] WebSocket connection: `connectWS(code)` opens `wss://{host}/ws/{code}` (or `ws://` on non-HTTPS). On any incoming message containing a `picks` key, it wholesale-replaces `localPicks`/`syncedPicks` (and `schedule`/`syncedSchedule` if present) and calls `applyHearts()`; a message carrying `readonly !== undefined` updates the read-only lock state live (mid-session downgrade to read-only is possible without a page reload).
- [ ] WebSocket reconnect backoff: on `onclose`, if `event.code === 1008` the socket is NOT retried at all (treated as a permanent/authorization-level close); otherwise a reconnect is scheduled after `_wsDelay + Math.random()*1000` ms, `_wsDelay` starts at 2000ms and doubles on each subsequent close (`Math.min(_wsDelay * 2, 60000)`, capped at 60000ms/60s): but only reconnects if the current `sessionId`/`shareToken` still matches the code the closed socket was for (guards against reconnecting a socket for a session that's since been superseded).
- [ ] Every artist card's calendar/schedule icon lives only in the timetable block markup (`.tt-cal`, calendar SVG symbol `i-cal`), not on the flat line-up list cards: the line-up view has no per-card schedule button, only the heart. `.tt-cal.active svg` recolors to `var(--color-schedule)` (`#4a90d9`); `.tt-block.scheduled` additionally gets `box-shadow: inset 0 0 0 2px var(--color-schedule)`.
- [ ] "Show My Picks" command-bar button (`#btn-filter`) toggles `document.body.classList.toggle('filter-active', filterActive)`; CSS then hides any `.artist-item:not(.hearted)` via `.filter-active .artist-item:not(.hearted){display:none}`, and hides an entire `date-section`/`ul.artist-list` if it contains NO hearted item via `:has()` selectors: EXCEPT `h4.location-heading` visibility, which is deliberately JS-driven (`updateGroupVisibility()` toggling a `.no-hearted` class) rather than a `:has()` selector, because a `:has(+ ul:not(:has(...)))` selector was found to re-evaluate broadly on every single heart toggle across the whole page (documented as a performance decision in a code comment).

## 6. Push Notifications (Lineup Side)

For the shared service-worker / cross-surface push mechanics (tag uniqueness, VAPID claims-dict isolation, TTL, Brave/FCM asymmetry, the lineup/chat shared-subscription invariant), see `docs/parity/pwa-shell.md`: this section documents only what `render.py` emits for the lineup page's own enable/disable control.
- [ ] Feature/environment detection constants computed once at load: `_isIOS = /iPad|iPhone|iPod/.test(navigator.userAgent)`; `_isStandalone = matchMedia('(display-mode: standalone)').matches || navigator.standalone === true`; `_needsSafariSwitch = _isIOS && !!navigator.brave` (an iOS browser identifying as Brave); `_supportsPush = 'serviceWorker' in navigator && 'PushManager' in window`.
- [ ] `updateBellState()` shows/hides `#btn-bell` based on `_supportsPush` (`display: none` if unsupported), toggles `.active` on it based on `localStorage.stc_push === '1'`, and independently shows/hides the hamburger-menu `#dd-bell` row based on `_supportsPush || _isIOS` (iOS is always shown even without PushManager support, since it can route through the "Add to Home Screen" instructional flow) plus sets its `aria-checked`.
- [ ] `enableNotifications()` branches in this exact order: (1) if `_needsSafariSwitch`, open `#m-ios-switch` modal and return; (2) else if `_isIOS && !_isStandalone`, open `#m-ios` modal (Add to Home Screen instructions) and return; (3) else if `!_supportsPush`, `showToast('This browser does not support notifications.')` and return; (4) otherwise call `Notification.requestPermission()`.
- [ ] If permission is not `'granted'`, `showToast('Notifications are blocked. Enable them for this site in your browser settings, then try again.')`: this is a deliberate departure from silently failing, called out in a code comment ("the switch never looks inert").
- [ ] On granted permission: fetches `GET /api/push/vapid-key`, gets `navigator.serviceWorker.ready`, and (critically) calls `reg.pushManager.getSubscription()` FIRST and only calls `subscribe()` if `!sub` (no existing subscription). It NEVER calls `unsubscribe()` before `subscribe()` in this flow, preserving a subscription shared with the chat page.
- [ ] After obtaining/reusing the subscription, it calls `ensureSession()`, then `POST /api/session/{sessionId}/push/subscribe` with the subscription's `toJSON()` body. Only on that request's success does it set `localStorage.stc_push = '1'` and `localStorage.stc_push_endpoint = sub.endpoint`, and track the `push-enable` analytics event.
- [ ] Brave-specific failure path: if `navigator.brave` is truthy and the subscribe call throws `AbortError`, open `#m-brave` modal (instructions: open `brave://settings/privacy`, enable "Use Google services for push messaging", come back and tap the bell again) instead of the generic error toast.
- [ ] Generic subscribe failure: `NotAllowedError` → the same "blocked, enable in browser settings" toast as the permission-denied case; any other error → `showToast('Could not enable notifications. Please try again.')`.
- [ ] `disableNotifications()`: fetches the current subscription; if one exists, always `DELETE`s the server-side lineup record (`/api/session/{sessionId}/push/subscribe` with `{endpoint}` body) when a `sessionId` exists, but only calls `sub.unsubscribe()` on the actual browser subscription when `localStorage.push_enabled !== '1'` (the CHAT page's own enabled-flag): i.e. disabling lineup notifications never revokes the shared browser subscription while chat notifications are still enabled. Always removes `localStorage.stc_push` and tracks `push-disable` regardless of the unsubscribe branch taken.
- [ ] `toggleNotifications()` is a pure dispatcher: calls `disableNotifications()` if `stc_push === '1'`, else `enableNotifications()`: bound to both the bell button (`#btn-bell`) and the hamburger-menu row (`#dd-bell`).
- [ ] At init, if `stc_push === '1'` and a service worker is available, the page re-fetches the current `pushManager.getSubscription()` and re-POSTs it to `/api/session/{sessionId}/push/subscribe` unconditionally (self-healing re-sync on every load, independent of whether it changed): if no subscription exists anymore (revoked at the OS/browser level), `stc_push` is removed from localStorage and `updateBellState()` is called to reflect the now-disabled state.
- [ ] Three instructional modals exist purely as static markup with copy-to-clipboard buttons (`navigator.clipboard.writeText(location.origin)`, button text flips to "Copied!" for 1500ms): `#m-ios` (Add to Home Screen, generic), `#m-ios-switch` (non-Safari iOS browser: copy link + switch to Safari + a tip to use Sync first to carry picks over), `#m-brave` (Brave settings steps, no clipboard button).
