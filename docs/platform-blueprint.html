Stone Techno → European festival platform

Build vs buy, and the architecture to get there.

The direction is a platform — content, community, ticketing, merch, and a database of festivals + transport + line-ups across Europe. This is the plan for what to build, what to adopt, and how it wires together — grounded in verified market + stack research, and positioned honestly against the closest real competitor.

A The competitive reality web-verified

No one owns all the pillars — but it is not empty ground. Woov already combines a line-up/timetable + community chat + ticketing across many events, so it's the real benchmark. The ticketing majors (RA/Shotgun/DICE) have ticketing but no timetable or chat; Clashfinder is timetable-only; Radiate is chat-first with light set-times.

B Build vs buy, pillar by pillar

Festival / line-up / transport data Build

Your moat. The scrape → enrich → normalize pipeline + realtime transit is the differentiator and the hardest thing to copy. Keep it in Python. No product does this well.

Community: chat / meetups Keep

Your bespoke E2EE chat + meetups + moderation is already more festival-tailored than Rocket.Chat/Matrix. Keep and scale it; don't replace it with off-the-shelf.

Ticketing Adopt

Do not build a payments/inventory engine. Adopt pretix (Django, festival-native, offline check-in) or Hi.Events (Laravel/React). Integrate, don't rebuild.

Merch shop Adopt

Medusa (MIT, headless, Node/TS) — variants, multi-region, bundle-with-tickets. Theme its storefront rather than writing commerce from scratch.

CMS (event/editorial content) Adopt

Payload (MIT core, TS/Next-native) for editorial + flexible schemas. Note: a paid Enterprise plugin tier exists on top of the free core.

Search Adopt

Meilisearch (or Typesense) — sub-50ms typo-tolerant search over artists / line-ups / POIs, one lightweight binary. Good on festival wifi.

Maps / geo / wayfinding Keep

You already run the right open stack: MapLibre + OpenFreeMap + OSRM. Add PMTiles for offline venue tiles. Nothing to buy.

Safety & awareness Build

Emergency alerts (organizer-pushed), first aid / awareness team / safe space / water locations on the map (POI categories you already have), one-tap contact button. Data-driven: organizers manage it via CMS or the dashboard. A trust differentiator when onboarding festivals.

Food & drink Build

Vendor directory with menus and prices, dietary filters (vegan, gluten-free), locations on the map. A major real pain point at festivals. Natural extension of the POI system + CMS (organizers or vendors maintain their listings).

Ride sharing Build

Offer/request rides to and from festivals. Route matching (origin, destination, date, seats), in-app coordination via chat (DM the driver), map visualization of pickup points, cost splitting. Complements the realtime transit board: public transport for the last mile, ride sharing for the long haul. Integrates with your existing maps, chat, and user identity.

AI support agent Build

A bot user inside the chat system in a dedicated support room, grounded in your festival data via tool-calling (schedule, artists, transport, POIs, FAQs). Answers "When does X play?", "How do I get there?", "Where's the nearest water?" from real data, not hallucination. Leverages existing chat infra, moderation, and the structured DB. Another differentiator no competitor offers.

On-sale / high-concurrency real-time Later

Only if ticket on-sales spike hard. The proven pattern is an Elixir/Phoenix waiting-room service (Veeps: 20 Rails → 2 Elixir nodes). Defer until scale demands it.

Each adopted system keeps its own datastore and is integrated at the API/BFF layer, not merged into one DB — the standard way these OSS pieces are meant to run. Postgres + Redis is the near-universal pairing across pretix, Saleor, Medusa and Discourse verified.

D The stack verdict matches how the market builds this

The empirical pattern across Eventbrite, Saleor, Medusa, DICE and the adoptable OSS — not a guess:

Next.js (React) front · Python (FastAPI/Django-family) backbone · Postgres + Redis · adopt pretix / Medusa / Payload / Meilisearch · Elixir/Phoenix only for on-sale/real-time at scale

React/Next.js recurs on every verifiable content-commerce-community platform, and every adoptable storefront (Saleor, Medusa) ships Next.js starters — so Next.js beats SvelteKit at platform scale on ecosystem + integration fit. Python stays central — it's where your pipeline already lives and what Eventbrite/Saleor/pretix run. This is the one place the "greenfield" answer and your existing investment finally agree.

E Phased path — today's app is phase 0

Phase 0

Harden the companion (now)

Single-event app, evolve in place. Ship only the migration-proof menu fixes (unified active state, the 768px breakpoint bug, chat desktop nav, aria/Escape gaps); the shared menu component is deferred to the Next.js front so the abstraction is built once (decision record: docs/menu-sequencing-strategy.md). Fill the deferred meetup gaps: push wiring (room_memberships branch for meetup rooms), attendee-list UI, pre-meetup reminders. No framework migration. Done when: the deferred list is empty and the next event runs on this codebase without incident.

Phase 1

Multi-event, data-driven + AI agent

Event picker + a festival/route/artist data model (config-driven, not files; the transport itineraries move from hand-edited JSON into it). Search (Meilisearch). AI support agent as a bot user in chat: tool-calling over the structured DB (schedule, transport, POIs) so answers are grounded in real data. This is the moat — build it before anything commercial. Done when: adding a third event needs only a scraper module plus config (no code changes elsewhere), and the agent answers schedule, transport and POI questions from live data.

Phase 2

The platform front-end

Stand up the Next.js app consuming the Python data + companion API; port the content pages; keep chat as-is behind it. Add Payload for editorial content. Done when: every content page is served by Next.js at feature parity on the same origin, render.py is retired, and the push harness (tests/notif_e2e, verify_push_both) passes against the new front in a Chromium-family browser.

Phase 3

Commerce & scale

Integrate pretix (ticketing) and Medusa (merch) at the BFF, behind the single identity service. Add an Elixir on-sale service only if/when spikes demand it. Now you're a platform. Done when: a real ticket and a real merch order complete end to end through the BFF with one login.

G Migration invariants: break any of these and users lose data silently

Hard-won constraints from the live system. Every one of them was learned the expensive way; the restructure inherits them as non-negotiables.

Slot UUIDs are permanent

slot_uuid() (in timetable_json.py) is the identity of a set for saved schedules, push dedup and ICS. Any schema change or Postgres migration must carry these ids unchanged, or every user's saved schedule and notification history resets.

One origin, one push subscription

Lineup and chat share the root-scope sw.js and a single browser subscription per origin. The new front must serve the same worker at root scope on the same origin, and must never unsubscribe-then-resubscribe (it rotates the endpoint and orphans the other surface's record).

One manifest, one app identity

A second manifest is a second iOS storage partition: separate sign-in, separate push. Already tried and reverted. The platform keeps start_url: "/" and exactly one manifest, whatever serves the pages.

VAPID keys migrate, never rotate

Rotating the key pair silently kills every existing subscription. The pem moves with the infrastructure, the startup consistency check stays, and every push change is tested on a Chromium-family browser (FCM is the only strict validator; Apple and Mozilla passing proves nothing).

E2EE keys live client-side

DM device keys sit in localStorage per browser profile and the server cannot re-encrypt history. Any origin change or storage-clearing migration strips users' ability to decrypt. DM migration means envelope and key continuity, not data transformation.

Identity continuity for moderation

Bans and strikes are provider-keyed and deliberately survive user deletion. Whatever identity layer fronts pretix/Medusa/Payload must preserve provider_id continuity, or every ban silently expires at cutover.

H The honest risks — where this is hard, not the framework

The data

Scraping all of Europe is a permanent ops + legal/ToS burden. Many sites forbid it; you'll need partnerships/APIs. This, not the UI, is the make-or-break.

Ticketing & payments

PCI, fraud, refunds, regulation. Adopt (pretix) or integrate existing sellers — do not build it. Ticketing is a business, not a feature.

Woov exists

Timetable + chat + ticketing already shipped, multi-event. Compete on the transit edge + scene depth, not by re-doing what they have.

Scope & people

This is 3–4 products. Realistically a funded, team effort — not a solo per-event side project. Sequence it; don't build it all at once.

I Open decisions: resolve each before its phase starts

D1 · Phase 1

Ingestion model for festival number two

Who adds the next event: a hand-written scraper module (the current contract), organizer self-serve through the CMS/dashboard, or partnership data feeds? The legal exposure in section H differs completely per path, and Phase 1's data model design depends on the answer.

D2 · Phase 1

Organizer dashboard owner

The safety and food pillars assume "organizers manage it via CMS or the dashboard", but no pillar builds that dashboard. Pick its home: Payload's admin, pretix's organizer area, or a custom surface. It also decides where POIs live long-term (today: a MapTiler dataset).

D3 · Phase 1

AI agent model and cost ceiling

Which model, what per-event budget, what rate limits, and how the bot interacts with the moderation pipeline (its answers post into a moderated room). Cheap to prototype, but the cost model must exist before it is public.

D4 · Phase 2

Postgres cutover order

Which SQLite database moves first and how the dual-write or cutover window works: lineup.db (regenerable, lowest risk), then hearts.db/chat.db (live user data, section G constraints). SQLite has served flawlessly at current scale; the move is driven by multi-service topology, not by load.

D5 · Phase 2

Repo and hosting shape

Monorepo vs polyrepo (where does the Next.js app live relative to the Python services), and hosting: one VPS will not hold Next.js + FastAPI + pretix + Medusa + Payload + Meilisearch + Postgres + Redis. Decide the orchestration story and the monthly budget line before Phase 2 begins.

D6 · Phase 3

Identity broker

Companion auth promoted to an OIDC identity provider, or an external IdP (Keycloak/Authentik/Ory) fronting everything? Must be answered before pretix and Medusa integration, and must satisfy the section G continuity invariants (provider ids, sessions, E2EE device keys).
