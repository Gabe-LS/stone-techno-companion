## What to do

### 1. SVG Icon Deduplication

Both pages paste full SVG markup inline every time an icon is used. The chat page already has constants for some icons (`ICON_HAMBURGER`, `ICON_BACK`, `USER_ICON`), but the calendar icon and chat icon are duplicated as raw SVG in HTML templates.

**Approach:**

1. Audit every inline SVG across all pages. For each, document: what icon it is, how many times it appears, whether it already has a JS constant.
2. For icons used on multiple pages (calendar, chat bubbles), add constants to `shared.js`.
3. For icons used only on one page, add constants at the top of that page's inline `<script>` block (alongside existing constants like `ICON_HAMBURGER`).
4. Replace every inline SVG paste with the constant reference.
5. The lineup page generates HTML via Python strings in `scraper/render.py` — SVG icons there are part of Python string concatenation. For icons that need to be JS constants, the Python code should output `${ICON_NAME}` inside template literals, or the icon should be injected via JS after page load. Choose the approach that's cleanest for each case. Some SVGs in the lineup are inside `<a>` tags generated by Python — these may need to stay as Python strings if they're not inside template literals.

**Naming convention:** `ICON_CALENDAR`, `ICON_CHAT`, `ICON_HAMBURGER`, `ICON_BACK`, `ICON_USER`, etc. UPPER_SNAKE for constants.

**SVG sizing:** Icons should use the same `width`/`height` in the constant. If different sizes are needed in different contexts, the constant uses the default size and CSS overrides it (e.g., `.header-cal svg { width: 20px; height: 20px; }`).

**SVG viewBox scaling:** Some icons (calendar, chat) use `viewBox="-1.7 -1.7 27.4 27.4"` instead of `viewBox="0 0 24 24"` — this adds visual padding to match the hamburger icon's apparent size. Preserve these viewBox values exactly when creating constants.

**Existing icon constants** in `chat.html`: `ICON_HAMBURGER`, `ICON_BACK`, `USER_ICON`. The lineup page has the hamburger SVG inline in Python. The calendar SVG appears in both the mobile nav icon and would appear in a desktop header link — deduplicate into one constant.

### 2. Event Delegation

The Phase 1 report noted: "239 inline `onclick="openBio(this)"` handlers — replace with one delegated listener on `#list-view`."

**Approach:**

1. Identify all inline event handlers (`onclick`, `onkeydown`, `onmousedown`, `oninput`, `ondblclick`, `ontouchstart`, etc.) across both pages. Count them per page.
2. For handlers on repeated elements (artist cards, room items, message bubbles, reaction buttons), replace with a single delegated listener on the parent container.
3. For handlers on unique elements (single buttons, inputs), inline handlers are acceptable — no need to delegate.
4. The delegated listener should use `event.target.closest('.class')` to find the relevant element.
5. Be careful with `event.stopPropagation()` — some inline handlers call it. Delegated handlers need equivalent behavior.

**Example transformation:**

Before (239 artist cards):
```html
<li onclick="openBio(this)" onkeydown="if(event.key==='Enter')openBio(this)">
```

After (one listener):
```javascript
document.getElementById('list-view').addEventListener('click', function(e) {
  const card = e.target.closest('.artist-item');
  if (card) openBio(card);
});
```

**Both pages have inline handlers.** The lineup has 239 `onclick="openBio(this)"` on artist cards. The chat has inline handlers on room items, message bubbles, reaction buttons, action menus, and more. Audit both pages — focus on repeated elements where delegation saves the most.

**Important:** Test that keyboard navigation, screen readers, and touch interactions still work after the change.

### 3. Empty Catch Blocks

Phase 1 fixed the most impactful one (WebSocket event handler). 13+ `catch {}` blocks remain in the lineup JS that silently swallow errors.

**Approach:**

1. Find every empty `catch {}` or `catch { }` block across all pages.
2. For each, determine what error it's catching and why it was silenced.
3. Categorize:
   - **Expected errors** (e.g., `localStorage` in private browsing, optional feature detection) — add a comment explaining why silence is correct: `catch { /* expected in private browsing */ }`
   - **User-facing errors** (e.g., session creation, sync PIN exchange, network requests) — add `dbg()` logging and show a toast where appropriate
   - **Programming errors** (e.g., JSON parse of data we control) — add `dbg()` logging so bugs are visible in console
4. No `catch {}` should remain without either a comment justifying the silence or a `dbg()` call.

### 4. storageGet/storageSet Adoption

Phase 1 added `storageGet(key, fallback)` and `storageSet(key, value)` to `shared.js` but didn't replace existing raw `localStorage` calls.

**Approach:**

1. Find every `localStorage.getItem` and `localStorage.setItem` across all pages.
2. Replace with `storageGet` and `storageSet`.
3. The wrappers handle try/catch for private browsing mode — this removes the need for individual try/catch blocks around storage calls.
4. For `localStorage.removeItem`, add a `storageRemove(key)` to `shared.js` if needed.

### 5. Browser Compatibility Fixes

Phase 1 identified two compatibility issues:

#### 5.1 OffscreenCanvas fallback

`OffscreenCanvas` + `convertToBlob` is used for avatar cropping and image resizing. Not available in Safari 15.4-16.3.

- Check where `OffscreenCanvas` is used in the chat page
- Add a feature check: `if (typeof OffscreenCanvas !== 'undefined')`
- Fallback: use a regular `<canvas>` element (create one off-screen, draw to it, use `canvas.toBlob()`)
- Test on a browser without OffscreenCanvas support (or mock it away)

#### 5.2 VideoEncoder guard

`VideoEncoder` (WebCodecs) is Chrome-only. Firefox and Safari don't support it.

- Check where `VideoEncoder` is referenced
- Add a guard: `if ('VideoEncoder' in window)` before any WebCodecs usage
- If unavailable, show a toast: "Video processing is not supported in this browser. Please use Chrome."
- Do NOT attempt a full polyfill — WebCodecs is too complex

### 6. Double updateUI Call

Phase 1 noted: "`saveLocal`/`applyHearts` double `updateUI` — remove `updateUI()` from inside `saveLocal()` since callers already handle it."

- Read `saveLocal()` and all its call sites in the lineup JS
- Verify that every caller of `saveLocal()` also calls `updateUI()` (or equivalent)
- If yes, remove the `updateUI()` call from inside `saveLocal()`
- If any caller doesn't handle UI update, keep it or add the call to that caller

### 7. CSS :has() for Filter Visibility (Optional)

Phase 1 noted that three of four hiding cases in `updateGroupVisibility` can become pure CSS:

```css
.filter-active section.date-section:not(:has(.artist-item.hearted)) { display: none }
```

**Approach:**

1. Check browser support for `:has()` — Safari 15.4+, Chrome 105+, Firefox 121+ (Dec 2023). Verify this meets the 95% target on caniuse.com before proceeding. If Firefox 121+ is below 95%, skip this task.
2. Identify the JS `updateGroupVisibility` function and what it does
3. Replace the JS hiding logic with CSS `:has()` rules where possible
4. Keep the JS for the one case that CSS can't handle (if any)
5. Test that filtering still works correctly — toggle "Show My Picks" on/off, verify correct sections hide/show

**Important:** `:has()` can have performance implications on large DOMs. The lineup may have 100+ artist cards. Test scroll performance after the change.
