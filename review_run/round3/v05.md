**Claim: A**
**Verdict: CONFIRMED**

Reasoning:
- `server/chat_api.py:1499-1505` (`chat_push_subscribe`) — `endpoint = body.get("endpoint", "")`, only checked via `if not endpoint or not p256dh or not auth` (truthiness/non-empty), then passed straight to `save_push_subscription`. No `urlparse`, no scheme check, no host allowlist.
- `server/chat_db.py:1589-1599` (`save_push_subscription`) — raw `INSERT`, no validation layer at the DB boundary either.
- `server/api.py:794-821` (`push_subscribe`, lineup) — same pattern: only non-empty check on `endpoint`, `p256dh`, `auth`; the only regex validation (`TOKEN_RE`) applies to the session `code` path param, not the endpoint body field.
- Both stored endpoints are later read back unmodified and handed to `pywebpush.webpush(subscription_info={"endpoint": sub["endpoint"], ...})` — confirmed at `server/chat_ws.py:463-479` and `server/api.py:359-374`.
- **Reachability is low-barrier on both surfaces**, confirmed via sub-agent: the chat endpoint only requires completing Google OAuth or email magic-link signup (trivial, self-service, no identity verification beyond owning an email address); the lineup endpoint requires **zero authentication** — any anonymous visitor can `POST /api/session` (rate-limited only) to mint a session code, then register a fake subscription against it.
- The attacker needs a syntactically valid P-256 `p256dh` key and 16-byte `auth` secret for `pywebpush`'s ECDH encryption step to succeed before the HTTP POST fires — but generating those is a two-line WebCrypto call, not a meaningful barrier.
- Pushes aren't attacker-triggered manually either: chat pushes fire automatically on normal room activity/DMs to idle members (`chat_ws.py` push trigger), and lineup pushes fire every 60s from the scheduler (`api.py` background task) — so registering the malicious endpoint is sufficient; no further attacker action needed to make the server issue the VAPID-signed outbound POST to an arbitrary internal or metadata-service URL.
- Nothing in CLAUDE.md's "Hard-won invariants" section for push discusses endpoint validation — it documents VAPID-claims-mutation and tag-uniqueness bugs only, consistent with this being an unaddressed gap.

**Claim: B**
**Verdict: CONFIRMED**

Reasoning:
- `server/chat_ws.py:465-479` and `server/api.py:361-374` both call `await asyncio.to_thread(webpush, subscription_info=..., data=..., vapid_private_key=..., vapid_claims=dict(vapid_claims))` — no `timeout=` kwarg at either site (confirmed by reading the full call args in both files).
- `asyncio.to_thread` runs on the default loop executor when none is set. A sub-agent search of `server/api.py`, `server/chat_api.py`, `server/chat_ws.py`, `server/chat_moderation.py` found **no custom executor/`ThreadPoolExecutor` configuration anywhere** — the app uses Python's default pool (`min(32, os.cpu_count()+4)` workers), and that same pool is shared by unrelated blocking work also dispatched via `asyncio.to_thread`: image processing (`chat_api.py:1308`), video ffprobe/ffmpeg frame extraction (`chat_api.py:1353/1407/1435`), email validation and Maileroo send (`chat_api.py:387/430`), and DNS resolution for link-preview SSRF guarding (`chat_ws.py:155`).
- On the `pywebpush`/`requests` behavior specifically (the one point I couldn't re-verify by execution this session): `pywebpush.webpush()` exposes a `timeout` parameter defaulting to `None`, which it forwards directly to `requests.post(..., timeout=timeout)`; `requests` treats `timeout=None` as "no timeout, block forever" — this is stable, long-documented behavior of both libraries. Since neither call site here passes `timeout=`, that default resolution applies.
- Combined with Claim A, an attacker-controlled endpoint that accepts the TCP/TLS handshake but never responds (or an internal service with a hung read) will pin the worker thread indefinitely per delivered push. Because the thread pool is shared app-wide, enough such subscriptions receiving pushes concurrently (trivial to arrange given B's endpoint has zero auth) can starve the pool and stall unrelated request handling — image/video uploads, email login, and link-preview DNS checks all funnel through the same executor.

No guard, allowlist, or timeout wrapper was found anywhere in the reviewed code that would prevent either issue — both stand as reported, Claim A as CRITICAL SSRF, Claim B as a HIGH-severity resource-exhaustion amplifier that depends on A being reachable (which it is).
