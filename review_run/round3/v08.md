**Claim: A**
**Verdict: CONFIRMED**

Reasoning: All three client auth call sites send no `device_fingerprint`:
- `/auth/google` body: `{ id_token: resp.credential }` (chat.html:527)
- `/auth/google/code` body: `{ code: resp.code }` (chat.html:570)
- `/login` body: `{ email }` (chat.html:598)

The generic `api()` helper (chat.html:462-474) is a plain `fetch` passthrough — it does not inject any fingerprint header/field, so there's no hidden lower-layer source for it either.

Server-side, `fingerprint = body.get("device_fingerprint")` (chat_api.py:263, 309, 383) is always `None` for every entry point, and this `None` is threaded correctly all the way through: into `email_tokens.fingerprint` (chat_api.py:413), read back on magic-link verify (chat_api.py:473), and into `_authenticate` → `is_banned(db, provider, provider_id, device_fingerprint)` (chat_api.py:217). In `is_banned` (chat_db.py:506-523), the fingerprint-based lookup is gated behind `if device_fingerprint:` (line 518) — with `None`, that branch is skipped entirely, leaving only the `(provider, provider_id)` exact-match check (line 512-515). So the fingerprint half of ban enforcement is dead: reachable in code, never reachable in practice. A banned user evades the ban simply by signing in with a different email/Google identity, since a fresh identity has no matching `(provider, provider_id)` row and the fingerprint fallback never engages.

**Claim: B**
**Verdict: CONFIRMED**

Reasoning: All three auto-ban call sites in chat_moderation.py call `ban_user` with only the single frozen identity from the `users` row:
- 4-strike ban: `ban_user(db, user_id, user["provider"], user["provider_id"], ..., user["device_fingerprint"])` (chat_moderation.py:355-362)
- mute-cycling ban: same pattern (chat_moderation.py:369-376)
- instant AI-moderation ban: same pattern (chat_moderation.py:482-489)

`ban_user` (chat_db.py:486-503) does a single `INSERT INTO bans` for exactly the `(provider, provider_id)` passed in — no internal fan-out to other linked providers.

By contrast, `admin_ban` (chat_api.py:1739-1787) explicitly queries `user_providers` for every linked `(provider, provider_id)` pair and inserts a ban row for each (lines 1749-1766), plus a fallback insert for the base `users` identity if not already covered (lines 1768-1776) — matching the CLAUDE.md note that admin ban "closes the second provider can't evade it gap."

Since `is_banned` matches only on exact `(provider, provider_id)` or a (currently inert, per Claim A) fingerprint, a user auto-banned via strikes/mute-cycling/AI moderation on provider A remains fully able to authenticate via any other linked provider B — the ban row for A never blocks B. This is a real, reachable gap: `add_user_provider` links Google+email accounts automatically on first sign-in when emails match (chat_api.py:295, 299, 359, 363), so most users acquire a second linked identity without any deliberate action, making the evasion path trivial for any user who has ever signed in via both a magic link and Google.
