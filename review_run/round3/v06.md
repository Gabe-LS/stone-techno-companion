**Blocking-call sub-claim — CONFIRMED.**
- `server/chat_api.py:321` calls `httpx.post("https://oauth2.googleapis.com/token", ...)` — this is the synchronous `httpx` API (module-level `httpx.post`, not `httpx.AsyncClient`), invoked with no `await` and no `asyncio.to_thread` wrapper, inside `async def auth_google_code` (chat_api.py:305).

Neither call site wraps this in `to_thread` either.
- Both are directly inside `async def` route handlers with no thread offload, so they block the single asyncio event loop for the full DNS+TLS+HTTP round-trip duration.
- Production runs a single uvicorn worker (`server/Dockerfile`: `CMD ["uvicorn", "api:app", "--host", "0.0.0.0", "--port", "8080", ...]`, no `--workers` flag) — i.e., one process, one event loop for the entire server (lineup + chat WS + everything else).

**Rate-limiting sub-claim — CONFIRMED.**
- `chat_api.py:369-376` shows `/login` is protected by the `_email_rate` per-IP dict limiter (5 requests / 900s).
- `/chat/api/auth/google` (line 259) and `/chat/api/auth/google/code` (line 305) have no equivalent — no `_email_rate`, `_check_rate`, decorator, or dependency anywhere in either handler.
- `server/api.py` has only a manual `_check_rate()` helper applied selectively per-route (lines 75-88), and no global middleware (`add_middleware`) — confirmed no slowapi/starlette rate-limiting middleware exists anywhere in the app.

This is a real event-loop-stall DoS, not a false positive — severity CRITICAL is justified.
