## Claim A: CONFIRMED (HIGH)

**Code path**: `server/api.py:335-402`. The per-endpoint loop at 359-385 catches only `WebPushException` (376). I verified against pywebpush's actual source (GitHub `main`, matching the unpinned `pywebpush>=2.0.0` in `server/requirements.txt`) — there is **no try/except anywhere around the internal `requests.post()` call** in `WebPusher.send()`. `WebPushException` is raised only from an already-received `Response` object (status > 202); if the POST itself fails at the transport layer (`ConnectionError`, `Timeout`, `SSLError`), that raw `requests` exception propagates straight out of `webpush()` uncaught.

That means: if subscription #1 in the `for endpoint, p256dh, auth in subs:` loop (api.py:359) succeeds (`any_sent = True`, line 375) and subscription #2 then raises a raw network exception, it skips past the `except WebPushException` (376) and is caught by the outer `except Exception` at line 398 — which logs via `logger.exception` and `continue`s to the next `(session_id, slot_id)` pair, **never reaching** the `if any_sent:` dedup INSERT at 387-392. Next scheduler tick (60s later, line loop at ~280s) re-evaluates the same slot, finds no `sent_notifications` row, and re-sends to subscription #1, which already received it. Reasoning and code citations both check out exactly as claimed.

However, the claim's own phrasing ("stale until the user reloads /line-up") already scopes the impact correctly: `scraper/render.py:2607-2619` runs an unconditional repair-on-load check on every lineup page visit (gated by the `stc_push` flag) that reads the browser's actual current subscription via `pushManager.getSubscription()` and re-POSTs it to the lineup endpoint.
