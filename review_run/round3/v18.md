## Verification Results

**Claim: A**
**Verdict: CONFIRMED**

Reasoning: `handle_chat_ws` reads `display_name`, `username`, `color_index`, `avatar_url` from the `users` row exactly once at handshake into closure locals (`server/chat_ws.py:1025-1030`). Every subsequent use inside the connection's `while True` loop reuses these same locals, never re-querying the DB:
- `join_room` presence/history flow passes the frozen vars straight into `manager.join_room(...)` (`chat_ws.py:1123-1132`).
- `send_message` passes the same frozen vars into `_moderate_and_broadcast(...)` (`chat_ws.py:1423-1442`), which is what actually broadcasts the message with sender identity to the room.

`PUT /chat/api/profile` (`server/chat_api.py:602-665`) only issues a `db.execute(f"UPDATE users SET {...}")` / `db.commit()` — there is no `from chat_ws import manager` in this handler and no call to any `manager.*` method (confirmed by enumerating every `manager.` call site in `chat_api.py`: none originate from the profile endpoint; the only identity-adjacent one is `key_rotated` at 1646/1659, unrelated to display fields).

The client reconnect path (`chat.html` `connectWS()`) re-sends `join_room` on `sock.onopen` but never refetches `/chat/api/me`, so even a client-side reconnect from the *same* browser doesn't pick up a fresh profile unless the WebSocket itself actually drops and a brand-new `handle_chat_ws` invocation re-reads the `users` row at handshake — which matches the claim's own caveat ("until that user's socket actually disconnects/reconnects"). Precondition (an open, idle-but-connected socket while another tab/device edits the profile) is trivially reachable since chat commonly stays connected across tabs/devices.

**Claim: B**
**Verdict: CONFIRMED**

Reasoning: `auth_delete_account` (`server/chat_api.py:494-516`) calls `delete_user_messages` then `delete_user(db, user["id"])` (which is just `DELETE FROM users WHERE id = ?` + commit, `chat_db.py:440-442`, relying on `ON DELETE CASCADE` for dependents) and clears the cookie — but never touches `manager.user_conns` or calls `ws.close()`. By contrast, `admin_delete_user` (`chat_api.py:2008-2038`) explicitly iterates `manager.user_conns.get(user_id, {})` and closes each with code 4003 at line 2031, and `admin_ban` (`chat_api.py:1739-1787`) does the identical close loop at line 1780. This is a genuine asymmetry, not an intentional design choice reflected elsewhere in the code or CLAUDE.md.

The failure mode is also confirmed mechanically: `messages.user_id` is `NOT NULL REFERENCES users(id) ON DELETE CASCADE` (`chat_db.py:123`), `foreign_keys=ON` is set on every connection (`chat_db.py:343`), and `create_message`/`join_room_membership` (`chat_db.py:763-807`, `632-639`) contain no try/except around their `db.execute`/`db.commit` calls — an `INSERT` referencing the now-nonexistent `user_id` raises an uncaught `sqlite3.IntegrityError`. That propagates out of the `while True` event loop and is caught by the bare `except Exception: logger.exception(...)` at the end of `handle_chat_ws`, after which `finally` runs `manager.disconnect(conn_id)` + `db.close()` — closing only that one connection, with no error surfaced to the client and no indication to the user that their own session is now broken. This precisely matches the claim.
