Both claims verified directly against code. Findings:

## Claim A — CONFIRMED (HIGH)

`server/chat_db.py:179-190` — `reports.reporter_id` and `reports.reported_user_id` are both `REFERENCES users(id) ON DELETE CASCADE`. `strikes.user_id` (`:192-199`) is likewise CASCADE. `bans.user_id` (`:82-90`) has **no FK constraint at all** — plain `TEXT`, confirming the asymmetry the claim describes.

`delete_user()` (`:440-442`) is a bare `DELETE FROM users WHERE id = ?` with `PRAGMA foreign_keys=ON` set at init (`:32`), so the CASCADE fires unconditionally on every call — no soft-delete, no archival step.

Reachability: `server/chat_api.py:2008-2038`, `DELETE /chat/api/admin/users/{user_id}`, gated by `_require_admin()`. This is a real, production admin action, not test-only code. Once fired, every `reports` row where the user was reporter *or* reported, and every `strikes` row, is gone — permanently erasing the evidence trail while the `bans` row (deliberately FK-less per the CLAUDE.md-documented design intent) survives with an orphaned `user_id`. The asymmetry is real and appears unintentional: bans were clearly designed to survive deletion; reports/strikes were not, despite serving the same "audit trail" purpose.

## Claim B — CONFIRMED (HIGH)

`get_user_admin_detail()` in `server/chat_db.py:1405-1411` selects `r.id, u.display_name AS reporter_name, r.reason, r.message_snapshot, r.status, r.created_at` — `r.unverified` is omitted. This feeds `GET /chat/api/admin/users/{user_id}` (`server/chat_api.py:1875-1890`), which returns the dict verbatim, so `unverified` never reaches the client for this endpoint.

`server/chat/admin.html:491-494` renders `u.reports_against` with only `esc(r.reason)`, `esc(r.message_snapshot)`, timestamp, and status — no `r.unverified` check, no banner.

Contrast confirmed: the pending-Reports tab (`admin.html:296`, backed by `/admin/reports` which does select `unverified`, `chat_api.py:~1698-1721`) correctly shows the `unverified-banner` ("Content provided by reporter — server cannot read encrypted messages"). The Users tab has no equivalent, so an admin browsing a user's report history sees E2EE-DM reporter-supplied plaintext presented with identical visual trust as a server-verified group-room report — exactly the mitigation the E2EE design (CLAUDE.md "End-to-End Encryption (DMs)") intended to prevent.

Both are real, reachable, unmitigated bugs — not test-only artifacts or guarded by something upstream.
