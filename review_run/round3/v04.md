## Verification Report

### Claim A — `</script>` breakout XSS in inline `TT_ARTISTS`
**Verdict: CONFIRMED (CRITICAL)**

- `_artists_json()` at `scraper/render.py:94-113` serializes artist `name`, `photo` (built from `photo_file`), and link `url` via `_json.dumps(...)` with zero escaping of `<`/`>`/`/`. Python's `json.dumps` only escapes `"`, `\`, and control characters — it never escapes `</`.
- `artist_lookup[artist_id]` is populated directly from that same unescaped structure at `render.py:1271-1273` (`_json.loads(_artists_json(group, photos_prefix))`).
- It's embedded raw into HTML at `render.py:1518`:
  ```python
  f"  <script>var TT_ARTISTS={_json.dumps(artist_lookup, separators=(',', ':'))};</script>"
  ```
  No `.replace("</", "<\\/")` or equivalent guard exists anywhere in this pipeline.
- Data provenance: artist `name` comes from raw scraped DOM text (`scraper/scrape.py:148-150`, `name_el.get_text(...)`, only whitespace-normalized) and link `url` from raw scraped `href` attributes (`scrape.py:152-167`) — both third-party-controlled (the festival's public lineup page / linked platforms), stored via parameterized-but-unsanitized INSERTs (`scraper/db.py`). `site_short` and `TT_SECTIONS`/`TT_DATES` (lines 1529, 1563-1570) are operator-controlled/derived from schedule dates, not attacker-reachable, so the concern there is theoretical only — correctly flagged as lower priority in the claim.
- This fires for **every visitor** who loads the timetable view (no click needed) — a crafted artist name containing `</script><script>...` on the real scraped site (or via a compromised/malicious upstream link source) breaks out of the tag and executes immediately on page load. This is a genuine stored, unauthenticated, zero-click XSS.

### Claim B — Unescaped innerHTML/outerHTML/attribute injection for artist name/photo/link
**Verdict: CONFIRMED (HIGH, arguably worse)**

Verified exact lines:
- `render.py:2111`: `photoEl.outerHTML = '<img ... src="' + data.photo + '" alt="' + data.name + '">'` — no `esc()`.
- `render.py:2118/2120`: `bioText.innerHTML = data.bio` — bio only passes through `_render_markdown()` (`markdown` lib + incomplete regex denylist for `<script|iframe|object|embed|form>` and `on\w+=` — not a real sanitizer like `bleach`/`nh3`), sourced from Resident Advisor's bio field (third-party API), not operator-authored.
- `render.py:2126/2129`: `esc(v.url)` / `esc(v.title)` — correctly escaped, confirming the omission elsewhere is inconsistency, not an intentional trust boundary.
- `render.py:2457-2466` (timetable popup): `bioClick` concatenates raw `a.oid`/`a.name` into an `onclick="..."` attribute string, only escaping single quotes (`a.name.replace(/'/g, "\\\\'")`) — a double quote in `a.name` breaks directly out of the `onclick="` attribute with no JS-string-escaping needed at all, since the whole string is parsed as HTML via `innerHTML` at line 2468. `a.photo`/`a.name` are also injected raw into `src`/`alt` (line 2459), and `l.u` raw into `href` via `_popupLink` (line 2422-2424, 2464).
- `esc()` is properly defined both server-side (`render.py:194-195`, `html.escape`) and client-side (`shared.js`, escapes `&<>"'`) and demonstrably used correctly two lines away in the same functions — confirming this is a real, exploitable gap rather than a deliberate trust decision.
- Same data provenance as Claim A: names/URLs are raw scraped third-party strings, reachable by any user who opens an artist's bio or timetable popup.

Note: severity could reasonably be argued up to CRITICAL rather than HIGH — the `onclick`/`alt`/`src` attribute-injection paths (2457-2466, 2111) execute JS directly via a simple `"` or `'`-based breakout without needing the `<script>`-tag trick, making them at least as reliably exploitable as Claim A.
