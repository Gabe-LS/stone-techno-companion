## Verification Results

**Claim A: Session token in WS URL → logged in plaintext**

Verdict: **CONFIRMED**

Reasoning:
- `server/chat/chat.html:1036` — `new WebSocket(\`${proto}://${location.host}/ws/chat/${token.split('=')[1]}\`)` puts the raw 7-day session token directly in the WS URL path.
- `server/chat_ws.py:1010-1016` — the handler takes that path param as `token` and calls `get_user_by_token(db, token)`, which (`chat_db.py:473-480`) does `SELECT ... FROM sessions WHERE token = ? AND expires_at > ?` — this is the live session cookie value, not a nonce; possession = full session hijack.
- `chat_db.py:461-470` — `create_session` sets `expires_at = now + timedelta(days=7)`, confirming the 7-day exposure window.
- `server/Dockerfile:10` — `CMD ["uvicorn", ..., "--proxy-headers", "--forwarded-allow-ips", "*"]` has no `--no-access-log` or `--log-level` override, so uvicorn's default access-log middleware is active and logs the request line (method + full path/target) for both HTTP and WS upgrade requests to container stdout.
- `server/docker-compose.yml:19-23` — only configures `json-file` rotation (`max-size: 10m`, `max-file: 5`); nothing suppresses or redacts access logging.
- No guard anywhere strips the token before logging, and reconnect logic (`chat.html:1050-1058`) re-triggers `connectWS()` — and thus a fresh log line with the token — on every reconnect/backoff cycle, not just initial login.

No mitigating control found. Anyone with read access to container/VPS logs can lift a live token and hijack the session for up to 7 days.

**Claim B: Magic-link verify token in URL → logged; no rate limit**

Verdict: **CONFIRMED**

Reasoning:
- `chat_api.py:450-477` (`GET /chat/api/verify`) reads `token` as a query parameter and `chat_api.py:2183-2185` (`GET /chat/v/{token}`) exposes the same value as a path segment — both forms place the single-use secret in the request target, which uvicorn's default access-log format includes verbatim (method + full request-line, including query string).
- Token lifecycle in the same function: looked up by exact match (`SELECT ... FROM email_tokens WHERE token = ?`), then `DELETE FROM email_tokens WHERE token = ?` at line ~460 — single-use is real, deletion happens on first hit regardless of who performs that hit.
- Rate limiting exists only on `POST /login` (`chat_api.py:369-376`, in-memory `_email_rate` dict, 5/900s per IP) — the verify GET endpoints have no equivalent limiter, no CAPTCHA, and are unauthenticated by design (that's the point of a magic link), so nothing throttles or flags rapid/replayed hits.
- 15-minute token expiry narrows the exploitation window but doesn't close it: an attacker with real-time log read access (e.g. `docker logs -f` or a log-shipping pipeline) can extract the token the instant the email is sent and complete `/chat/v/{token}` (or `/verify?token=`) before the legitimate user clicks, since the endpoint has no rate limit and the token isn't bound to a client fingerprint/IP at redemption time (only `fingerprint` from token creation is stored/passed through, not re-checked against the request).

No mitigating control found (no rate limit, no logging suppression, no request-binding check at redemption). Both attack surfaces (`/verify?token=`, `/chat/v/{token}`) are exploitable identically.
