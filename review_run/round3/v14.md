## Claim A: Unbounded request body buffered before size check

**Verdict: CONFIRMED**

**Reasoning:**

Verified all three handlers in `server/chat_api.py` read the full body before any size check:

- `upload_avatar` (chat_api.py:1128-1138): rate check at 1132, then `data = await file.read()` at 1136, size check `len(data) > 500*1024` at 1137 — **after** the read.
- `upload_image` (chat_api.py:1224-1235): rate check at 1228, `data = await file.read()` at 1233, size check at 1234 — **after** the read.
- `upload_video` (chat_api.py:1321-1332): rate check at 1325, `data = await file.read()` at 1330, size check at 1331 — **after** the read.

`_check_upload_rate` (chat_api.py:92-98) only throttles call *count* (10/60s per user, in-memory dict, no lock) — it does nothing about body size or concurrency. Because it's a synchronous function called before the first `await`, and requests interleave at `await file.read()`, a single authenticated user can have up to 10 uploads in flight concurrently within the window, each streaming an arbitrarily large multipart body — the client-declared `Content-Type`/size limits are never enforced server-side until after full buffering.

I confirmed via a sub-agent search that `server/api.py` registers **no middleware** (no `add_middleware`, `BaseHTTPMiddleware`, body-size checks, or `Content-Length` validation anywhere), and there is **no Caddyfile in this repo** — Caddy's actual config lives outside the repo (on the VPS) and isn't verifiable from source, but nothing in-repo provides a body-size backstop. `server/Dockerfile:9-10` only notes Caddy is the ingress and sets `--proxy-headers`; no size cap. Starlette's default multipart parser also has no overall body-size ceiling (only a per-part spool threshold before it starts writing to disk), so a large body is fully buffered — to disk via spooling, then again into memory via `.read()` — before the app's `len(data)` check ever runs.

This requires an authenticated chat account (trivial to obtain via magic-link/Google auth), so it's not fully anonymous, but the precondition is easily reachable and the described attack (concurrent multi-GB bodies) is real and unmitigated. Confirmed as HIGH.

## Claim B: No memory/CPU limits on the app container

**Verdict: CONFIRMED**

**Reasoning:**

`server/docker-compose.yml:1-27` defines the `stone-techno` service with `build`, `container_name`, `restart`, `env_file`, `volumes`, `networks`, `healthcheck`, and `logging` — no `mem_limit`, `cpus`, `mem_reservation`, or `deploy.resources.limits` block anywhere. Without an explicit cgroup limit, the container can consume all host memory/CPU.

This directly compounds Claim A: video upload (chat_api.py:1321-1476) spawns `ffprobe`/`ffmpeg` subprocesses (via `asyncio.to_thread(subprocess.run, ...)`, lines ~1353-1368 and ~1407-1425) plus pyvips frame conversion, and image upload decodes/resizes via pyvips (chat_api.py:1244-1305) — all running in-process/in-subprocess with no memory ceiling from Docker. Combined with the read-before-check gap in Claim A and the 10-concurrent-uploads-per-user allowance, a burst of large concurrent uploads has no infrastructure-level backstop. Confirmed as HIGH, consistent with the claim.
