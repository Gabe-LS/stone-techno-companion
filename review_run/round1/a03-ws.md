## WebSocket Server Review — `server/chat_ws.py` / `server/chat_db.py`

**[CRITICAL] chat_ws.py:1361-1370, 1683-1702 (cross-referenced with chat_db.py:820, 810)** — Arbitrary file deletion via unvalidated `media_url` on E2EE messages.
For non-E2EE image/video messages, `send_message` validates the media URL against `_UPLOAD_URL_RE` (chat_ws.py:1262-1276, `^/chat/uploads/[a-f0-9]{32}\.(webp|mp4)$`). That check is gated on `not is_e2ee_msg`, so for E2EE messages the client-supplied top-level `media_url` field (line 1363) is stored to `messages.media_url` with **no validation at all**. The `delete_message` handler (lines 1683-1702) later takes `msg_row["media_url"]` verbatim, does `url.rsplit("/",1)[-1]` and `unlink()`s it inside `chat/uploads/`.
Reproduction: attacker views any image/video already posted in any room (non-E2EE image URLs are visible client-side in `<img src>`), learns its `/chat/uploads/<hash>.webp` filename, opens/uses a DM, sends an E2EE "image" message whose `media_url` is set to that other user's filename, then deletes their own message within the 120s window (age check line 1676-1681) — the server unlinks the *other user's* real file. Impact: any authenticated user can destroy arbitrary media belonging to any other user/room.

**[CRITICAL] chat_ws.py:141-269 (`_is_safe_preview_url`, `_fetch_og_preview`)** — SSRF via DNS rebinding (TOCTOU) in the link-preview fetcher.
`_is_safe_preview_url` resolves the hostname once via `socket.getaddrinfo` and rejects private/loopback/link-local/reserved IPs, but the actual request in `_fetch_og_preview`/redirect-target requests (lines 217-228) is made with `client.get(url, ...)`, which performs its **own independent DNS resolution** at connect time. A malicious link-preview target (any URL a chat user pastes into a message, auto-triggered at line 881-898) can answer the validation lookup with a public IP and the real connection with `127.0.0.1`, an internal service, or the cloud metadata address — completely bypassing the blocklist. Impact: SSRF into internal network/metadata services, triggerable by any chat user simply posting a URL.

**[HIGH] chat_db.py:886-920 (`purge_expired_messages`) and chat_db.py:1256-1287 (`delete_user_messages`)** — E2EE media files are never cleaned up by TTL purge or ban/mute mass-delete.
Both functions `SELECT ... content` only and derive the file path via `json.loads(content).get("url")`. For E2EE image/video messages the plaintext URL lives exclusively in the `media_url` column (the `content` field is an opaque `{e2ee:true,...}` envelope with no `url` key), so `content.get("url")` is always absent and no `unlink()` ever runs. This contradicts the invariant documented in CLAUDE.md ("both the TTL purge and manual delete read that column to unlink the served file") — only the WS `delete_message` handler (chat_ws.py:1683-1692) actually reads `media_url`. Impact: every E2EE image/video DM that expires via TTL, or whose sender gets muted/banned, permanently orphans its file(s) on disk — unbounded disk growth.

**[HIGH] chat_ws.py:1444-1457 (`typing`), 1601-1665 (`add_reaction`/`remove_reaction`), 580-643 (`join_room`/`leave_room` presence)** — Broadcast storm: several room-fanout events have no rate limiting.
`check_rate_limit` (chat_ws.py:1334, 1460) is only invoked for `send_message` and `create_meetup`. `typing`, `add_reaction`, `remove_reaction`, `join_room`, and `leave_room` all call `ChatRoom.broadcast`/`broadcast_to_room` (O(room member count) per call) with zero throttling. Reproduction: a single authenticated connection sends `{"event":"typing","room_id":...,"active":true}` (or repeatedly join/leave) as fast as the socket allows — each one fans out to every connected member of a room with no per-user cap. Impact: trivial CPU/bandwidth amplification DoS against large rooms (e.g. the main event room).

**[MEDIUM] chat_ws.py:1423-1442 (`_moderate_and_broadcast` scheduling)** — No message-ordering guarantee for recipients.
Each `send_message` spawns an independent `asyncio.create_task(_moderate_and_broadcast(...))`; the room broadcast only happens after `await moderate_message(...)` completes inside that task. Moderation latency varies per message (image/video frame extraction + AI calls vs. plain word-filter-only text), so two messages sent back-to-back by the same user can have their `message` broadcasts delivered to other room members out of send order, even though `created_at` in the DB is correctly ordered. Reproduction: send an image message immediately followed by a short text message in the same room — the text (fast moderation) can broadcast before the image (slow moderation) finishes.

**[MEDIUM] chat_ws.py:1171-1195 vs. 917-939** — Race between `mark_read` and a concurrent incoming message's badge update.
`mark_read` pops the in-memory unread count and then `await`s `send_to_user(... "count": 0 ...)` (lines 1179-1188). If a message lands in that same room for that user at nearly the same time, the message-broadcast task's badge loop (lines 923-939) also `await`s a `send_to_user` with the incremented count. The two independently-scheduled sends to the same WebSocket can be delivered in either order — if the "new message" badge_update is written to the socket before the `mark_read` badge_update, the client ends up showing `count: 0` for a room that actually has an unread message, until the next full resync (reconnect/`badge_counts`).

**[MEDIUM] chat_ws.py:1038-1083** — Reconnect can silently drop a concurrently-arriving badge increment.
On every new connection for a user (not just the first), `manager.user_unread[user_id]` is reset to `{}` (line 1050), then repopulated from a `get_unread_counts` DB snapshot taken at line 1064, with an `await manager.send_to_user(...)` (`badge_counts`, lines 1066-1081) in between the snapshot and the repopulate loop (lines 1082-1083). If a new message for this user arrives in that window, the message-broadcast task increments `mgr.user_unread[uid][room_id]` (lines 926-927) using data that is then unconditionally overwritten by the pre-await DB snapshot once the repopulate loop runs — the increment for that just-arrived message is lost from in-memory tracking (though the DB-backed `get_unread_counts` will reflect it correctly on the *next* fetch).

**[LOW] chat_ws.py:1087-1091** — No application-level cap on incoming WebSocket frame size before `json.loads(raw)`; size limits (if any) come entirely from the ASGI/`websockets` library defaults rather than an explicit check, inconsistent with the deliberate `max_content` enforcement applied later for parsed message content (lines 1239-1260).
