## Review Findings — Scraper & Static Site Generation

**[CRITICAL] scraper/render.py:1518 (root cause: scraper/render.py:94-113, `_artists_json`)** — Artist name, photo path, and social URLs are JSON-serialized via `_json.dumps(artist_lookup, ...)` and embedded raw inside an inline `<script>var TT_ARTISTS=...;</script>` tag with no `</script>` escaping. `json.dumps` never escapes the `/` in `</`, so an artist name (or override URL) containing the literal string `</script><script>...` breaks out of the tag and executes arbitrary JS for every visitor who loads the timetable view — no click required. Fix: replace `</` with `<\/` in the dumped string before embedding (same treatment needed anywhere else JSON is inlined into a `<script>` block, e.g. `site_short` at line 1529 and `sections_json`/`TT_DATES` at lines 1563-1570, though those are lower risk since they aren't scrape-controlled).

**[HIGH] scraper/render.py:2111, 2457, 2459, 2464/2422-2423, 2466** — Client-side JS builds HTML via string concatenation with `data.name`/`a.name`/`a.photo`/`l.u` (bio modal photo `alt`, timetable popup photo `alt` + `onclick`, popup name text, and `_popupLink(href, ...)` inserting `href` into an `<a href="...">` unescaped) and assigns the result to `.innerHTML`/`.outerHTML`. The `esc()` helper is imported from `/shared.js` and is already used two lines away for `v.url`/`v.title` (lines 2126, 2129) but is skipped for artist name/photo/link URL. Since artist names and RA/SoundCloud/Instagram profile URLs are scraped from third-party pages (or set via `overrides.toml`), a crafted display name (e.g. `"><img src=x onerror=alert(1)>`) reflects as DOM XSS the moment a user opens that artist's timetable popup or bio. Fix: wrap all four/five insertion points with `esc(...)`.

**[HIGH] scraper/scrape.py:187 (crashes via scraper/db.py:212-230, `upsert_lineup`)** — In `scrape_lineup`'s assignment loop, `overlay_id = (li.get("data-overlay-identifier") or "").strip()` has no `if not overlay_id: continue` guard (unlike the equivalent overlay-detail loop at line ~144-146). Any lineup list item matched by the assignments selector (`li.tab-list__list-item.lineup-name[data-timestamp]`) that lacks a `data-overlay-identifier`, or whose ID has no matching entry in `div.overlay.line-up-overlay[data-overlay-identifier]` (e.g. a "TBA"/placeholder slot with no detail panel yet), produces a `schedule` row referencing an `artist_id` with no corresponding `artists` row. `INSERT OR IGNORE` does **not** suppress foreign-key violations in SQLite (`ON CONFLICT` handling is documented to exclude FK constraints) — this raises `sqlite3.IntegrityError`, uncaught, aborting the entire scrape run with no HTML/JSON regenerated. Fix: skip assignments whose `overlay_id` is empty or not present in `parsed["artists"]`.

**[MEDIUM] scraper/scrape.py:639-729 (`fetch_ra_profile` / `fetch_all_ra`)** — Unlike `fetch_sc_profile`, `fetch_ig_profile`, and `fetch_spotify_listeners`, which each wrap their Playwright/network calls in `try/except Exception` and degrade gracefully, `fetch_ra_profile`'s `_ra_gql` calls (page.evaluate + fetch) and the initial `page.goto("https://ra.co", ...)` at line 693 have no exception handling. A single network timeout or ra.co outage raises uncaught, aborting `fetch_all_ra` and losing progress on every remaining artist in that batch (though already-committed rows persist via per-call `db.commit()`). Fix: wrap the RA fetch loop/goto the same way the other three fetchers are.

**[MEDIUM] scraper/db.py:315 (`apply_overrides`, floor_curators)** — `date, stage_id = key.split(".", 1)` has no guard for a malformed key without a `.` separator; `ValueError` propagates uncaught and aborts `apply_overrides` (and thus the whole run, including `--render-only`). A single typo'd `[floor_curators]` key in `overrides.toml` (e.g. missing the date prefix) takes down the entire regeneration. Fix: validate/skip malformed keys with a printed warning instead of letting `split` raise.

**[MEDIUM] scraper/render.py:156-178 (`_render_markdown`)** — Bio sanitization is regex-based (strip `script/iframe/object/embed/form` tags, `on\w+=` attributes, `javascript:` hrefs) rather than an allowlist HTML sanitizer. Bio text originates from RA scraping (an artist's own RA profile content, which the artist or an impersonator controls) and from `overrides.toml`. The current regexes catch the common vectors, but regex-based stripping of arbitrary nested/malformed HTML is inherently bypassable (e.g. unclosed tags interacting with markdown's own HTML passthrough, attribute values using unusual quoting) and any future edge case silently reintroduces stored XSS into `bios.json`. Fix: sanitize with a real allowlist library (e.g. `bleach`) instead of hand-rolled regexes.

**[LOW] scraper/images.py:26-38 (`resize_and_crop`)** — `pyvips.Image.new_from_file(raw_path)` has no pixel-dimension cap before decoding. The 20MB byte-size cap in `process_artist_photos` (images.py:117-119) doesn't bound decoded pixel count — a crafted/corrupted image well under 20MB compressed can still decode to a very large bitmap (decompression-bomb pattern), causing a memory spike before the per-photo `except Exception` at line 126 can catch anything. Low severity because photo URLs are scrape/override-sourced, not directly attacker-submitted, but worth a `pyvips` field-based size check (`im.width * im.height`) before resize if festival photo sources are ever less trusted.

**[LOW] scraper/db.py:212-234 (`upsert_lineup`)** — The staleness warning only fires when `parsed["artists"]` is non-empty but `parsed["assignments"]` is empty (line 231-234). If the site's overlay-detail selector breaks entirely (e.g. markup restructure) such that `parsed["artists"]` is also empty, no warning is printed at all — the run silently keeps serving old data indefinitely with no operator signal that scraping has stopped working. Fix: warn whenever the scrape yields zero artists overall, not only in the assignments-empty case.
