## Findings

**[HIGH] server/chat_ws.py:1423 (also :328, :959) — untracked background tasks lose messages/moderation on every deploy**
`asyncio.create_task(_moderate_and_broadcast(...))` fires after a message is already committed to `messages` and ack'd to the sender (`chat_ws.py:1384-1397`), but the task handle is discarded — nothing in `api.py`'s `lifespan` shutdown (`api.py:446-476`) or anywhere else awaits it. `deploy.sh:107` runs `docker compose up -d --build --force-recreate` on every deploy, which sends SIGTERM (and SIGKILLs after Docker's default 10s grace period — no `stop_grace_period` override in `docker-compose.yml`) to the running container. Any message whose moderation task is killed mid-flight is left permanently un-moderated and un-broadcast in the DB (the delete-on-fail path never runs), and it will be silently served to other users via `room_history` on next `join_room` (`chat_ws.py:1148`) — i.e., an actual moderation bypass, not just a UX glitch, and it reproduces on every deploy under load. Fix: track spawned tasks in a set, and in `mount_chat`'s purge-loop shutdown path (or a chat-specific lifespan hook) await/cancel them with a timeout before the process exits; also raise/verify the container's stop grace period.

**[HIGH] server/docker-compose.yml — no memory or CPU limits on the only application container**
No `mem_limit`/`cpus` (or `deploy.resources.limits`) is set. Video upload (`chat_api.py:1321-1476`, up to 100MB) triggers `ffprobe`/`ffmpeg` subprocesses plus `pyvips` decode of full frames, and image upload (`chat_api.py:1224-1319`) decodes up to 40M pixels via `pyvips` — all in the app's own process/subprocess pool with no ceiling. A burst of concurrent uploads (stress_test drives up to 10 concurrent image uploads) or a crafted large payload can exhaust host memory/CPU with nothing to stop it, potentially taking down Caddy and other services sharing the VPS. Fix: add `mem_limit`/`cpus` (or `deploy.resources.limits` if using `docker compose --compatibility`) to the service.

**[HIGH] deploy.sh:109-143 — health-check failure does not fail the script or trigger rollback**
After Step 5's health/chat checks (`deploy.sh:112-131`), a non-"healthy" container status or failed chat API probe only prints a warning — the script has no `exit 1` on that path and continues straight to backup cleanup and "Deploy complete" (`deploy.sh:137-143`), exiting 0. A broken deploy (bad code, crash-looping container) can go unnoticed unless someone reads the console output closely, and there is no automated rollback (no re-deploy of previous image/commit) even when the failure is detected. Fix: `exit 1` (or trigger an automatic rollback to the previous image/commit) when health/chat checks fail.

**[MEDIUM] deploy.sh:39-79 — production `.env` is unconditionally overwritten from local dev `.env` with only a non-empty check**
Step 0 reads `PROD_VARS` straight from `server/.env` on the operator's machine and `cat`s a freshly built file over the VPS's `server/.env` on every run (`deploy.sh:77`). Validation is only "is this var non-empty" (`deploy.sh:46-56`) — there's no check that e.g. `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET` are actually the production-scoped credentials rather than a local-dev OAuth client, and no diff/confirmation shown before overwriting. A stale or dev-scoped local `.env` silently clobbers working production secrets. Fix: show a diff of changed vars (or require `--yes`) before overwriting, or keep prod secrets out of the sync path entirely (fetch-then-merge instead of full overwrite).

**[LOW] server/docker-compose.yml:15 — healthcheck target doesn't reflect actual first-deploy state**
`curl -f http://localhost:8080/` hits the catch-all route (`api.py:1027-1036`), which 404s until `output/index.html` exists under `./static`. Content is deployed via a separate step (`stone_techno_companion.py --render-only --deploy`), not `deploy.sh`. On a fresh VPS or after a wiped `./static` volume, the container reports "unhealthy" indefinitely even though the process is running fine — purely cosmetic today (no `restart: on-unhealthy`/autoheal is configured) but worth knowing before it's wired into any alerting.

**[LOW] deploy.sh:81-97 — backup only covers `server/data/`, not `chat-uploads/`**
The VPS-side and local backups (`deploy.sh:84-97`) snapshot `server/data/` only; `docker-compose.yml:11` also mounts `./chat-uploads:/app/chat/uploads`, which is never backed up. If the VPS disk fails, any media attached to messages still inside their TTL window (up to 24h for rooms/DMs) is unrecoverable. Given the app's ephemeral-by-design TTL model this is a small window, but it's an easy one-line addition to the existing rsync step if the gap isn't intentional.

**[LOW] server/chat_api.py:1545-1552 — `/chat/api/swlog` logs raw client-supplied JSON, unauthenticated**
This diagnostic endpoint (explicitly marked "Temporary" in the comment) accepts any POST body and logs up to 500 chars of it verbatim via `logger.info`, with no auth check and no redaction — unlike every other push-related log line in the codebase, which truncates endpoints to `[:60]`/`[-16:]`. If it's still needed, at minimum truncate/redact endpoint-shaped fields before logging; if the July 2026 iOS click investigation it was added for is closed, remove it.
