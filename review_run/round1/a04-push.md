## Push Notification Review Findings

**[HIGH] server/api.py:335–402 — Scheduler dedup can be silently skipped by a partial-loop exception, causing duplicate lineup pushes.**
The `for endpoint, p256dh, auth in subs:` loop only catches `WebPushException` (line 376). `pywebpush.webpush()` calls `requests.post()` internally and does **not** wrap raw network failures (timeouts, `ConnectionError`, SSL errors) in `WebPushException` — those propagate past line 376 straight to the outer `except Exception` at line 398, which aborts the whole `(session_id, slot_id)` block *before* the `if any_sent: INSERT INTO sent_notifications` at lines 387–392 runs. If subscription #1 already succeeded (`any_sent = True`) and subscription #2 then hits a transient network error, the dedup row is never written — the next 60s scheduler tick re-sends the "starts in 10 min" push to subscription #1 again. Affects any browser; most likely triggered by transient network blips to FCM/Mozilla/Apple push endpoints. Fix: wrap each `webpush()` call in `except Exception` (not just `WebPushException`) so one endpoint's failure can't erase a sibling endpoint's success, or move the dedup INSERT to run per-successful-endpoint rather than per-session/slot.

**[HIGH] server/static/sw.js:95–112 — `pushsubscriptionchange` repairs only the chat-side subscription; the lineup-side record has no repair path in the SW.**
The handler POSTs the new subscription only to `/chat/api/push/subscribe`. It has no way to also hit `/api/session/{code}/push/subscribe` because the SW has no access to the lineup page's session code. Per CLAUDE.md, lineup and chat share one browser subscription and `pushsubscriptionchange` is exactly the event that can rotate it (Brave "forget me" revocation, etc.). If the browser fires this event while the user has only chat open (never reopens `/line-up`), lineup's stored subscription in `push_subscriptions` (hearts.db) is permanently stale — pointing at a dead endpoint — until the user happens to load the lineup page again and its client-side resync runs. No timebound, no alerting. Affects any browser subject to subscription rotation (documented as more likely on Brave). Fix: add a generic server-side reconciliation (e.g. a route that accepts old+new endpoint and updates both `push_subscriptions` and `chat_push_subscriptions` regardless of which surface calls it), invoked from the SW's `pushsubscriptionchange` handler.

**[MEDIUM] server/static/sw.js:62–89 — `notificationclick` targets `list[0]` with no URL-based matching; wrong tab can be focused/navigated when both surfaces are open.**
`self.clients.matchAll({type: 'window', includeUncontrolled: true})` returns all same-origin window clients — lineup and chat tabs alike, since both register the SW at root scope. `list[0]` is used unconditionally for `postMessage` + `focus()` (lines 75–78) with no check against `event.notification.data.url` (e.g. preferring a `/chat*` client for a chat push). If a user has both a lineup tab and a chat tab open, a chat push notification could focus/postMessage the lineup tab instead, leaving the chat tab in the background with no navigation applied (it only self-heals once the user *manually* switches to it and the `visibilitychange`/`focus`/`pageshow` cache-poll fallback fires). Affects any multi-tab desktop/PWA usage. Fix: filter `list` by URL prefix matching the target route before falling back to `list[0]` or `openWindow`.

**[MEDIUM] server/api.py:277–306 — No backfill for a missed scheduler tick; a delayed loop permanently skips due notifications.**
The due-slot check is a fixed `[now+9:30, now+10:30]` window evaluated once every 60s tied to wall-clock time, with no query for "slot windows that already elapsed since the last successful check." If a tick is delayed beyond ~60s (slow DB query, container restart, an unhandled hang recovering only on the next `sleep(60)`), any slot whose window fell entirely inside the gap is never picked up by a later tick and, since `sent_notifications` only records what *was* sent, is never retried — affected users silently get no "10 min" reminder for that set. Fix: widen the catch-up window (e.g. also check slots whose window has already opened but has no `sent_notifications` row and hasn't fully closed by more than one tick), or log a diagnosable warning when a due slot is discovered outside its intended window.

**[MEDIUM] server/api.py:376–385 — Dead-subscription pruning covers only HTTP 404/410; other permanent failures accumulate forever.**
Only `WebPushException` with `response.status_code in (404, 410)` triggers `DELETE FROM push_subscriptions`. A permanently failing subscription that returns e.g. 400 (malformed) or 403 (exactly the failure mode documented for VAPID key mismatches in `_check_vapid_key_consistency`) is retried on every scheduler tick that matches a due slot for that session, forever, with no periodic staleness sweep — unlike chat's `purge_stale_push_subscriptions` (chat_db.py:1541–1554), lineup's `push_subscriptions` table has no equivalent. Fix: add a periodic prune (age + failure-count based) for lineup subscriptions mirroring the chat implementation.

**[LOW] server/api.py:824–844 — `push_unsubscribe` has no rate limiting, unlike every other mutating endpoint (`push_subscribe` at line 798 calls `_check_rate(..., "pick")`).**
Low impact (scoped to a resolved session/endpoint), but inconsistent with the rest of the API's rate-limiting posture.

**[LOW] server/static/sw.js:97–111 — `pushsubscriptionchange` resubscribe failure is silently swallowed (`.catch(function(){})`) with no `swlog()` call.**
Every other code path in this file (push received, click, dismiss) logs via `swlog()`/`ackPush()`; this is the one push-lifecycle event with zero server-side visibility if `pushManager.subscribe()` rejects (e.g. OS-level permission revoked). Makes a real production failure ("why did push silently stop after a rotation") undiagnosable from logs.

**[LOW] server/chat_ws.py:1189–1194 vs 349–359 — `mark_read` can only cancel a not-yet-started deferred push flush; a flush already past its `asyncio.sleep` and mid-`webpush()` call is unstoppable.**
`task.cancel()` on `_push_flush_tasks[key]` works cleanly while the task is still in `await asyncio.sleep(delay)`, but if cancellation lands after the task has begun `_do_send_push`'s `asyncio.to_thread(webpush, ...)`, cancellation won't recall the already-dispatched network call — a push with a now-stale unread count can still arrive on another device just after the user read the message elsewhere. Narrow window, cosmetic impact (stale badge count in the notification body), not a missed/duplicate-message bug.
