# Moderation System Review — Bypass Vectors & Correctness

## Word filter (chat_moderation.py)

**[HIGH] chat_moderation.py:116-132 (`WordFilter.check`)** — No substring/embedded-word matching. `check()` tokenizes on whitespace and only matches when a normalized *whole word* equals a blocklist term exactly (`words[i:i+len(term_words)] == term_words`). Removing spaces around/inside a flagged term completely defeats it — e.g. `"buymdmanow"` or `"areyouanigger"` glued into one token never matches `"mdma"`/`"nigger"` because neither is a standalone token. Note the inconsistency: `check_username` (line 143-147) *does* have a substring fallback, but the message-content path (`check()`, the one that actually gates chat messages) has none at all. This is the highest-value target since it's the instant, pre-AI gate. Fix: apply the same (or better) substring scan used in `check_username` to `check()`.

**[MEDIUM] chat_moderation.py:134-148 (`WordFilter.check_username`)** — The substring fallback only fires for terms with `len(term) >= 5` (`>=3` only for drug terms, line 145). Many blocklist entries are 3-4 chars (e.g. `nig`, `wop`, `wog`, `kkk`, `cok` — see `chat/blocklist.txt`), so a username like `"bignigfan"` bypasses the filter entirely: the whole part (`"bignigfan"`) doesn't equal `"nig"`, and `"nig"` is too short (3 < 5) to trigger the substring check. Fix: lower or remove the length gate, at minimum for slur-class terms.

**[MEDIUM] chat_moderation.py:76-88 (`_normalize`)** — `CHAR_SUBSTITUTIONS` (lines 17-28) only maps ASCII leetspeak (`@,0,1,3,$,5,!,4,7,+`); it has no mapping for Unicode letter homoglyphs (Cyrillic а/е/о/р/с/х, Greek, fullwidth Latin, etc.). The final sanitizer `re.sub(r"[^a-z0-9\s]", "", text)` (line 86) *deletes* any such character outright rather than transliterating it. So a single Cyrillic look-alike inserted into a blocklisted word (e.g. Cyrillic `а` in `"аsshole"`) mangles it into `"sshole"`, which matches nothing — while remaining visually near-identical to a human reader. Combined with the lack of substring matching above, this is a repeatable, low-effort evasion of the instant filter (the AI layers, which see the raw un-normalized text, are the only backstop).

**[MEDIUM] chat_moderation.py:81-88 (`_normalize`)** — No run-length/repeated-character collapsing. Classic evasions like `"fuuuuck"` or `"shiiiit"` never equal the canonical blocklist token, so they pass the instant filter untouched (again relying on the AI layers as the only real defense).

## AI moderation failure handling (chat_moderation.py)

**[HIGH] chat_moderation.py:460-475 (`moderate_message`)** — Partial AI-layer failure is not fail-closed. The block-on-failure branch only triggers when **both** layers raise (`if ai_errored and drug_errored and os.environ.get("OPENAI_API_KEY")`, line 469). `check_openai_moderation` (sexual/violence/hate/self-harm coverage) and `check_content_detection` (drugs/spam/payment/external-link coverage) cover disjoint category sets. If only the omni-moderation call 5xx's/times out (a real-world OpenAI outage or rate-limit scenario) while the drug/spam layer succeeds and doesn't flag the message (because the content — e.g. genuinely violent or hateful text — is outside that layer's scope), the message falls through to `return {"allowed": True}` (line 519) with no operator-visible signal beyond a `logger.error` line. Fix: treat any single AI-layer failure (when `OPENAI_API_KEY` is configured) as fail-closed, not just simultaneous failure of both.

## Optimistic delivery race (chat_ws.py / chat_db.py)

**[HIGH] chat_db.py:763-807 + chat_db.py:810-824 + chat_ws.py:1148-1164** — The live broadcast to other room members is correctly gated on moderation completion (`chat_ws.py:862-878`, only reached after `mod_result["allowed"]` is true), so real-time viewers never see rejected content. However, `create_message()` persists the row **synchronously**, before the async moderation task (`asyncio.create_task` at `chat_ws.py:1423`) even starts, and `get_room_messages()` (used to build `room_history` on every `join_room` event, `chat_ws.py:1148-1164`) has no moderation-status column to filter on — it returns every non-expired message regardless of moderation state. Any user who (re)joins/reconnects to the room during the moderation window (two sequential/parallel network calls, `httpx` `timeout=5.0` each) will see the message via `room_history` even though it was never broadcast live. If moderation subsequently rejects it, `message_removed` is sent **only to the sender** (`chat_ws.py:814-822`), never broadcast to the room, so any member who already pulled the flagged message into their `room_history` has no signal it was retracted. Fix: filter `get_room_messages` to exclude very-recent, not-yet-cleared messages, or broadcast `message_removed`/`messages_expired` to the whole room instead of just the sender.

## Video moderation coverage (server/chat_api.py, `upload_video`)

**[MEDIUM] server/chat_api.py:1371-1404** — `duration = float(info["format"].get("duration", 0))` defaults to `0` when ffprobe can't determine duration, and that same value later drives the "3 frames at 25/50/75%" sampling: `seek = duration * frac if duration > 0 else 0` (~line 1402). When duration is unresolvable, all three sample points collapse to `seek=0`, so the "3 representative frames" become three copies of the same first frame — content appearing mid/late in the video is never sampled by omni-moderation, and the upload isn't rejected (fails open; only rejected if *all three* frame extractions fail, ~line 1448-1453). Fix: reject (or force manual review) uploads where duration can't be determined rather than silently sampling t=0 three times.

## Ban evasion via linked providers (chat_moderation.py)

**[HIGH] chat_moderation.py:354-363, 365-377, 482-489** — All three auto-ban call sites (4-strike ban, mute-cycling ban, instant AI-moderation ban) call `ban_user(db, user_id, user["provider"], user["provider_id"], reason, user["device_fingerprint"])` using only the single frozen identity on the `users` row. This is narrower than the admin-initiated ban path (`server/chat_api.py:1739-1787`, `admin_ban`), which explicitly queries `user_providers` and inserts a ban row for **every** linked `(provider, provider_id)` pair — the exact hardening CLAUDE.md documents as closing the "second provider can't evade it" gap. That hardening was never factored into a shared helper, so it doesn't apply to auto-bans. Since `is_banned()` (chat_db.py:506-523) matches on `(provider, provider_id)` or `device_fingerprint`, a user who has linked two providers (e.g. Google + Email) and gets auto-banned via strikes/AI moderation on one provider can immediately re-authenticate through the other linked provider on a different device — the ban row for that second identity was never created. Fix: factor the multi-provider fan-out from `admin_ban` into `chat_db.ban_user` (or a shared wrapper) and use it from all four ban call sites.
