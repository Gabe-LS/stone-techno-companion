[CRITICAL] server/chat_api.py:1494-1508 (chat_push_subscribe), server/chat_db.py:1589-1599 (save_push_subscription), server/api.py:794-821 (push_subscribe) — Push subscription `endpoint` field is accepted from the client with only a non-empty check, no scheme/hostname allowlist (no restriction to fcm.googleapis.com, updates.push.services.mozilla.com, web.push.apple.com, etc.). This endpoint is later passed unmodified as the `subscription_info["endpoint"]` target of an outbound `pywebpush.webpush()` POST in `server/chat_ws.py:463-479` and `server/api.py:359-374`. Any authenticated chat user (or anyone hitting the unauthenticated lineup session endpoint at api.py:794) can register `endpoint` = an arbitrary internal or attacker-controlled URL (e.g. `http://169.254.169.254/...`, an internal service, or an attacker's own listener), and the server will make VAPID-signed outbound HTTP POST requests to it whenever a message/notification fires — a genuine SSRF primitive independent of the already-reported link-preview SSRF. Fix: validate `endpoint` starts with `https://` and its hostname matches an allowlist of known push-service domains before storing.

[HIGH] server/chat_ws.py:465-479 and server/api.py:361-374 — `webpush(...)` is invoked via `asyncio.to_thread` with no `timeout=` argument; pywebpush's underlying `requests.post` call defaults to no timeout (blocks indefinitely) when none is passed. Combined with the endpoint SSRF above, an attacker who registers a subscription pointing at a server that never responds (e.g. a TCP listener that accepts but never replies) causes that thread-pool worker to hang forever. The default asyncio executor thread pool is shared app-wide (image/video processing in chat_api.py, `validate_email` DNS lookups, link-preview `getaddrinfo`), so a handful of such subscriptions receiving pushes (e.g. via repeated messages/DMs to the attacker's own account) can exhaust the pool and stall unrelated request handling across the whole server. Fix: pass an explicit `timeout=` to every `webpush()` call.

[CRITICAL] server/chat_api.py:259-303 (`auth_google`) and :305-367 (`auth_google_code`) — Both handlers make synchronous, blocking network calls directly inside `async def` route handlers with no `asyncio.to_thread` wrapper and no rate limiting: `httpx.post("https://oauth2.googleapis.com/token", ...)` at line 321 (blocking, not `await`), and `google_id_token.verify_oauth2_token(...)` at lines 275/339, which internally performs a synchronous `requests`-based HTTP fetch of Google's public certs (cached after first success, but blocking on every call until cached, and again after cache expiry). Since these routes are pre-authentication and confirmed to have zero rate limiting anywhere in the stack (no global middleware in `server/api.py`, no per-route `_check_rate`/`_email_rate` call in either handler — verified against all of `server/api.py` and the `/login` limiter pattern at `chat_api.py:371-376`), an unauthenticated caller can flood `/chat/api/auth/google/code` with garbage `code` values; each request blocks the single asyncio event loop for the duration of the DNS+TLS+HTTP round trip to Google, serializing and stalling all other HTTP requests and WebSocket message processing (chat, lineup, admin) server-wide. Fix: wrap both external calls in `asyncio.to_thread`, and add rate limiting to both routes.
